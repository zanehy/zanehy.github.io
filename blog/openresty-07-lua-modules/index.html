07 | Lua 模块系统与代码组织

发表于： 2026-03-11 分类于： OpenResty
字数： 1634 阅读：≈ 4分钟 浏览：

上一篇我们把 table 和元表研究透了。现在代码量开始多起来，总不能把所有逻辑都塞进一个文件。这篇来聊 Lua 的模块系统——如何拆分代码、如何复用，以及在 OpenResty 里该怎么组织项目结构。

require 是什么

Lua 用 require 加载模块，用法很简单：

local cjson = require "cjson"
local redis = require "resty.redis"
local my_util = require "utils.string_helper"

require 背后做了几件事：

  1. 检查 package.loaded 缓存，如果已经加载过，直接返回缓存结果（不会重复执行）
  2. 在 package.path 指定的路径里查找对应的 .lua 文件
  3. 找到后执行该文件，把返回值存入缓存并返回

这个缓存机制很重要：同一个模块无论 require 多少次，只执行一次。

定义一个模块

Lua 模块本质上就是一个返回 table 的文件：

-- utils/math_helper.lua
local M = {}

function M.add(a, b)
    return a + b
end

function M.clamp(val, min, max)
    if val < min then return min end
    if val > max then return max end
    return val
end

-- 私有函数，不挂在 M 上，外部访问不到
local function internal_check(val)
    return type(val) == "number"
end

function M.safe_sqrt(val)
    if not internal_check(val) then
        return nil, "not a number"
    end
    return math.sqrt(val)
end

return M

使用时：

local math_helper = require "utils.math_helper"

print(math_helper.add(3, 4))       -- 7
print(math_helper.clamp(15, 0, 10)) -- 10

local result, err = math_helper.safe_sqrt(16)
print(result)  -- 4.0

模块文件名和目录名用 . 分隔对应到路径：utils.math_helper → utils/math_helper.lua。

package.path 的配置

package.path 决定了 require 去哪里找文件，默认值大概长这样：

./?.lua;/usr/local/share/lua/5.1/?.lua;...

? 是占位符，会被模块名替换（. 转换为 /）。

在普通 Lua 程序里，可以这样追加路径：

package.path = package.path .. ";/my/project/lib/?.lua"

但在 OpenResty 里，不要在请求阶段动态修改 package.path——这会影响所有 worker，且不是线程安全的。正确的做法是在 nginx.conf 里配置：

http {
    lua_package_path "/path/to/your/lua/lib/?.lua;;";
    lua_package_cpath "/path/to/your/c/lib/?.so;;";
    
    server {
        # ...
    }
}

末尾的 ;; 表示保留默认路径，加在后面一起搜索。

模块的几种写法

写法一：标准 table 返回（推荐） 

-- auth.lua
local M = {}

M.VERSION = "1.0.0"

function M.verify_token(token)
    -- ...
    return true
end

return M

简洁、明确，最常见。

写法二：模块即对象（带状态） 

-- counter.lua
local Counter = {}
Counter.__index = Counter

function Counter.new(start)
    return setmetatable({value = start or 0}, Counter)
end

function Counter:increment()
    self.value = self.value + 1
end

function Counter:get()
    return self.value
end

return Counter

使用：

local Counter = require "counter"
local c = Counter.new(10)
c:increment()
print(c:get())  -- 11

写法三：直接返回函数 

-- validator.lua
return function(input)
    return input ~= nil and input ~= ""
end

适合简单的单一功能模块。

在 OpenResty 中组织代码

OpenResty 项目的典型目录结构：

project/
├── conf/
│   └── nginx.conf
├── lua/
│   ├── app/
│   │   ├── router.lua        -- 路由逻辑
│   │   ├── middleware.lua     -- 中间件
│   │   └── handlers/
│   │       ├── auth.lua      -- 鉴权处理
│   │       └── proxy.lua     -- 代理处理
│   └── lib/
│       ├── redis_client.lua  -- Redis 封装
│       ├── jwt.lua           -- JWT 工具
│       └── response.lua      -- 统一响应格式
└── logs/

nginx.conf 里引用：

http {
    lua_package_path "$prefix/lua/?.lua;;";
    
    init_by_lua_block {
        -- 应用启动时预加载模块
        require "app.router"
        require "lib.redis_client"
    }
    
    server {
        location /api/ {
            access_by_lua_block {
                local auth = require "app.handlers.auth"
                auth.verify()
            }
            
            content_by_lua_block {
                local router = require "app.router"
                router.dispatch()
            }
        }
    }
}

模块缓存与热更新

require 的缓存存在 package.loaded 里。在开发调试时，有时需要强制重新加载模块：

-- 清除缓存，下次 require 会重新执行
package.loaded["my.module"] = nil
local m = require "my.module"

但在生产环境，OpenResty 的模块缓存是 per-worker 的。如果需要热更新配置（不重启 nginx），通常不依赖模块缓存，而是把数据放在 ngx.shared.DICT 或 Redis 里，模块只是读取数据的逻辑层。

init_by_lua 预加载的意义 

init_by_lua_block {
    require "resty.core"
    require "app.config"
    require "lib.redis_client"
}

init_by_lua 在 master 进程 fork 出 worker 之前执行。提前 require 模块有两个好处：

  1. 写时复制（COW）：fork 后 worker 共享 master 的内存页，模块代码不需要每个 worker 单独加载，节省内存
  2. 首次请求不卡顿：模块代码已经编译好，第一个请求不需要花时间加载

这是 OpenResty 性能优化的常见手段之一。

几个容易踩的坑

坑一：循环依赖

-- a.lua
local b = require "b"
-- b.lua
local a = require "a"  -- 循环！

Lua 处理循环依赖时，第二个 require 拿到的是未完全初始化的 table。解决方案是重构代码，把公共部分抽出来，或者把 require 放到函数内部（延迟加载）。

坑二：模块级别的副作用

-- bad_module.lua
local redis = require "resty.redis"
local red = redis:new()
red:connect("127.0.0.1", 6379)  -- 模块加载时就建立连接！

return { red = red }

模块被 require 时会执行一次，在 init_by_lua 阶段（master 进程）执行网络操作会报错。正确做法是把连接逻辑放在工厂函数里，按需调用。

坑三：模块里存储请求级别的状态

-- counter.lua（错误示范）
local M = { count = 0 }

function M.increment()
    M.count = M.count + 1  -- 所有请求共享这个 count！
end

模块 table 是 worker 级别的，被所有请求共享。请求级别的状态要用 ngx.ctx，不能存在模块变量里。

小结

Lua 模块系统简单却强大：一个文件，返回一个 table，这就是模块的全部。掌握几个要点：

  • require 有缓存，同一模块只加载一次
  • package.path 在 OpenResty 里通过 lua_package_path 配置，不要在运行时修改
  • init_by_lua 预加载模块可以节省内存、提升首次请求速度
  • 模块变量是 worker 级别共享的，不要在里面存请求状态
