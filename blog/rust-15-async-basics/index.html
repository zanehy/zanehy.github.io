15 | Rust 异步编程基础

发表于： 2026-04-26 分类于： Rust
字数： 2751 阅读：≈ 6分钟 浏览：

上一篇我们聊了 Rust 的并发基础：线程、Mutex、Channel。那是"多线程"模型的并发——每个任务跑在独立线程上，操作系统负责调度。这套模型对 CPU 密集型任务很合适，但一旦遇到大量 I/O——网络请求、数据库查询、文件读写——就会遇到麻烦。

为什么需要 async

先看一个具体场景。假设你要同时处理 10000 个 HTTP 连接，每个连接大部分时间在等待客户端发数据。如果每个连接创建一个线程，就需要 10000 个线程。

问题在哪？线程不便宜。每个线程默认有几 MB 的栈空间，10000 个线程光内存就要几十 GB。更要命的是线程切换有开销——操作系统在线程间切换时要保存和恢复寄存器、切换内存上下文，这些开销在高并发时会显著拖慢系统。

Go 用 goroutine 解决这个问题：goroutine 是用户态轻量级线程，几十 KB 起步，运行时负责调度，可以轻松开百万个。Python 的 asyncio 也是类似思路——用协程避免线程开销。

Rust 的 async/await 走的是同一条路，但实现方式更底层、更零成本。理解它的关键是 Future Trait。

Future Trait：异步的核心抽象

Rust 标准库中 Future 的定义非常简单：

pub trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}

pub enum Poll<T> {
    Ready(T),
    Pending,
}

这就是 Rust 异步的本质：一个可以被"轮询"的对象。每次调用 poll，它要么返回 Ready(值)（任务完成了），要么返回 Pending（还没好，待会再来）。

用餐厅取餐类比：你点了餐，服务员给你一个取餐号（Future）。你可以去问"好了吗？"（poll）。如果好了，拿走；没好，回去等。与其一直站在窗口等（阻塞），不如去做别的事，等被叫到再来（异步）。

Context 里包含一个 Waker——当 Future 准备好时，它会调用 waker.wake() 通知运行时"可以来取了"。这样运行时就不用不停地轮询，而是被动等通知，效率高很多。

Pin<&mut Self> 是另一个关键设计——它保证 Future 在内存中的位置不会移动，这对异步状态机（有自引用结构）至关重要。初学阶段不用深究，知道它存在的意义就够了。

async fn 展开成了什么

写 async 代码时，你看到的是这样的：

async fn fetch_data(url: &str) -> String {
    let response = reqwest::get(url).await.unwrap();
    response.text().await.unwrap()
}

编译器实际上把这个函数展开成一个实现了 Future 的状态机。大致等价于：

enum FetchDataFuture<'a> {
    Start { url: &'a str },
    WaitingForResponse { /* 中间状态 */ },
    WaitingForText { /* 中间状态 */ },
    Done,
}

impl<'a> Future for FetchDataFuture<'a> {
    type Output = String;

    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<String> {
        // 根据当前状态决定做什么
        // 每次 .await 点都是一个可能返回 Pending 的地方
        todo!()
    }
}

每个 .await 就是状态机里的一个"检查点"——如果子 Future 还没好，整个状态机暂停在这里，把控制权还给运行时。运行时可以去推进其他 Future，等这个 Future 的 Waker 被触发再回来继续。

这就是 async 的零成本抽象：没有额外的堆分配（状态机大小编译期确定），没有运行时线程开销，只是在 .await 点把执行权让出去。

为什么需要 Tokio

关键问题来了：谁来调用 poll？

Future 本身是惰性的——你创建一个 Future，什么都不会发生，必须有人不断地驱动它（调用 poll）。这个"驱动者"就是异步运行时，Tokio 是 Rust 生态最主流的选择。

Tokio 做了什么：

  1. 任务调度器：管理所有 async 任务，决定哪个先跑
  2. I/O 事件循环：监听文件描述符的就绪状态（用 epoll/kqueue 等系统调用）
  3. 线程池：默认用多个工作线程充分利用多核
  4. Waker 实现：把系统 I/O 事件和 Future 的唤醒机制连接起来

没有 Tokio（或其他运行时），async fn 只是语法糖，写了也跑不起来。这和 Go 不同——Go 的调度器内置在语言运行时里，你不需要选；Rust 把这个选择权留给你。

使用 Tokio 最简单的方式：

// Cargo.toml
// tokio = { version = "1", features = ["full"] }

#[tokio::main]
async fn main() {
    println!("异步世界，你好");
    let result = fetch_something().await;
    println!("{}", result);
}

async fn fetch_something() -> String {
    // 模拟 I/O 等待
    tokio::time::sleep(std::time::Duration::from_millis(100)).await;
    "数据到手".to_string()
}

#[tokio::main] 是个宏，它把 async fn main 展开成：

fn main() {
    tokio::runtime::Builder::new_multi_thread()
        .enable_all()
        .build()
        .unwrap()
        .block_on(async {
            // 你的 main 逻辑
        })
}

block_on 是同步世界和异步世界的桥梁——它在当前线程上运行 Future，直到完成。

并发执行多个 Future

async 真正的价值在于同时推进多个任务：

use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    // 顺序执行：总耗时 = 200ms + 300ms = 500ms
    sequential().await;

    // 并发执行：总耗时 ≈ max(200ms, 300ms) = 300ms
    concurrent().await;
}

async fn task_a() -> &'static str {
    sleep(Duration::from_millis(200)).await;
    "A 完成"
}

async fn task_b() -> &'static str {
    sleep(Duration::from_millis(300)).await;
    "B 完成"
}

async fn sequential() {
    let a = task_a().await;
    let b = task_b().await;
    println!("顺序: {} {}", a, b);
}

async fn concurrent() {
    // tokio::join! 同时推进多个 Future
    let (a, b) = tokio::join!(task_a(), task_b());
    println!("并发: {} {}", a, b);
}

tokio::join! 在同一个任务里并发推进多个 Future——不是多线程，而是在同一个线程上交替推进。当 task_a 在等待时，调度器去推进 task_b，反之亦然。

如果想要真正的并行（利用多核），用 tokio::spawn：

async fn parallel() {
    let handle_a = tokio::spawn(task_a()); // 可能在另一个线程上运行
    let handle_b = tokio::spawn(task_b());

    let a = handle_a.await.unwrap();
    let b = handle_b.await.unwrap();
    println!("并行: {} {}", a, b);
}

spawn 把 Future 作为独立任务提交给 Tokio 的线程池，真正并行执行。

常见陷阱

陷阱一：在 async 中持有标准库 Mutex 跨越 await 

use std::sync::Mutex;

async fn bad_example(data: &Mutex<Vec<i32>>) {
    let guard = data.lock().unwrap(); // 持有锁
    some_async_operation().await;     // 跨越 await 点！
    println!("{:?}", *guard);         // 锁在这里才释放
}

问题：std::sync::Mutex 的锁守卫（MutexGuard）没有实现 Send，当任务在 await 点暂停时可能被调度到另一个线程继续执行，这会导致编译错误。即使编译通过，长时间持锁跨越 await 也会阻塞其他需要这把锁的任务，破坏并发性。

正确做法：

use tokio::sync::Mutex; // 使用 tokio 的异步 Mutex

async fn good_example(data: &tokio::sync::Mutex<Vec<i32>>) {
    let guard = data.lock().await; // 异步加锁
    some_async_operation().await;  // 可以安全跨越
    println!("{:?}", *guard);
}

// 或者缩小锁的持有范围
async fn also_good(data: &std::sync::Mutex<Vec<i32>>) {
    {
        let guard = data.lock().unwrap();
        // 只在这个块内持锁，不跨越 await
        do_sync_work(&guard);
    } // 锁在这里释放
    some_async_operation().await; // await 时已经没有锁了
}

陷阱二：在 async 中调用阻塞操作 

async fn bad() {
    // std::thread::sleep 会阻塞整个线程！
    // 这个线程上的所有其他 async 任务都会被卡住
    std::thread::sleep(std::time::Duration::from_secs(1));
}

async fn good() {
    // tokio::time::sleep 只暂停当前任务，线程可以去做别的
    tokio::time::sleep(std::time::Duration::from_secs(1)).await;
}

如果必须调用阻塞操作（比如 CPU 密集计算、同步文件 I/O），用 spawn_blocking：

async fn with_blocking() {
    // 把阻塞工作移到专用线程池，不占用 async 工作线程
    let result = tokio::task::spawn_blocking(|| {
        heavy_computation()
    }).await.unwrap();

    println!("{}", result);
}

fn heavy_computation() -> i32 {
    (0..1_000_000).sum()
}

陷阱三：忘记 .await 

async fn example() {
    fetch_data();       // 什么都没发生！Future 被创建又立刻丢弃
    fetch_data().await; // 正确：驱动 Future 到完成
}

Rust 编译器会对未使用的 Future 发出 unused_must_use 警告，不会完全沉默，但初学者容易忽略。

小结

  • 为什么需要 async：I/O 密集场景下，线程太重；async 用状态机替代线程，在等待时让出 CPU 给其他任务
  • Future Trait：核心是 poll 方法，返回 Ready(值) 或 Pending；Waker 机制避免空转轮询
  • async fn 展开：编译器将 async 函数变成状态机，每个 .await 是可暂停点，无额外堆分配
  • 运行时：Future 是惰性的，必须有运行时（Tokio）来驱动；Tokio 提供调度器、I/O 事件循环和线程池
  • 并发工具：tokio::join! 在单任务内并发，tokio::spawn 跨线程并行
  • 常见陷阱：持 std::sync::Mutex 跨 await 要改用 tokio::sync::Mutex；阻塞操作要用 spawn_blocking；不要忘记 .await
