19 | 本地缓存与多级缓存

发表于： 2026-03-27 分类于： OpenResty
字数： 2040 阅读：≈ 5分钟 浏览：

缓存是高并发系统的核心优化手段。OpenResty 的独特之处在于它有多层缓存可以使用：进程内的 LRU 缓存、worker 级的共享内存、以及外部的 Redis。合理组合这三层，可以在极低延迟下扛住非常高的并发。

ngx.shared.DICT 进阶

第 12 篇已经介绍了 ngx.shared.DICT 的基础用法，这里补充几个进阶场景。

带标志位的缓存

set、get 等方法支持一个可选的 flags 参数（uint32 整数），可以用来存储元信息：

local cache = ngx.shared.my_cache

-- 用 flags 存数据类型标记（1=string, 2=number, 3=json）
cache:set("config", '{"debug":true}', 300, 3)  -- 第4个参数是 flags

local val, flags = cache:get("config")
if flags == 3 then
    val = cjson.decode(val)
end

分布式锁（简易版）

add 方法的原子性可以用来实现简单的分布式锁：

local lock_key = "lock:cache_rebuild:" .. resource_id
local ok, err, forcible = ngx.shared.locks:add(lock_key, 1, 10)  -- 10 秒超时

if not ok then
    -- 已有其他请求在重建缓存，等待后读取
    ngx.sleep(0.1)
    return get_from_cache(resource_id)
end

-- 拿到锁，重建缓存
local data = fetch_from_db(resource_id)
ngx.shared.my_cache:set(resource_id, cjson.encode(data), 300)

-- 释放锁
ngx.shared.locks:delete(lock_key)
return data

内存占满时的行为

当 ngx.shared.DICT 内存用满，新写入时有两种结果：

  • 如果有可以 LRU 淘汰的 key，自动淘汰旧 key（set 返回的第三个值 forcible = true）
  • 如果所有 key 都没有过期时间且不可淘汰，写入失败
local ok, err, forcible = cache:set("key", "val", 60)
if forcible then
    ngx.log(ngx.WARN, "cache memory full, evicted a key")
end
if not ok then
    ngx.log(ngx.ERR, "cache set failed: ", err)
end

lua-resty-lrucache：进程内 LRU 缓存

ngx.shared.DICT 是 worker 内所有协程共享的，读写有锁竞争。对于超高频访问的热点数据，可以用 lua-resty-lrucache 把数据缓存在每个协程独享的 Lua table 里，完全无锁，速度极快。

lua-resty-lrucache 是 OpenResty 内置库。

基本用法 

local lrucache = require "resty.lrucache"

-- 创建一个最多缓存 200 个 key 的 LRU cache
-- 注意：这个对象是模块级别的，每个 worker 一份
local user_cache, err = lrucache.new(200)
if not user_cache then
    error("failed to create lrucache: " .. err)
end

-- 写入（带 TTL，单位秒）
user_cache:set("user:123", {name="alice", role="admin"}, 60)

-- 读取
local val, stale = user_cache:get("user:123")
-- stale = true 表示数据已过期（但还能读到，用于降级）

-- 删除
user_cache:delete("user:123")

在模块里正确使用 

-- lib/user_cache.lua
local lrucache = require "resty.lrucache"

-- 模块加载时创建 cache 实例（每个 worker 各一份）
local M = {}
local cache = lrucache.new(500)

function M.get(user_id)
    local key = "u:" .. user_id
    local val, stale = cache:get(key)

    if val and not stale then
        return val  -- 命中缓存
    end

    -- 缓存未命中或已过期，从下游获取
    local user = fetch_user_from_redis(user_id)
    if user then
        cache:set(key, user, 30)  -- 缓存 30 秒
    end

    return user
end

return M

lrucache 的局限

  • 不跨 worker 共享：4 个 worker 就有 4 份独立的 lrucache，数据更新后各 worker 的缓存需要等 TTL 自然过期
  • 不能存太多数据：受 Lua GC 管理，容量不宜设太大（几百到几千条）
  • 适合读多写极少的热点数据：用户基本信息、权限配置、黑白名单等

多级缓存架构

把 lrucache、ngx.shared.DICT、Redis 三层组合起来，形成多级缓存：

请求 → L1(lrucache，协程内) → L2(shared.DICT，worker内) → L3(Redis) → DB

每一层的特性：

层级速度容量共享范围适合数据
L1 lrucache极快（ns级）小（百～千条）单协程超热点、小数据
L2 shared.DICT快（μs级）中（MB级）单worker热点数据
L3 Redis较快（ms级）大（GB级）所有实例通用缓存

完整的多级缓存实现 

-- lib/multilevel_cache.lua
local lrucache   = require "resty.lrucache"
local redis      = require "resty.redis"
local cjson      = require "cjson"

local M = {}

-- L1：每个 worker 的 lrucache
local l1 = lrucache.new(1000)

-- L2：共享内存
local l2 = ngx.shared.app_cache

-- 从 Redis 获取数据
local function get_from_redis(key)
    local red = redis:new()
    red:set_timeouts(500, 500, 500)
    local ok = red:connect("127.0.0.1", 6379)
    if not ok then return nil end

    local val = red:get(key)
    red:set_keepalive(10000, 100)

    if not val or val == ngx.null then return nil end
    return val
end

-- 多级缓存读取
function M.get(key, fetch_fn, ttl)
    ttl = ttl or 300

    -- L1 查询
    local val = l1:get(key)
    if val then return val end

    -- L2 查询
    local val_str = l2:get(key)
    if val_str then
        local data = cjson.decode(val_str)
        l1:set(key, data, math.min(ttl, 10))  -- L1 TTL 更短
        return data
    end

    -- L3 查询
    val_str = get_from_redis(key)
    if val_str then
        local data = cjson.decode(val_str)
        l2:set(key, val_str, ttl)
        l1:set(key, data, math.min(ttl, 10))
        return data
    end

    -- 全部未命中，调用实际获取函数
    if not fetch_fn then return nil end
    local data, err = fetch_fn(key)
    if not data then return nil, err end

    -- 回填各层缓存
    local encoded = cjson.encode(data)
    get_from_redis_write(key, encoded, ttl)  -- 写 Redis
    l2:set(key, encoded, ttl)
    l1:set(key, data, math.min(ttl, 10))

    return data
end

return M

缓存穿透、击穿、雪崩

这是缓存系统三个经典问题，OpenResty 里都有对应的解决方案。

缓存穿透

问题：查询一个不存在的数据，每次都穿透缓存打到数据库。

解决方案：缓存空值

local val = cache:get(key)

-- 空字符串 "" 作为"不存在"的标记值
if val == "" then
    return nil  -- 缓存了空值，直接返回 nil，不查 DB
end

if not val then
    local data = fetch_from_db(key)
    if not data then
        -- 数据不存在，缓存空值（较短的 TTL）
        cache:set(key, "", 30)
        return nil
    end
    cache:set(key, cjson.encode(data), 300)
    return data
end

return cjson.decode(val)

缓存击穿

问题：某个热点 key 过期的瞬间，大量请求同时穿透到数据库（也叫"缓存失效风暴"）。

解决方案：互斥锁（只让一个请求去重建缓存）

local function get_with_lock(key)
    -- 先查缓存
    local val = l2:get(key)
    if val then return cjson.decode(val) end

    -- 用 add 实现互斥锁
    local lock_key = "lock:" .. key
    local locked = l2:add(lock_key, 1, 5)  -- 5 秒锁超时

    if not locked then
        -- 没拿到锁，等待后再读缓存（其他请求正在重建）
        ngx.sleep(0.05)
        val = l2:get(key)
        return val and cjson.decode(val) or nil
    end

    -- 拿到锁，重建缓存
    local data = fetch_from_db(key)
    if data then
        l2:set(key, cjson.encode(data), 300)
    end
    l2:delete(lock_key)
    return data
end

缓存雪崩

问题：大量 key 同时过期，导致瞬间大量请求打到数据库。

解决方案：TTL 加随机抖动

local base_ttl = 300
-- 在 base_ttl 基础上加 0～60 秒的随机值
local ttl = base_ttl + math.random(0, 60)
cache:set(key, value, ttl)

随机 TTL 把集中过期分散到一个时间段，避免雪崩。

小结

OpenResty 的多级缓存体系：

  • lrucache：进程内 LRU，无锁极速，适合超热点小数据，容量有限
  • ngx.shared.DICT：worker 内共享，有原子操作，适合通用缓存和计数器
  • 多级缓存：L1(lrucache) + L2(shared.DICT) + L3(Redis)，按热度分层存储

三大问题的应对：

  • 穿透：缓存空值（用特殊标记区分"数据不存在"和"缓存未命中"）
  • 击穿：互斥锁（add 的原子性保证只有一个请求重建缓存）
  • 雪崩：TTL 加随机抖动（把集中过期分散开）
