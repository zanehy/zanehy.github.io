14 | 定时器与后台任务

发表于： 2026-03-10 分类于： OpenResty
字数： 1920 阅读：≈ 4分钟 浏览：

上一篇提到，在 log_by_lua 阶段不能用 cosocket，但可以用 ngx.timer.at 把任务推到后台异步执行。这篇专门来讲 OpenResty 的定时器机制——它是实现后台任务的核心工具。

ngx.timer.at：一次性定时器

ngx.timer.at(delay, callback, ...) 在 delay 秒后执行 callback。

-- 0 秒后立即执行（下一个事件循环）
ngx.timer.at(0, function(premature)
    if premature then
        -- premature = true 表示 nginx 正在关闭（worker 退出时）
        -- 应该尽快清理并返回，不要做耗时操作
        return
    end
    ngx.log(ngx.INFO, "timer fired!")
end)

-- 5 秒后执行
ngx.timer.at(5, function(premature)
    if premature then return end
    -- 做一些事情
end)

-- 传递额外参数
local function report_log(premature, log_data)
    if premature then return end
    -- 上报 log_data
    ngx.log(ngx.INFO, "reporting: ", log_data.uri)
end

ngx.timer.at(0, report_log, {
    uri    = ngx.var.uri,
    status = ngx.status,
    time   = ngx.now(),
})

关于 premature 参数： 这是 OpenResty 传给所有定时器回调的第一个参数。当 nginx worker 正在关闭（收到 SIGTERM 或 SIGQUIT）时，所有未触发的定时器会被强制触发，premature 置为 true。回调里应该检查这个参数，在 premature = true 时优雅退出，不要再发起新的 I/O。

ngx.timer.every：周期定时器

ngx.timer.every(interval, callback, ...) 每隔 interval 秒重复执行一次。

-- 每 60 秒同步一次配置
local function sync_config(premature)
    if premature then return end

    local red = require("resty.redis"):new()
    red:settimeout(1000)
    local ok, err = red:connect("127.0.0.1", 6379)
    if not ok then
        ngx.log(ngx.ERR, "redis connect failed: ", err)
        return
    end

    local config_json = red:get("app:config")
    if config_json and config_json ~= ngx.null then
        local cache = ngx.shared.app_config
        cache:set("config", config_json)
    end

    red:set_keepalive(10000, 100)
end

-- 在 init_worker_by_lua 里启动周期任务
init_worker_by_lua_block {
    local ok, err = ngx.timer.every(60, sync_config)
    if not ok then
        ngx.log(ngx.ERR, "failed to start timer: ", err)
    end
}

ngx.timer.every 从 OpenResty 1.11.2.2 版本开始支持。如果版本较旧，可以用 ngx.timer.at 实现递归定时器（见下面的例子）。

定时器的触发环境

定时器回调在一个独立的"轻线程"里执行，这个线程不属于任何 HTTP 请求。这意味着：

  • 可以使用 cosocket（连 Redis、发 HTTP 请求等）
  • 不能使用 ngx.say、ngx.exit、ngx.req.*、ngx.header 等请求相关 API
  • 可以访问 ngx.shared.DICT、ngx.log、ngx.now 等全局 API
ngx.timer.at(0, function(premature)
    if premature then return end

    -- OK：可以用 cosocket
    local httpc = require("resty.http").new()
    local res = httpc:request_uri("http://monitor.internal/ping")

    -- OK：可以写日志
    ngx.log(ngx.INFO, "health check done: ", res.status)

    -- OK：可以读写共享内存
    ngx.shared.stats:incr("ping_count", 1, 0)

    -- 错误！不能用请求相关 API
    -- ngx.say("hello")      -- 报错
    -- ngx.var.uri           -- 报错
end)

定时器的数量限制

OpenResty 对同时挂起的定时器数量有限制，由 lua_max_pending_timers 控制（默认 1024）。超出限制后，新建定时器会失败并返回错误。

http {
    lua_max_pending_timers 2048;   -- 调高上限
    lua_max_running_timers 256;    -- 同时运行的定时器上限（默认 256）
}

如果你的业务需要频繁创建定时器（比如每个请求都用 timer.at(0, ...) 做异步上报），要估算一下高峰期的定时器积压量，避免超限。

实战：后台异步上报日志

这是 OpenResty 里最常见的定时器使用场景：不阻塞请求响应，把日志批量异步上报到后端（Kafka、ES、HTTP 接口等）。

方案设计

  1. 每个请求在 log_by_lua 阶段把日志写入 ngx.shared.DICT（内存队列）
  2. 后台定时器定期从队列里取出批量数据，发送到后端
  3. 请求处理和日志上报完全解耦
http {
    lua_shared_dict log_queue  10m;

    init_worker_by_lua_block {
        -- 每 5 秒批量上报一次
        ngx.timer.every(5, function(premature)
            if premature then return end

            local queue = ngx.shared.log_queue
            local logs = {}

            -- 取出所有积压的日志（最多 500 条）
            for i = 1, 500 do
                local key = "log:" .. i
                local val = queue:get(key)
                if not val then break end
                logs[#logs + 1] = val
                queue:delete(key)
            end

            if #logs == 0 then return end

            -- 批量发送到日志服务
            local httpc = require("resty.http").new()
            local cjson = require "cjson"
            local ok, err = httpc:request_uri("http://log-service/batch", {
                method = "POST",
                body   = cjson.encode(logs),
                headers = { ["Content-Type"] = "application/json" },
            })

            if not ok then
                ngx.log(ngx.ERR, "failed to report logs: ", err)
            end
        end)
    }

    server {
        location / {
            # ... 正常处理请求 ...

            log_by_lua_block {
                local queue = ngx.shared.log_queue
                local cjson = require "cjson"

                -- 生成递增 key（简化版，生产环境需要更健壮的实现）
                local seq, _ = queue:incr("__seq", 1, 0)
                queue:set("log:" .. seq, cjson.encode({
                    uri    = ngx.var.uri,
                    status = ngx.status,
                    time   = ngx.now(),
                    ip     = ngx.var.remote_addr,
                }), 60)   -- 60 秒过期兜底，防止内存泄漏
            }
        }
    }
}

这个方案的优点：日志写入只是一次内存操作（微秒级），完全不影响请求响应时间。

用 timer.at 实现周期任务（兼容旧版本）

如果 OpenResty 版本不支持 timer.every，可以用递归的 timer.at 模拟：

local function periodic_task(premature)
    if premature then return end

    -- 执行实际任务
    do_some_work()

    -- 重新注册，实现周期执行
    local ok, err = ngx.timer.at(60, periodic_task)
    if not ok then
        ngx.log(ngx.ERR, "failed to reschedule timer: ", err)
    end
end

-- 启动
ngx.timer.at(60, periodic_task)

注意：这种方式每次执行后重新注册，间隔时间是"上次执行完成后等 60 秒"，而不是严格的每 60 秒触发一次。

常见陷阱

陷阱一：在请求处理阶段大量创建定时器

content_by_lua_block {
    -- 每个请求都创建定时器，高并发时可能瞬间耗尽定时器配额
    for i = 1, 100 do
        ngx.timer.at(0, some_task, i)
    end
}

一个请求里别创建太多定时器，更好的做法是把数据写进共享内存，由统一的后台定时器处理。

陷阱二：定时器里的错误被静默吞掉

定时器回调里如果抛出未捕获的错误，不会影响其他请求，但也不会有明显的报错提示（除非你开了 debug 日志）。建议在回调里加 pcall 保护：

ngx.timer.at(0, function(premature)
    if premature then return end
    local ok, err = pcall(function()
        do_risky_work()
    end)
    if not ok then
        ngx.log(ngx.ERR, "timer error: ", err)
    end
end)

陷阱三：worker 退出时任务丢失

定时器里的数据如果只在内存里，worker 退出（nginx reload/stop）时数据就丢了。对于不能丢的数据（比如账单、关键日志），要先持久化到 Redis 或数据库，再做异步处理。

小结

ngx.timer 是 OpenResty 后台任务的基础设施：

  • ngx.timer.at：一次性延迟执行
  • ngx.timer.every：周期重复执行
  • 回调里可以用 cosocket，但不能用请求相关 API
  • 检查 premature 参数，优雅应对 worker 关闭
  • 控制定时器创建频率，避免超出 lua_max_pending_timers 限制

到这里，第三阶段的核心 API 全部讲完了。下一篇进入第四阶段——中间件与存储，从 Redis 开始。
