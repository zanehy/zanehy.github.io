19 | Unsafe Rust 与 FFI：踏入无人监管的领地

发表于： 2026-04-28 分类于： Rust
字数： 5129 阅读：≈ 11分钟 浏览：

Rust 的招牌卖点是内存安全，但安全是有代价的——借用检查器的规则有时会拦住一些"你知道是对的，但编译器无法证明"的操作。unsafe 就是那把钥匙：打开一扇门，让你进入编译器不再保护的区域。

这篇文章聊清楚三件事：unsafe 边界在哪里、怎么安全地包装 unsafe、以及如何用 FFI 和 C 库对话。

unsafe 的本质：不是"危险代码"，是"你担保"

很多人看到 unsafe 关键字就紧张，以为写了 unsafe 代码就一定会崩。这个理解有偏差。

unsafe 的真实含义是：编译器将验证责任从自己转移给你。Safe Rust 里，编译器能证明内存安全；unsafe 里，编译器说"这段我证明不了，你来担保"。

你担保的是：这段代码在所有合法调用路径下不会引发未定义行为（Undefined Behavior，UB）。如果你担保错了，程序可能崩溃、数据可能损坏、安全漏洞可能出现——但这是你的责任，不是编译器的。

一个很重要的认知：unsafe 块只是告诉编译器"这里我负责"，它不会关闭 Rust 的所有检查。生命周期检查、类型检查、借用检查仍然在运行，只是解锁了五个额外操作。

五件只有 unsafe 才能做的事

Rust 精确划定了 unsafe 的边界：只有以下五类操作需要 unsafe，其他任何操作都是 safe 的。

1. 解引用裸指针

裸指针（raw pointer）是 *const T 和 *mut T，类似 C 的指针。创建裸指针是 safe 的，但解引用它需要 unsafe：

fn main() {
    let x = 42i32;
    let p: *const i32 = &x; // 创建裸指针，safe
    
    // 解引用裸指针，unsafe
    let val = unsafe { *p };
    println!("{}", val); // 42
}

为什么解引用需要 unsafe？因为裸指针可能是空指针、悬空指针（指向已释放内存）、未对齐指针，编译器无法在编译期验证这些情况。

裸指针在哪里会出现？FFI 调用返回的指针、从整数转换的内存地址、手动内存布局。

2. 调用 unsafe 函数或方法

某些函数被标记为 unsafe fn，调用它们需要包裹在 unsafe 块中：

unsafe fn dangerous_operation(ptr: *mut i32, len: usize) {
    // 直接操作内存
    for i in 0..len {
        *ptr.add(i) = i as i32;
    }
}

fn main() {
    let mut data = vec![0i32; 5];
    unsafe {
        dangerous_operation(data.as_mut_ptr(), data.len());
    }
    println!("{:?}", data); // [0, 1, 2, 3, 4]
}

标准库里有不少 unsafe 函数，比如 std::mem::transmute（重新解释内存布局）、std::ptr::copy_nonoverlapping（类似 C 的 memcpy）等。

3. 访问或修改可变静态变量

静态变量（static）在多线程环境中访问可变版本（static mut）是 unsafe 的，因为存在数据竞争风险：

static mut COUNTER: u32 = 0;

fn increment() {
    unsafe {
        COUNTER += 1;
    }
}

fn main() {
    increment();
    increment();
    unsafe {
        println!("Counter: {}", COUNTER); // 2
    }
}

实践中，static mut 很少直接用。更好的做法是用 Mutex<T> 或 AtomicU32 等同步原语，它们提供 safe 接口且线程安全。

4. 实现 unsafe trait

某些 trait 被标记为 unsafe trait，实现它们需要 unsafe：

unsafe trait MyUnsafeTrait {
    fn do_something(&self);
}

struct MyType;

unsafe impl MyUnsafeTrait for MyType {
    fn do_something(&self) {
        println!("unsafe trait implemented");
    }
}

最典型的例子是 Send 和 Sync。编译器能自动推导大部分类型的 Send/Sync 实现，但如果你有特殊的内存共享结构，需要手动 unsafe impl Send for MyType，这时你在承诺：我保证跨线程传递/共享这个类型是安全的。

5. 访问 union 的字段

Rust 支持 union（主要用于 FFI 兼容 C 的联合体），访问 union 字段是 unsafe 的，因为同一块内存可能被解释为不同类型：

union IntOrFloat {
    i: u32,
    f: f32,
}

fn main() {
    let x = IntOrFloat { i: 0x3F800000 }; // 对应 IEEE 754 的 1.0f32
    unsafe {
        println!("as f32: {}", x.f); // 1.0
        println!("as u32: {}", x.i); // 1065353216
    }
}

如何封装安全的 unsafe 抽象

允许写 unsafe，但更重要的原则是：用 safe 接口包装 unsafe 实现。

这是 Rust 生态中最普遍的模式。标准库的 Vec<T>、String、HashMap 内部都有大量 unsafe 代码，但对外暴露的 API 全部是 safe 的。用户永远不需要写 unsafe 就能使用它们。

封装原则

好的 unsafe 封装需要：

  1. 最小化 unsafe 范围：unsafe 块只包含真正需要的代码，不要将无关代码放进去
  2. 清晰的安全不变量（safety invariants）：用注释说明"为什么这段 unsafe 是安全的"
  3. 对外暴露 safe 接口：内部 unsafe，外部 safe

一个完整的例子：安全的切片包装

假设我们要实现一个 split_at 函数，把切片分成两半：

fn split_at<T>(slice: &[T], mid: usize) -> (&[T], &[T]) {
    let len = slice.len();
    assert!(mid <= len, "mid > len");
    
    // SAFETY: 
    // - slice 是有效的，其生命周期由调用者保证
    // - mid <= len，所以两个切片都在原切片范围内
    // - 两个切片合起来不超过原切片，不会有内存重叠问题
    unsafe {
        (
            std::slice::from_raw_parts(slice.as_ptr(), mid),
            std::slice::from_raw_parts(slice.as_ptr().add(mid), len - mid),
        )
    }
}

fn main() {
    let v = [1, 2, 3, 4, 5];
    let (left, right) = split_at(&v, 3);
    println!("{:?}", left);  // [1, 2, 3]
    println!("{:?}", right); // [4, 5]
}

注意 // SAFETY: 注释——这是 Rust 社区的惯例，用来解释 unsafe 代码为什么安全。标准库和主流 crate 都遵循这个规范。

带有状态的安全抽象

一个更复杂的例子——实现一个"多路复用缓冲区"，允许从同一个 Vec 中拿出两个不重叠的可变切片：

pub struct SplitMut<'a, T> {
    left: &'a mut [T],
    right: &'a mut [T],
}

impl<'a, T> SplitMut<'a, T> {
    pub fn new(slice: &'a mut [T], mid: usize) -> Self {
        assert!(mid <= slice.len());
        let ptr = slice.as_mut_ptr();
        let len = slice.len();
        
        // SAFETY:
        // - ptr 有效，来自合法的 &mut [T]
        // - 两个切片不重叠（left 是 [0, mid)，right 是 [mid, len)）
        // - 生命周期 'a 绑定到原始切片，不会超出
        // - 原始 slice 的借用在这里结束，two subslices 接管了它的独占性
        unsafe {
            SplitMut {
                left: std::slice::from_raw_parts_mut(ptr, mid),
                right: std::slice::from_raw_parts_mut(ptr.add(mid), len - mid),
            }
        }
    }
}

fn main() {
    let mut data = vec![1, 2, 3, 4, 5];
    let split = SplitMut::new(&mut data, 3);
    split.left[0] = 10;
    split.right[0] = 40;
    println!("{:?}", data); // [10, 2, 3, 40, 5]
}

借用检查器无法证明 left 和 right 不重叠，但我们能用数学证明：mid <= len 保证两个区间没有交集。这就是"你担保"的含义。

FFI：与 C 库对话

FFI（Foreign Function Interface）是 Rust 与其他语言互操作的桥梁，最常见的是与 C 库交互。

为什么需要 FFI？

C 语言有几十年积累的生态——OpenSSL、SQLite、BLAS、各种系统 API——不可能全部重写。FFI 让 Rust 能直接调用这些库，复用成熟代码。

基础：调用 C 标准库函数 

use std::ffi::{c_char, CStr, CString};
use std::os::raw::c_int;

// 声明 C 函数
extern "C" {
    fn strlen(s: *const c_char) -> usize;
    fn abs(x: c_int) -> c_int;
}

fn main() {
    // 调用 C 的 abs 函数
    let result = unsafe { abs(-42) };
    println!("abs(-42) = {}", result); // 42

    // 调用 C 的 strlen
    let s = CString::new("hello").unwrap();
    let len = unsafe { strlen(s.as_ptr()) };
    println!("strlen = {}", len); // 5
}

extern "C" 块声明外部 C 函数，告诉编译器使用 C 调用约定（ABI）。调用这些函数需要 unsafe，因为编译器无法验证 C 函数的行为。

字符串处理：CString 与 CStr

Rust 的 String 和 C 的 char* 不兼容（Rust 字符串不是 null 结尾的），要在 FFI 边界转换需要专门的类型：

  • CString：Rust 侧拥有数据，作用域结束后自动释放。用于把 Rust 字符串传给 C。
  • CStr：借用，不拥有数据。用于读取 C 返回的字符串指针。
use std::ffi::{CStr, CString};
use std::os::raw::c_char;

// 从 Rust 字符串传给 C：创建 null 结尾的 CString
fn rust_to_c(s: &str) -> CString {
    CString::new(s).expect("字符串中不能含有 null 字节")
}

// 从 C 返回的指针读取字符串——生命周期绑定到指针所指向的数据
// 注意：不要标成 'static，实际生命周期由调用者保证
unsafe fn c_ptr_to_str<'a>(ptr: *const c_char) -> &'a str {
    assert!(!ptr.is_null(), "C 字符串指针为空");
    // SAFETY: 调用者保证 ptr 在 'a 生命周期内有效且 null 结尾
    CStr::from_ptr(ptr)
        .to_str()
        .expect("C 字符串不是合法 UTF-8")
}

fn main() {
    // CString 持有数据，c_str 只是它的借用视图
    let owned = CString::new("Hello, FFI").unwrap();
    let borrowed: &CStr = owned.as_c_str();
    println!("as_ptr: {:?}", borrowed);

    // 传给 C 函数时传裸指针，CString 必须保持存活直到 C 用完
    let _ptr: *const c_char = owned.as_ptr();
    // owned 在这里还活着，_ptr 是安全的
}

一个常见的错误是把 CString 临时变量立刻 drop，导致指针悬空：

// 错误！CString 在这一行结束后就被 drop 了，ptr 成为悬空指针
let ptr = CString::new("hello").unwrap().as_ptr(); // 危险！

正确做法是让 CString 的生命周期覆盖整个指针使用期间：

let s = CString::new("hello").unwrap();
let ptr = s.as_ptr(); // ptr 有效，s 还活着
unsafe { some_c_function(ptr); }
// s 在这里才 drop

一个完整的 FFI 示例：调用 libm 

// Cargo.toml 中不需要添加依赖，libm 是系统库

#[link(name = "m")]  // 链接 libm（数学库）
extern "C" {
    fn sin(x: f64) -> f64;
    fn cos(x: f64) -> f64;
    fn sqrt(x: f64) -> f64;
}

fn safe_sin(x: f64) -> f64 {
    // SAFETY: sin 是标准数学函数，对任意 f64 输入都有定义
    unsafe { sin(x) }
}

fn safe_sqrt(x: f64) -> f64 {
    // 我们在 safe 层做前置检查
    assert!(x >= 0.0, "sqrt 不能接受负数");
    // SAFETY: 已断言 x >= 0.0，sqrt 不会返回 NaN
    unsafe { sqrt(x) }
}

fn main() {
    use std::f64::consts::PI;
    println!("sin(π/2) = {:.6}", safe_sin(PI / 2.0)); // 1.000000
    println!("cos(0) = {:.6}", unsafe { cos(0.0) });   // 1.000000
    println!("sqrt(2) = {:.6}", safe_sqrt(2.0));        // 1.414214
}

#[link(name = "m")] 告诉链接器链接 libm.so。

用 bindgen 自动生成绑定

手写 FFI 声明很繁琐，尤其是大型 C 库。bindgen 是官方工具，能从 C 头文件自动生成 Rust 绑定。

# Cargo.toml
[build-dependencies]
bindgen = "0.70"

// wrapper.h（C 库头文件）
typedef struct {
    int x;
    int y;
} Point;

int distance(Point a, Point b);

// build.rs
fn main() {
    println!("cargo:rerun-if-changed=wrapper.h");
    
    bindgen::Builder::default()
        .header("wrapper.h")
        .parse_callbacks(Box::new(bindgen::CargoCallbacks::new()))
        .generate()
        .expect("bindgen 生成失败")
        .write_to_file(std::path::PathBuf::from(std::env::var("OUT_DIR").unwrap()).join("bindings.rs"))
        .expect("写入绑定文件失败");
}

// src/lib.rs
include!(concat!(env!("OUT_DIR"), "/bindings.rs"));

fn main() {
    let a = Point { x: 0, y: 0 };
    let b = Point { x: 3, y: 4 };
    let d = unsafe { distance(a, b) };
    println!("distance = {}", d); // 5（如果 C 实现正确）
}

bindgen 处理了 C 结构体、枚举、函数指针、宏常量等，是调用复杂 C 库的标配工具。

从 C 调用 Rust

反过来，也可以把 Rust 代码编译成 C 能调用的库：

// src/lib.rs

// 禁用名称修饰（name mangling），让 C 能找到这个函数
#[no_mangle]
pub extern "C" fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[no_mangle]
pub extern "C" fn greet(name_ptr: *const std::ffi::c_char) {
    if name_ptr.is_null() {
        println!("Hello, anonymous!");
        return;
    }
    // SAFETY: 调用者保证 name_ptr 是有效的 null 结尾字符串
    let name = unsafe {
        std::ffi::CStr::from_ptr(name_ptr)
            .to_str()
            .unwrap_or("unknown")
    };
    println!("Hello, {}!", name);
}

# Cargo.toml
[lib]
crate-type = ["cdylib"]  # 编译为动态库 .so/.dll/.dylib

生成的 .so 文件可以直接被 C/C++/Python/Go 等语言加载调用。这是 Rust 作为"粘合层"嵌入其他系统的常见模式。

什么时候真的需要 unsafe

很多人刚接触 unsafe 时会走两个极端：要么完全回避，要么滥用。实际项目中，需要自己写 unsafe 的场景比你想象的少得多。

确实需要 unsafe 的情况

底层数据结构：实现链表、树、图等需要手动管理指针的数据结构。标准库的 LinkedList、BTreeMap 都有 unsafe 实现。

性能关键路径：比如游戏引擎中的批量内存操作，避免边界检查的开销：

fn sum_unchecked(data: &[f32]) -> f32 {
    let mut total = 0.0f32;
    // 在性能测试证明有瓶颈后才考虑
    for i in 0..data.len() {
        // SAFETY: i 在 0..data.len() 范围内，get_unchecked 安全
        total += unsafe { *data.get_unchecked(i) };
    }
    total
}

FFI 边界：如上所述，所有 C 函数调用都需要 unsafe。

实现标准库缺少的功能：比如某些平台特定的系统调用。

通常不需要 unsafe 的情况

遇到下面这些需求，先想想有没有 safe 替代方案：

  • 共享可变状态：想在多处持有可变引用？借用检查器拦住了？先试试 Arc<Mutex<T>> 或 Arc<RwLock<T>>。它们内部有 unsafe，但对外是 safe 接口。
  • 多线程通信：不需要共享内存，用 Channel（tokio::sync::mpsc 等）传消息更清晰。
  • 绕过借用检查：借用检查报错不代表需要 unsafe，通常是数据结构设计问题。重新拆分所有权，或用 RefCell<T> 把检查推迟到运行时。
  • 全局可变变量：static mut 要 unsafe，改用 OnceLock<T>（延迟初始化）或 std::sync::atomic::AtomicU64 等原子类型，都是 safe 的。
  • 类型转换：大部分转换用 From/Into，数值截断用 as，内存重解释才需要 transmute——而 transmute 是高危操作，用之前确认真的没有别的办法。

实际项目的比例

一个中等规模的应用项目（如 Web 服务、CLI 工具），通常 unsafe 代码量接近零。unsafe 主要集中在：

  • 底层库（tokio、hyper、crossbeam 等有少量 unsafe 核心）
  • 操作系统绑定层（nix、windows-sys 等）
  • 高性能计算库（ndarray、BLAS 绑定等）

作为应用层开发者，大多数时候你只是在 safe 接口上调用这些库，无需接触 unsafe。

unsafe 代码的审计

当你确实写了 unsafe 代码，或者在 code review 中审查别人的 unsafe，有几个检查点：

指针有效性：是否是空指针？是否已释放？是否对齐？

内存边界：裸指针运算有没有越界？add(n) 中的 n 是否在合法范围？

别名规则：有没有同时存在对同一内存的可变和不可变引用（哪怕是裸指针形式）？

生命周期：从裸指针重建引用时，给出的生命周期是否正确？

并发安全：如果多线程访问，是否有适当的同步？

工具辅助： Miri 是 Rust 的解释器，能在运行时检测 UB。它不是编译器——编译能过不代表没有 UB，Miri 才是真正的"运行时警察"。

rustup component add miri
cargo miri test

Miri 能检测出哪些问题？举几个实际案例：

越界读取（编译正常，Miri 报错）：

fn main() {
    let v = vec![1, 2, 3];
    let p = v.as_ptr();
    // 编译通过，但越界了
    let _ = unsafe { *p.add(10) };
    // Miri: error: Undefined Behavior: out-of-bounds pointer
}

use-after-free（最常见的内存安全问题）：

fn main() {
    let p: *const i32;
    {
        let x = 42i32;
        p = &x as *const i32;
    } // x 在这里 drop
    // 编译通过，但 x 已释放
    let _ = unsafe { *p };
    // Miri: error: use-after-free
}

不对齐访问（特定平台会崩，Miri 都能检测）：

fn main() {
    let bytes = [0u8; 5];
    // 强制把字节数组的第 1 字节地址解释为 u32 指针——未对齐
    let p = bytes[1..].as_ptr() as *const u32;
    let _ = unsafe { *p };
    // Miri: error: accessing memory with alignment 1, but alignment 4 required
}

对于包含 unsafe 的库，建议把 cargo miri test 加入 CI，能拦截大量只在特定环境下崩溃的隐患。

小结

unsafe 不是洪水猛兽，也不是随便用的免死金牌。它是一份精确的责任转移合同：编译器放弃验证，你承担担保。

记住几个核心原则：

  • 五个边界：只有解引用裸指针、调用 unsafe 函数、访问可变静态变量、实现 unsafe trait、访问 union 字段这五件事需要 unsafe
  • 最小化范围：unsafe 块只包含真正必要的代码
  • safe 封装：内部 unsafe，外部暴露 safe 接口，并用 // SAFETY: 注释解释原因
  • FFI 必备：调用 C 库是 unsafe 最常见的合理用途；CString/CStr 处理字符串边界
  • 优先 safe 替代：90% 的"我需要 unsafe"场景其实有更好的 safe 解法
