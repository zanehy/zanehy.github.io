番外 | 与 APISIX 对比：手写还是用成品？

发表于： 2026-04-08 分类于： OpenResty
字数： 2058 阅读：≈ 5分钟 浏览：

走完这 30 篇，你已经能手写一个具备生产能力的 API 网关。但现实中，很多团队会选择直接用 APISIX，而不是从零开始写。这篇来聊清楚这两条路的区别，帮你做出合适的选择。

APISIX 是什么

APISIX 是 Apache 软件基金会的顶级项目，由 API7.ai 主导开发。它的底层正是 OpenResty——Nginx + LuaJIT。

简单说：APISIX 是一个用 OpenResty 写好的、功能完整的 API 网关产品。你在这个系列里手写的路由、鉴权、限流、熔断、插件系统……APISIX 都已经实现好了，而且经过了大规模生产验证。

APISIX 的技术特点：

  • etcd 存储路由：路由、插件配置存在 etcd，实时推送到每个实例，毫秒级生效
  • 插件丰富：80+ 内置插件，覆盖鉴权、限流、可观测性、流量管理各类场景
  • Dashboard：提供 Web 管理界面，可以可视化配置路由和插件
  • Admin API：RESTful 接口管理一切，方便自动化集成
  • 高性能：单核可处理数万 RPS，延迟极低

功能对比

功能手写网关APISIX
路由匹配自己实现内置，支持前缀/精确/正则/域名
JWT 鉴权自己实现jwt-auth 插件，开箱即用
API Key自己实现key-auth 插件
限流自己实现limit-req/limit-count/limit-conn 三种
熔断自己实现api-breaker 插件
请求改写自己实现proxy-rewrite/response-rewrite 插件
灰度发布自己实现traffic-split 插件
日志自己实现http-logger/kafka-logger/clickhouse-logger 等
监控自己实现prometheus 插件，内置 Grafana Dashboard
配置热更新Redis + 定时器etcd 推送，毫秒级
管理界面无Dashboard + Admin API
负载均衡自己实现轮询/加权/一致性哈希/EWMA
服务发现无支持 Nacos/Consul/Eureka/K8s

可以看到，APISIX 在功能完整性上全面领先手写方案。那手写还有意义吗？

手写网关的适用场景

深度定制的业务逻辑

当你的网关需要做非常特殊的事情——比如私有的签名协议、特殊的流量染色规则、与内部系统深度集成——用 APISIX 的插件系统可能不够灵活，自己写反而更直接。

超轻量部署

APISIX 依赖 etcd（一个分布式 KV 存储），在资源受限的环境（嵌入式设备、边缘节点）里部署有一定门槛。手写的网关只需要 OpenResty + Redis，更轻。

学习和理解原理

这也是这个系列存在的意义——不是为了在生产上用手写网关替代 APISIX，而是通过手写理解底层原理。理解了原理，用 APISIX 时才知道每个配置在做什么，出了问题才知道去哪里找。

已有 OpenResty 基础设施

如果团队已经有大量运行在 OpenResty 上的业务逻辑（Lua 代码），新增网关功能在现有基础上扩展比引入新系统成本更低。

直接用 APISIX 的适用场景

快速落地

需要在短时间内把 API 网关跑起来，APISIX 开箱即用，配置一下就能用，不需要花几周写代码。

功能需求标准化

大多数网关需求（鉴权、限流、日志、监控）都是标准化的，APISIX 的插件已经覆盖，没必要重复造轮子。

运维友好

APISIX 有 Dashboard、Admin API、完善的文档和社区，运维接手成本低。手写的方案只有你们自己懂。

云原生环境

APISIX 对 Kubernetes 支持很好（Ingress Controller、Gateway API），在云原生体系里集成顺畅。

两者结合使用

现实中很多团队是这样的：

  • 标准流量：走 APISIX，用它的内置插件处理通用的鉴权、限流、日志
  • 特殊逻辑：在 APISIX 里写自定义插件（本质上还是 OpenResty Lua 代码）

理解了 OpenResty，写 APISIX 自定义插件就是顺水推舟：

-- APISIX 自定义插件结构
local plugin_name = "my-custom-plugin"

local schema = {
    type = "object",
    properties = {
        key = { type = "string" },
    },
    required = { "key" },
}

local _M = {
    version  = 0.1,
    priority = 2000,
    name     = plugin_name,
    schema   = schema,
}

function _M.access(conf, ctx)
    -- 和我们手写插件的 access 方法一模一样
    local key = conf.key
    ngx.log(ngx.INFO, "custom plugin key: ", key)
end

return _M

APISIX 插件和我们手写的插件接口几乎一样，学过这个系列，上手 APISIX 插件开发没有任何障碍。

系列总结

30 篇走下来，从 OpenResty 是什么，到能手写一个生产级 API 网关，覆盖了：

  • Lua 语言基础（table、模块、LuaJIT）
  • OpenResty 核心 API（ngx.*、cosocket、定时器）
  • 中间件集成（Redis、MySQL、HTTP 客户端、多级缓存、日志）
  • 网关完整实现（路由、鉴权、限流、熔断、改写、插件系统）
  • 工程化（测试、调优、部署、监控）

这个系列的目标从来不是"让你写一个比 APISIX 更好的网关"，而是让你真正理解 API 网关在做什么。带着这份理解，无论是用 APISIX、Kong、还是继续用 OpenResty 自研，都会更加得心应手。
