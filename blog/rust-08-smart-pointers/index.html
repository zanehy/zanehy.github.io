08 | 智能指针——当引用不够用时

发表于： 2026-04-22 分类于： Rust
字数： 2718 阅读：≈ 6分钟 浏览：

前两篇，我们深入理解了借用和生命周期。Rust 的引用机制很强大，但有一个根本限制：引用不拥有数据。

当你需要在多个地方共享同一份数据的所有权，或者需要在堆上存储数据而不想手动管理内存时，普通引用就不够用了。这时候，智能指针登场。

什么是智能指针

先说普通指针。在 C/C++ 里，指针就是一个数字，存着内存地址，没有任何附加功能，忘记释放就内存泄漏，多次释放就崩溃。

智能指针在指针的基础上，附加了"智能"——通过实现 Drop trait 在离开作用域时自动清理，通过实现 Deref trait 让你像使用引用一样使用它。

在 Rust 中，String 和 Vec<T> 本质上都是智能指针。本篇重点讲四个核心类型：Box<T>、Rc<T>、Arc<T>、RefCell<T>。

Box<T>：最简单的堆分配

基本用法 

fn main() {
    let b = Box::new(5);
    println!("b = {}", b); // 像普通变量一样使用
} // b 离开作用域，堆内存自动释放

Box<T> 把数据放在堆上，自身（一个指针）存在栈上。当 Box 离开作用域，堆上的数据也被释放。

使用场景一：递归类型

Rust 在编译期需要知道每个类型的大小。递归类型（自引用）会导致大小无法确定：

// 编译错误：recursive type has infinite size
enum List {
    Cons(i32, List),
    Nil,
}

用 Box 包一层，把"未知大小"变成"指针大小"（固定 8 字节）：

enum List {
    Cons(i32, Box<List>),
    Nil,
}

fn main() {
    let list = List::Cons(1,
        Box::new(List::Cons(2,
            Box::new(List::Cons(3,
                Box::new(List::Nil))))));
}

使用场景二：Trait 对象

当你需要存储实现了某个 trait 的不同类型时：

trait Animal {
    fn speak(&self);
}

struct Dog;
struct Cat;

impl Animal for Dog {
    fn speak(&self) { println!("Woof!"); }
}

impl Animal for Cat {
    fn speak(&self) { println!("Meow!"); }
}

fn main() {
    let animals: Vec<Box<dyn Animal>> = vec![
        Box::new(Dog),
        Box::new(Cat),
    ];

    for animal in &animals {
        animal.speak();
    }
}

Box<dyn Animal> 是一个 trait 对象，编译期不知道具体类型，运行时通过虚函数表分发。

使用场景三：大数据转移所有权

当数据很大，直接移动（拷贝栈上数据）代价高时，先装箱再传指针：

// 一个大结构体
struct BigData {
    data: [u8; 1024 * 1024], // 1MB
}

fn process(data: Box<BigData>) {
    // 传的是指针（8字节），不是 1MB 数据
}

Rc<T>：共享所有权（单线程）

引用计数的思路

有时候，一份数据需要被多个地方"共同拥有"，不确定谁最后离开。这就像办公室里的咖啡机——大家都在用，谁都不是唯一的主人，最后一个走的人才关机。

Rc<T>（Reference Counted）通过引用计数解决这个问题：每克隆一个 Rc，计数加一；每次 Rc 离开作用域，计数减一；计数归零时，数据释放。

use std::rc::Rc;

fn main() {
    let a = Rc::new(String::from("hello"));
    println!("count after creating a = {}", Rc::strong_count(&a)); // 1

    let b = Rc::clone(&a); // 不是深拷贝，只是增加引用计数
    println!("count after creating b = {}", Rc::strong_count(&a)); // 2

    {
        let c = Rc::clone(&a);
        println!("count after creating c = {}", Rc::strong_count(&a)); // 3
    } // c 离开作用域，计数减一

    println!("count after c goes out of scope = {}", Rc::strong_count(&a)); // 2
}

注意 Rc::clone(&a) 和 a.clone() 的区别：前者语义上明确表示"只增加引用计数"，代价极低；后者可能意味着深拷贝（取决于类型实现）。使用 Rc 时习惯用 Rc::clone。

Rc 只能用于单线程

Rc<T> 的引用计数操作不是线程安全的，Rust 编译器会阻止你跨线程使用它：

use std::rc::Rc;
use std::thread;

fn main() {
    let a = Rc::new(5);
    // 编译错误：`Rc<i32>` cannot be sent between threads safely
    thread::spawn(move || {
        println!("{}", a);
    });
}

跨线程需要用 Arc<T>。

Arc<T>：共享所有权（多线程）

Arc<T>（Atomically Reference Counted）是 Rc<T> 的线程安全版本，用原子操作更新引用计数。

use std::sync::Arc;
use std::thread;

fn main() {
    let data = Arc::new(vec![1, 2, 3]);

    let mut handles = vec![];

    for i in 0..3 {
        let data = Arc::clone(&data);
        let handle = thread::spawn(move || {
            println!("Thread {}: {:?}", i, data);
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }
}

每个线程都克隆一个 Arc，实际指向同一份数据，直到所有线程结束后数据才释放。

Arc 只解决了共享，没有解决修改

Arc<T> 让多个线程能读同一份数据，但数据本身是不可变的。要在多线程中修改共享数据，需要配合 Mutex<T>：

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            let mut num = counter.lock().unwrap();
            *num += 1;
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("Result: {}", *counter.lock().unwrap()); // 10
}

Arc<Mutex<T>> 是 Rust 并发编程中最常见的组合，基本上是"多线程共享可变状态"的标准答案。

RefCell<T>：运行时借用检查

内部可变性

Rust 的借用规则在编译期检查：你无法通过不可变引用修改数据。但有些场景下，你在逻辑上知道代码是安全的，但编译器无法证明这一点。

RefCell<T> 提供"内部可变性"：允许在持有不可变引用的情况下修改内部数据，但把借用检查推迟到运行时。

use std::cell::RefCell;

fn main() {
    let data = RefCell::new(vec![1, 2, 3]);

    // 不可变借用
    {
        let r = data.borrow();
        println!("{:?}", r);
    }

    // 可变借用
    {
        let mut r = data.borrow_mut();
        r.push(4);
    }

    println!("{:?}", data.borrow()); // [1, 2, 3, 4]
}

borrow() 返回 Ref<T>，borrow_mut() 返回 RefMut<T>，它们在 Drop 时自动释放借用。

违反规则时 panic

运行时借用检查的代价是：如果你违反了借用规则，程序会 panic 而不是编译失败：

use std::cell::RefCell;

fn main() {
    let data = RefCell::new(5);

    let r1 = data.borrow();
    let r2 = data.borrow_mut(); // panic! 已经有不可变借用了
}

所以使用 RefCell 时要格外小心，确保在逻辑上保证不会同时存在冲突借用。

典型场景：Mock 对象

RefCell 最常见的用途是在测试中实现 Mock 对象，或者在不可变接口背后维护内部状态（比如缓存）：

use std::cell::RefCell;

pub trait Notifier {
    fn send(&self, message: &str);
}

struct MockNotifier {
    sent_messages: RefCell<Vec<String>>,
}

impl MockNotifier {
    fn new() -> MockNotifier {
        MockNotifier {
            sent_messages: RefCell::new(vec![]),
        }
    }
}

impl Notifier for MockNotifier {
    fn send(&self, message: &str) {
        // self 是不可变引用，但我们可以修改内部状态
        self.sent_messages.borrow_mut().push(String::from(message));
    }
}

fn main() {
    let mock = MockNotifier::new();
    mock.send("hello");
    mock.send("world");

    println!("Messages: {:?}", mock.sent_messages.borrow());
}

组合使用

实际项目中，这些类型常常组合使用：

Rc<RefCell<T>> — 单线程中，多个地方共享且需要修改：

use std::cell::RefCell;
use std::rc::Rc;

#[derive(Debug)]
struct Node {
    value: i32,
    children: Vec<Rc<RefCell<Node>>>,
}

fn main() {
    let leaf = Rc::new(RefCell::new(Node {
        value: 3,
        children: vec![],
    }));

    let branch = Rc::new(RefCell::new(Node {
        value: 5,
        children: vec![Rc::clone(&leaf)],
    }));

    // 修改 leaf 的值，branch 也能看到变化
    leaf.borrow_mut().value = 10;

    println!("branch: {:?}", branch.borrow());
}

Arc<Mutex<T>> — 多线程中，多个线程共享且需要修改（前面已展示）。

选择决策树

面对具体场景，按这个顺序思考：

需要在堆上存数据，或者递归类型？
  → Box<T>

需要多个地方共享所有权？
  → 单线程：Rc<T>
  → 多线程：Arc<T>

需要通过不可变引用修改内部数据？
  → 单线程：RefCell<T>（配合 Rc 时用 Rc<RefCell<T>>）
  → 多线程：Mutex<T> 或 RwLock<T>（配合 Arc 时用 Arc<Mutex<T>>）

大多数业务代码只需要 Box<T> 和 Arc<Mutex<T>> 就够了。Rc 和 RefCell 在特定场景（树形结构、图、测试 Mock）才会用到。

小结

  • Box<T>：最基础的堆分配，解决递归类型和 trait 对象问题
  • Rc<T>：单线程引用计数，多个地方共享所有权
  • Arc<T>：多线程引用计数，Rc 的线程安全版本
  • RefCell<T>：运行时借用检查，提供内部可变性
  • 生产中最常用的组合：Arc<Mutex<T>>（多线程共享可变状态）

智能指针是 Rust 所有权系统的延伸，理解它们的本质——谁拥有数据、如何释放、借用规则在哪里检查——比记住 API 更重要。
