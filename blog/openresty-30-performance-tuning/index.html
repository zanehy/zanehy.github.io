30 | 性能调优

发表于： 2026-04-08 分类于： OpenResty
字数： 1972 阅读：≈ 4分钟 浏览：

系统上线前，最后一件事是调优和压测。不调优直接上线，性能可能只发挥了三四成；不压测就上线，遇到流量高峰会猝不及防。这篇把 OpenResty 性能调优的关键参数和压测方法系统整理一遍。

worker 数量与 CPU 绑定

worker 数量 

# 自动设置为 CPU 核心数（推荐）
worker_processes auto;

# 或者手动指定
worker_processes 8;

auto 会读取 CPU 核心数并自动设置。对于 I/O 密集型场景（OpenResty 通常是），worker 数量等于 CPU 核心数是最优的——更多的 worker 会增加上下文切换开销，更少则浪费 CPU。

CPU 亲和性绑定

把每个 worker 绑定到固定的 CPU 核心，减少跨核调度，提升 CPU 缓存命中率：

worker_processes 4;
worker_cpu_affinity 0001 0010 0100 1000;
# 4 个 worker 分别绑定到 CPU 0、1、2、3

# 自动绑定（Nginx 1.9.10+）
worker_cpu_affinity auto;

worker 连接数 

events {
    worker_connections 10240;  # 每个 worker 最大连接数
    use epoll;                 # Linux 下用 epoll（默认已选择）
    multi_accept on;           # 一次接受多个新连接
}

系统最大文件描述符数要配合调整：

# /etc/security/limits.conf
nginx   soft    nofile  65535
nginx   hard    nofile  65535

# 或者在 nginx.conf 里
worker_rlimit_nofile 65535;

连接池参数调优

连接池的参数影响深远，设置不合理会导致连接频繁建立销毁，或者资源浪费。

Redis 连接池 

-- 连接池计算公式：
-- pool_size = worker_connections * 平均每请求Redis操作数 * 平均Redis操作耗时 / 平均请求耗时
-- 简化经验值：50～200，根据压测结果调整

red:set_keepalive(
    10000,   -- max_idle_timeout：连接在池中最长空闲时间（ms）
    100      -- pool_size：连接池大小
)

调优方向：

  • pool_size 太小：高并发时连接不够用，新请求等待或建立新连接
  • pool_size 太大：占用 Redis 服务端的连接数配额（注意 Redis 的 maxclients）
  • max_idle_timeout 太小：连接频繁失效，每次都要重建
  • max_idle_timeout 太大：长时间不用的连接占用资源

用 Redis 的 CLIENT LIST 命令监控连接数，压测时观察是否接近 maxclients 上限。

MySQL 连接池 

db:set_keepalive(
    30000,   -- 空闲超时 30 秒（MySQL 默认 wait_timeout 是 8 小时，可以设长）
    20       -- pool_size（MySQL 连接比 Redis 更重，一般设小一点）
)

MySQL 连接池大小要对应 MySQL 的 max_connections 配置，所有 worker 的连接数加起来不能超过这个值。

HTTP 客户端连接池 

httpc:request_uri(url, {
    keepalive_timeout = 60000,  -- 连接保持 60 秒
    keepalive_pool    = 30,     -- 到同一 host 的连接池大小
})

共享内存大小规划

ngx.shared.DICT 的大小在启动时就固定了，用完会 LRU 淘汰旧数据（或写入失败）。

规划方法：估算存储数据量。

http {
    # 路由配置：几百条路由，每条约 500 字节
    # 100 条 × 500 字节 = 50KB，留 10 倍余量
    lua_shared_dict routes       5m;

    # 限流计数器：每个 key 约 100 字节
    # 假设 10 万个 IP/用户，100 字节 × 100000 = 10MB
    lua_shared_dict rate_limit   20m;

    # 应用缓存：根据实际缓存的数据量估算
    lua_shared_dict app_cache    50m;

    # 熔断器状态：每个服务几百字节
    lua_shared_dict circuit_breaker 1m;

    # 日志队列：峰值 QPS × 每条日志大小 × 缓冲时间
    # 1000 QPS × 500 字节 × 5 秒 = 2.5MB
    lua_shared_dict log_queue    10m;
}

监控共享内存使用情况：

location /admin/shm-stats {
    content_by_lua_block {
        local cjson = require "cjson"
        local names = {"routes", "rate_limit", "app_cache", "log_queue"}
        local stats = {}
        for _, name in ipairs(names) do
            local shm = ngx.shared[name]
            if shm then
                stats[name] = {
                    capacity = shm:capacity(),
                    free_space = shm:free_space(),
                }
            end
        end
        ngx.say(cjson.encode(stats))
    }
}

wrk 压测实战

wrk 是一个高性能 HTTP 压测工具，可以用少量线程产生大量并发请求。

安装 

# Ubuntu/Debian
apt-get install wrk

# 从源码编译
git clone https://github.com/wg/wrk.git && cd wrk && make

基本用法 

# 基础压测：2 个线程，100 个并发连接，持续 30 秒
wrk -t2 -c100 -d30s http://localhost:8080/api/test

# 输出示例：
# Running 30s test @ http://localhost:8080/api/test
#   2 threads and 100 connections
#   Thread Stats   Avg      Stdev     Max   +/- Stdev
#     Latency    2.34ms    1.12ms  45.23ms   89.23%
#     Req/Sec   21.45k     2.13k   25.67k    72.00%
#   1284732 requests in 30.03s, 198.45MB read
# Requests/sec:  42784.23
# Transfer/sec:      6.61MB

关键指标：

  • Req/Sec：每秒请求数（吞吐量）
  • Latency：响应延迟（关注 99% 分位）
  • Non-2xx：非 200 响应数量（有则说明有错误）

带请求头和 Body

wrk 支持 Lua 脚本自定义请求：

-- wrk_scripts/auth_api.lua
-- 带 JWT Token 的 POST 请求压测

local token = "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."

wrk.method  = "POST"
wrk.body    = '{"key":"value"}'
wrk.headers["Content-Type"]  = "application/json"
wrk.headers["Authorization"] = "Bearer " .. token

-- 每个请求可以做不同的处理（可选）
function request()
    local path = "/api/v1/user/" .. math.random(1, 1000)
    return wrk.format("GET", path)
end

-- 统计响应
local counter = 0
local errors  = 0
function response(status, headers, body)
    counter = counter + 1
    if status >= 400 then
        errors = errors + 1
    end
end

function done(summary, latency, requests)
    io.write("Total requests: " .. counter .. "\n")
    io.write("Errors: " .. errors .. "\n")
    io.write("Error rate: " .. string.format("%.2f%%", errors/counter*100) .. "\n")
    -- 打印延迟百分位
    for _, p in pairs({50, 75, 90, 99, 99.9}) do
        local n = latency:percentile(p)
        io.write(string.format("  %g%% <= %dμs\n", p, n))
    end
end

wrk -t4 -c200 -d60s -s wrk_scripts/auth_api.lua http://localhost:8080

wrk2：更精准的延迟测量

wrk 的延迟统计有协调遗漏问题（Coordinated Omission），wrk2 修复了这个问题，延迟数据更准确：

# 指定目标 RPS（每秒请求数），测量在该负载下的延迟分布
wrk2 -t4 -c200 -d60s -R10000 http://localhost:8080/api/test
# -R10000：目标 10000 RPS

压测策略

逐步加压，找到系统的极限：

# 第一轮：找大概上限
wrk -t2 -c50  -d10s http://localhost:8080/api/test  # 低并发热身
wrk -t4 -c200 -d10s http://localhost:8080/api/test  # 中等并发
wrk -t8 -c500 -d10s http://localhost:8080/api/test  # 高并发

# 第二轮：在极限附近细化，找到错误率开始上升的点
wrk -t4 -c300 -d30s http://localhost:8080/api/test
wrk -t4 -c400 -d30s http://localhost:8080/api/test
wrk -t4 -c500 -d30s http://localhost:8080/api/test

压测时同时观察：

  • top：CPU 使用率，是否打满
  • ss -s：TCP 连接数统计
  • tail -f logs/error.log：有没有错误产生
  • Redis/MySQL 的连接数和 QPS

其他优化点

开启 gzip 压缩

gzip on;
gzip_types application/json text/plain;
gzip_min_length 1024;
gzip_comp_level 6;

调整 TCP 参数

# 减少 TCP 握手次数
tcp_nopush  on;
tcp_nodelay on;

# keepalive 连接复用
keepalive_timeout 65;
keepalive_requests 1000;

关闭不必要的日志

压测期间可以关闭 access_log，减少 I/O 开销：

access_log off;
# 或者写到 /dev/null
access_log /dev/null;

小结

性能调优的几个关键点：

  • worker 数量：等于 CPU 核心数，配合 worker_cpu_affinity auto 绑定
  • 连接池：Redis 100～200，MySQL 20～50，根据压测结果调整
  • 共享内存：按数据量估算，留 5～10 倍余量，用监控接口观察使用率
  • 压测：wrk 快速找上限，wrk2 精确测延迟，逐步加压找极限点
