08 | LuaJIT 与性能

发表于： 2026-03-12 分类于： OpenResty
字数： 2136 阅读：≈ 5分钟 浏览：

OpenResty 之所以能在高并发场景下打出不俗的性能，Nginx 的事件驱动模型是一方面，另一方面就是 LuaJIT。这篇文章来讲清楚 LuaJIT 是什么、它为什么快、以及在实际开发中有哪些性能陷阱值得注意。

LuaJIT 与标准 Lua 的区别

标准 Lua（PUC-Rio Lua）是一个解释器，每次执行都要把 Lua 代码翻译成字节码再逐条解释运行。

LuaJIT 则是一个带有即时编译（JIT，Just-In-Time Compilation）能力的实现。它能在运行时分析热点代码，把频繁执行的路径直接编译成机器码，性能可以接近 C 语言。

除了 JIT，LuaJIT 还有：

  • 更快的解释器：即便 JIT 没有触发，LuaJIT 的解释器本身也比标准 Lua 快 2～5 倍
  • FFI 库：可以直接调用 C 函数和使用 C 数据结构，无需编写 C 扩展
  • 更好的数字处理：LuaJIT 默认使用双精度浮点，数学运算路径更短

在 OpenResty 里，LuaJIT 是默认且唯一支持的 Lua 运行时。

JIT 编译是怎么工作的

LuaJIT 的 JIT 编译器叫 tracing JIT。它的工作方式是：

  1. 先用解释器执行代码，同时记录执行路径（trace）
  2. 当某条路径被执行超过阈值（默认 56 次），触发编译
  3. 把这条热路径编译成优化过的机器码
  4. 后续执行直接走机器码，跳过解释阶段

这意味着 JIT 的收益主要体现在循环和频繁调用的函数上。短命的代码（只执行一两次就结束）JIT 来不及发挥。

查看 JIT 编译状态

LuaJIT 提供了 jit 库可以控制和查看 JIT 状态：

-- 查看 JIT 是否开启
print(jit.status())   -- true / false

-- 关闭 JIT（调试用）
jit.off()

-- 开启 JIT
jit.on()

-- 针对特定函数关闭 JIT
jit.off(some_function)

在 OpenResty 里，可以用 luajit -jdump 工具查看哪些代码被编译了（生产环境一般不用，开发调试时有用）。

FFI：直接调用 C 库

FFI（Foreign Function Interface）是 LuaJIT 最强大的特性之一。它让你可以直接在 Lua 代码里声明和调用 C 函数，不需要写 C 扩展模块。

基本用法 

local ffi = require "ffi"

-- 声明 C 函数原型
ffi.cdef[[
    int printf(const char *fmt, ...);
    double sqrt(double x);
    
    typedef struct {
        int x;
        int y;
    } Point;
]]

-- 直接调用 C 标准库函数
ffi.C.printf("Hello from C! sqrt(2) = %f\n", ffi.C.sqrt(2))

-- 创建 C 结构体
local p = ffi.new("Point", {x = 3, y = 4})
print(p.x, p.y)  -- 3  4

调用自定义共享库 

local ffi = require "ffi"

ffi.cdef[[
    int add(int a, int b);
    void process_data(const char *data, size_t len);
]]

local mylib = ffi.load("/path/to/libmylib.so")

print(mylib.add(10, 20))  -- 30

FFI 的性能优势

FFI 调用比传统的 Lua C API 扩展快很多，原因有两点：

  1. 无需类型转换开销：FFI 直接操作 C 数据类型，不需要在 Lua 值和 C 值之间反复转换
  2. 可以被 JIT 内联：JIT 编译器理解 FFI 调用，可以把整个调用链内联优化

OpenResty 的很多核心模块（包括 resty.core）都通过 FFI 重写了原来的 C 扩展实现，性能提升显著。

热点代码优化技巧

技巧一：局部化全局变量

每次访问全局变量都需要查哈希表，而局部变量直接走寄存器。

-- 慢：每次循环都查全局
for i = 1, 1000000 do
    local x = math.sin(i)
end

-- 快：局部化后 JIT 可以优化
local sin = math.sin
for i = 1, 1000000 do
    local x = sin(i)
end

在模块顶部局部化常用的全局函数是个好习惯：

local type = type
local tostring = tostring
local string_format = string.format
local table_insert = table.insert
local ngx_log = ngx.log
local ngx_ERR = ngx.ERR

技巧二：避免在循环里创建 table 

-- 慢：每次循环分配新 table，增加 GC 压力
for i = 1, 10000 do
    local t = {x = i, y = i * 2}
    process(t)
end

-- 更好：复用 table
local t = {}
for i = 1, 10000 do
    t.x = i
    t.y = i * 2
    process(t)
end

技巧三：字符串拼接用 table.concat

Lua 的字符串是不可变的，每次 .. 拼接都会创建新字符串。

-- 慢：O(n²) 的字符串拼接
local result = ""
for i = 1, 1000 do
    result = result .. tostring(i)
end

-- 快：先收集再一次性拼接
local parts = {}
for i = 1, 1000 do
    parts[#parts + 1] = tostring(i)
end
local result = table.concat(parts)

技巧四：使用 table.new 预分配

LuaJIT 提供了 table.new 可以预分配 table 容量，减少动态扩容：

local table_new = require "table.new"

-- 预分配一个有 100 个数组元素、10 个哈希元素的 table
local t = table_new(100, 10)

在 OpenResty 里处理大量数据时，预分配可以减少内存碎片和 GC 停顿。

常见性能陷阱

陷阱一：NYI（Not Yet Implemented）

LuaJIT 的 JIT 编译器无法编译所有 Lua 代码，某些操作被标记为 NYI，遇到后会退出 JIT 编译的 trace，回退到解释执行。

常见的 NYI 操作：

  • string.format 中的某些格式符（%g 等）
  • pairs() 迭代（ipairs 可以被 JIT）
  • pcall 和 xpcall（在某些版本中）
  • table.sort 的某些用法
  • tostring() 对某些类型

查看 NYI 列表：https://wiki.luajit.org/NYI

在热路径里遇到 NYI，可以考虑用 FFI 或者换一种写法规避。

陷阱二：过度使用 pcall

pcall 在 LuaJIT 里的开销比普通函数调用大，而且会中断 JIT trace。如果你在一个高频路径里频繁用 pcall 包裹每一个小操作，性能会受影响。

更好的做法是：只在真正需要捕获错误的边界用 pcall，内部逻辑用返回值传递错误。

-- OpenResty 推荐的错误处理风格（不用 pcall）
local result, err = some_operation()
if not result then
    ngx.log(ngx.ERR, "failed: ", err)
    return ngx.exit(500)
end

陷阱三：大量使用正则表达式

Lua 的内置模式匹配（string.find、string.match）每次都从头编译，对于高频调用，推荐用 ngx.re.* 系列（底层用 PCRE，支持编译缓存）：

-- 慢：每次都重新编译模式
local ok = string.find(uri, "/api/v%d+/")

-- 快：PCRE 编译结果会被缓存（加 "jo" 选项）
local ok = ngx.re.find(uri, [[/api/v\d+/]], "jo")

jo 选项中：j 启用 JIT（PCRE JIT），o 开启缓存。

陷阱四：忽视 GC 压力

LuaJIT 的 GC 是增量式的，但大量临时对象依然会造成停顿。在 OpenResty 里，可以手动调整 GC 参数：

-- 在 init_by_lua 里调整 GC 步长，减少每次 GC 的停顿时间
collectgarbage("setstepmul", 100)  -- 默认 200，调小减少每步工作量

更重要的是在业务代码里减少不必要的临时对象创建。

小结

LuaJIT 是 OpenResty 性能的重要支柱，理解它有助于写出更高效的代码：

  • JIT 编译针对热路径，循环和高频函数收益最大
  • FFI 可以直接调用 C 库，性能接近原生 C
  • 局部化全局变量、避免循环内创建 table、用 table.concat 拼接字符串——这些是基础优化习惯
  • 注意 NYI 操作，避免在热路径里触发回退
