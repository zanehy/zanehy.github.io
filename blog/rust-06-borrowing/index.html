06 | 借用与引用——不转移所有权，也能用你的数据

发表于： 2026-04-20 分类于： Rust
字数： 4179 阅读：≈ 9分钟 浏览：

上一篇讲到一个尴尬的问题：把值传给函数，所有权就转移了，函数用完你就再也拿不回来了。

fn print_length(s: String) {
    println!("长度是：{}", s.len());
}  // s 在这里被 drop

fn main() {
    let name = String::from("Alice");
    print_length(name);
    // println!("{}", name);  // 编译错误！name 已经没了
}

为了"打印一下长度"就把数据永久交出去，这显然不合理。实际写代码时，大多数情况我们只是想让函数"看一眼"数据，并不想转移所有权。

Rust 为此提供了借用（Borrowing）机制——通过引用（Reference），让别人使用你的数据，但所有权还在你这里。

图书馆借书：引用的直觉模型

想象你有一本书，朋友想看。

有两种方式：

  1. 把书送给朋友（Move）——书就不是你的了
  2. 借给朋友看（引用）——你还是书的主人，朋友看完还你

引用就是第二种方式。你把数据"借"给别人用，借的人用完了，数据还在你这里。

Rust 里，引用用 & 符号表示：

fn print_length(s: &String) {  // 接收引用，不拿走所有权
    println!("长度是：{}", s.len());
}

fn main() {
    let name = String::from("Alice");
    print_length(&name);           // 传引用，&name 就是"把 name 借出去"
    println!("name 还在：{}", name); // 没问题，所有权从没离开过
}

&name 的意思是：创建一个指向 name 的引用，而不是转移 name 本身。

引用在内存里长什么样 

name 变量（栈）：
┌─────────────────────────┐
│  ptr ───────────────────┼──→  堆："Alice"
│  len: 5                 │
│  cap: 5                 │
└─────────────────────────┘
         ↑
         │
s（引用，也在栈上）：
┌─────────────────────────┐
│  ptr ───────────────────┘   （指向 name，而不是直接指向堆）
└─────────────────────────┘

引用本身是个指针，指向原始数据的位置。它很轻量，不复制任何数据。

引用有一个关键约束：引用的生命周期不能超过它所指向的数据。书还没被销毁，借书才有意义；数据都没了，引用指向哪里？这个问题 Rust 在编译期帮你检查，第07篇讲生命周期时会细说。

不可变引用：只能看，不能改

默认情况下，引用是不可变的——你可以读数据，但不能修改它。

fn main() {
    let s = String::from("hello");
    let r = &s;  // 不可变引用

    println!("{}", r);    // 可以读
    // r.push_str(" world");  // 编译错误！不能通过不可变引用修改数据
}

类比：你借给朋友一本书，朋友只能看，不能在上面乱写乱画。

不可变引用可以同时存在多个——多个人同时看同一本书，完全没问题：

fn main() {
    let s = String::from("hello");

    let r1 = &s;  // 第一个不可变引用
    let r2 = &s;  // 第二个不可变引用
    let r3 = &s;  // 第三个也行

    println!("{} {} {}", r1, r2, r3);  // 全部有效，可以同时使用
}

多个人同时读一本书，互不干扰，没有任何问题。

可变引用：借出去，允许修改

如果想让函数修改你的数据，需要可变引用，用 &mut 表示：

fn add_world(s: &mut String) {  // 接收可变引用
    s.push_str(" world");       // 可以修改
}

fn main() {
    let mut s = String::from("hello");  // 变量本身必须是 mut
    add_world(&mut s);                  // 传可变引用
    println!("{}", s);                  // "hello world"
}

注意两个地方都要写 mut：

  • let mut s：声明变量是可变的
  • &mut s：创建可变引用

类比：你借给朋友一支笔和一本空白本子，并明确说"你可以在上面写"。但你只能借给一个人——如果两个人同时拿着笔往同一本子上写，内容就乱了。

借用规则：Rust 最核心的安全保证

Rust 对引用的使用有严格规定，可以用一句话概括：

同一时刻，要么只有多个不可变引用，要么只有一个可变引用，二者不能共存。

用表格理解：

场景是否允许
多个不可变引用同时存在允许
一个可变引用允许
一个可变引用 + 一个不可变引用同时存在不允许
多个可变引用同时存在不允许

为什么要有这个限制？

想象你在一个共享文档上工作：

  • 多个人同时只读：完全没问题，大家看到的内容一致
  • 一个人编辑：没问题，但这时候别人不能读——因为文档正在被修改，读到的可能是半截的内容
  • 两个人同时编辑：冲突，内容会乱

Rust 的借用规则就是把这个常识变成了编译期强制检查。

规则一：不能同时有可变引用和不可变引用

fn main() {
    let mut s = String::from("hello");

    let r1 = &s;      // 不可变引用
    let r2 = &mut s;  // 编译错误！r1 还活着，不能创建可变引用

    println!("{} {}", r1, r2);
}

错误信息：cannot borrow 's' as mutable because it is also borrowed as immutable

规则二：不能同时有多个可变引用

fn main() {
    let mut s = String::from("hello");

    let r1 = &mut s;  // 第一个可变引用
    let r2 = &mut s;  // 编译错误！已经有一个可变引用了

    println!("{} {}", r1, r2);
}

错误信息：cannot borrow 's' as mutable more than once at a time

引用的作用域：比你想的要短

很多初学者被借用规则搞得很困惑，觉得处处受限。但 Rust 的编译器其实很聪明——引用的作用域在它最后一次被使用时就结束了，而不是等到花括号结束。

fn main() {
    let mut s = String::from("hello");

    let r1 = &s;      // 不可变引用
    let r2 = &s;      // 第二个不可变引用
    println!("{} {}", r1, r2);
    // r1 和 r2 在这之后不再被使用，它们的作用域在这里结束

    let r3 = &mut s;  // 可以！r1 和 r2 已经"用完"了
    r3.push_str(" world");
    println!("{}", r3);
}

这个特性叫做 NLL（Non-Lexical Lifetimes，非词法生命周期），编译器分析引用的实际使用位置，而不是机械地看花括号范围。实际写代码时没那么多限制。

实际场景：为什么借用规则能防止 bug

借用规则不是为了让你难受，是为了在编译期拦住一类经典的运行时 bug——数据竞争（Data Race）。

数据竞争发生在三个条件同时满足时：

  1. 两个或更多指针同时访问同一数据
  2. 至少一个指针在写数据
  3. 没有同步机制

在多线程程序里，这是极难调试的 bug，往往不稳定复现。Rust 的借用规则从根本上杜绝了这种情况——编译时就不让你写出来。

单线程下也有类似的问题。比如这个 Python 反例：

# Python 代码
items = [1, 2, 3, 4, 5]
for i, item in enumerate(items):
    if item % 2 == 0:
        items.remove(item)  # 边遍历边删除，结果不可预期
print(items)  # [1, 3, 5]？其实是 [1, 3, 5]，但原因很微妙，容易出错

Rust 不允许你在持有不可变引用的同时修改数据，这类问题根本无从发生。

字符串切片：引用的常见形态

聊完引用，顺带说一个非常常用的类型——字符串切片（&str）。

你可能注意到 Rust 里有两种字符串：String 和 &str，一直搞不清楚区别。

  • String：在堆上分配的、可增长的字符串，你拥有它的所有权
  • &str：对某段字符串数据的借用，只是一个引用，不拥有数据
fn main() {
    let s = String::from("hello world");

    // 切片：借用 s 的一部分
    let hello = &s[0..5];   // "hello"
    let world = &s[6..11];  // "world"

    println!("{} {}", hello, world);

    // 字符串字面量本身就是 &str
    let literal: &str = "I am a literal";  // 存在程序的只读数据段里
}

切片在内存里长这样：

s（String，栈上）：
┌─────────────────────────┐
│  ptr ───────────────────┼──→  堆："hello world"
│  len: 11                │      0123456789...
│  cap: 11                │
└─────────────────────────┘

hello（&str，栈上）：
┌─────────────────────────┐
│  ptr ───────────────────┼──→  指向堆上第 0 个字节
│  len: 5                 │
└─────────────────────────┘

world（&str，栈上）：
┌─────────────────────────┐
│  ptr ───────────────────┼──→  指向堆上第 6 个字节
│  len: 5                 │
└─────────────────────────┘

切片不复制任何数据，只是记录"从哪里开始，多长"。非常轻量。

函数参数用 &str 还是 &String？

写函数时，如果参数是字符串，优先用 &str：

// 推荐：接收 &str，更灵活
fn greet(name: &str) {
    println!("Hello, {}!", name);
}

fn main() {
    let s = String::from("Alice");
    greet(&s);           // String 可以自动转换为 &str
    greet("Bob");        // 字面量本来就是 &str，直接传
}

&str 比 &String 更通用，能接受字符串字面量，也能接受 String 的引用（Rust 会自动做转换）。

悬空引用：Rust 绝对不让你写出来的错误

有一类错误在 C/C++ 里极其常见，叫悬空引用（Dangling Reference）——引用指向的数据已经被释放了，但引用还在用。

// C 代码（危险！）
char* get_string() {
    char s[] = "hello";  // 局部变量，在栈上
    return s;            // 返回局部变量的地址
}                        // 函数返回后，s 被销毁，地址指向垃圾数据

int main() {
    char* p = get_string();
    printf("%s\n", p);  // 未定义行为，可能崩溃，可能打印乱码
}

Rust 在编译期完全拦住了这种代码：

fn get_string() -> &String {  // 编译错误！
    let s = String::from("hello");
    &s  // 想返回 s 的引用
}  // 但 s 在这里被 drop，引用将指向已释放的内存

编译器报错：missing lifetime specifier 或 returns a reference to data owned by the current function

修复方式：把所有权直接返回出去，不要返回引用：

fn get_string() -> String {  // 返回值本身，转移所有权
    let s = String::from("hello");
    s  // 所有权转移给调用者，数据不会被释放
}

fn main() {
    let s = get_string();
    println!("{}", s);  // 完全没问题
}

新手常见误区

误区一：忘记在变量声明时加 mut

// 错误代码
let s = String::from("hello");
let r = &mut s;  // 编译错误！s 本身不是 mut

// 修复
let mut s = String::from("hello");
let r = &mut s;  // 正确

变量默认是不可变的，要创建可变引用，变量本身必须先声明为 mut。

误区二：可变引用和不可变引用同时存在

// 错误代码
fn main() {
    let mut s = String::from("hello");
    let r1 = &s;       // 不可变引用
    s.push_str(" world");  // 编译错误！修改 s 相当于通过可变方式访问，而 r1 还活着
    println!("{}", r1);
}

// 修复：先用完不可变引用，再修改
fn main() {
    let mut s = String::from("hello");
    let r1 = &s;
    println!("{}", r1);  // r1 最后一次使用，作用域结束

    s.push_str(" world");  // 现在可以修改了
    println!("{}", s);
}

误区三：切片索引越界（运行时 panic，不是编译错误）

fn main() {
    let s = String::from("hello");
    let slice = &s[0..10];  // 编译通过，但运行时 panic！s 只有 5 个字节
}

Rust 的字符串切片索引是字节位置，不是字符位置。对于纯 ASCII 字符串没问题，但遇到中文等多字节字符要小心：

fn main() {
    let s = String::from("你好");
    // let slice = &s[0..1];  // panic！"你"占 3 个字节，不能在字节 1 处切割

    // 正确方式：用 chars() 按字符遍历
    for c in s.chars() {
        println!("{}", c);
    }
}

小结

借用与引用是 Rust 日常编程用得最多的机制，核心概念只有几条：

  • 引用（&T）：借用数据，不转移所有权；像图书馆借书，用完还回去
  • 不可变引用：只能读，可以同时存在多个
  • 可变引用（&mut T）：可以读写，同一时刻只能有一个，且不能与不可变引用共存
  • 借用规则：多个只读 OR 一个可写，二选一，编译器强制检查
  • 字符串切片（&str）：对字符串数据的借用，函数参数优先用 &str
  • 悬空引用：Rust 在编译期完全拒绝，不让你写出来

借用规则初看觉得严，但它消灭了一整类运行时 bug——数据竞争、use-after-free、悬空指针，这些在 C/Python 里折磨程序员的问题，在 Rust 里编译都过不了。

但还有一个问题没解决：引用到底能活多久？如果函数返回一个引用，编译器怎么知道这个引用安不安全？这就是下一篇的主题——生命周期（Lifetime）：Rust 如何在编译期追踪每个引用的有效范围。
