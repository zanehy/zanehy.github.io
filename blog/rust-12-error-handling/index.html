12 | 错误处理体系

发表于： 2026-04-24 分类于： Rust
字数： 3775 阅读：≈ 8分钟 浏览：

每门语言都有自己处理错误的哲学。Python 和 Java 用异常——出错时向上抛，谁想处理谁 catch，不想处理就继续往上冒。Go 用多返回值——每个函数都返回 (result, error)，调用方强制检查。

Rust 的答案更彻底：错误是类型，必须被处理，不能被忽视。编译器会在你忽略错误时发出警告，在你错误地传播时拒绝编译。

这带来了一个结果：Rust 程序很少有"运行时突然崩溃"的意外——因为大多数错误在编译期就被逼着处理掉了。

两类错误：panic 和 Result

Rust 把错误分成两类，处理方式截然不同。

不可恢复错误：panic!

panic! 代表"程序遇到了不应该发生的事"，立即终止当前线程，打印错误信息和调用栈。

fn main() {
    let v = vec![1, 2, 3];
    println!("{}", v[10]); // panic: index out of bounds
}

什么时候该用 panic？

  • 测试代码：unwrap()、expect() 在测试里完全合理
  • 原型和脚本：先让代码跑起来，后面再处理错误
  • 违反不变式：调用方传入了不合法的参数（程序员的错误，不是用户的错误）
  • 确实无法继续：比如启动时配置文件格式完全错误

什么时候不该用 panic？

  • 库代码：永远不要在库里 panic，把错误返回给调用方
  • 可预期的失败：网络超时、文件不存在、用户输入非法——这些都是正常情况，用 Result
// 坏的写法——库函数里 panic
fn parse_config(s: &str) -> Config {
    serde_json::from_str(s).unwrap() // 调用方无法处理这个 panic
}

// 好的写法——返回 Result
fn parse_config(s: &str) -> Result<Config, serde_json::Error> {
    serde_json::from_str(s)
}

Result：把错误变成类型

Result<T, E> 是 Rust 错误处理的核心：

enum Result<T, E> {
    Ok(T),   // 成功，携带结果值
    Err(E),  // 失败，携带错误值
}

调用可能失败的函数，返回值就是 Result，你必须处理两种情况：

use std::fs;

fn main() {
    let result = fs::read_to_string("config.txt");

    match result {
        Ok(content) => println!("文件内容: {}", content),
        Err(e) => println!("读取失败: {}", e),
    }
}

常用处理方法

match 是最通用的处理方式，但有时候太啰嗦。标准库提供了一系列简洁方法：

fn main() {
    // unwrap：成功返回值，失败 panic（只在确定不会失败或测试时用）
    let content = fs::read_to_string("config.txt").unwrap();

    // expect：和 unwrap 一样，但 panic 时打印自定义消息
    let content = fs::read_to_string("config.txt")
        .expect("无法读取配置文件");

    // unwrap_or：失败时返回默认值
    let content = fs::read_to_string("config.txt")
        .unwrap_or_else(|_| String::from("{}"));

    // map：对成功值做变换，错误原样传递
    let length: Result<usize, _> = fs::read_to_string("config.txt")
        .map(|s| s.len());

    // map_err：对错误值做变换，成功原样传递
    let result = fs::read_to_string("config.txt")
        .map_err(|e| format!("IO错误: {}", e));

    // ok：把 Result 转成 Option，丢弃错误信息
    let maybe_content: Option<String> = fs::read_to_string("config.txt").ok();
}

?：错误传播的语法糖

当你写一个可能失败的函数，经常需要调用其他可能失败的函数，然后把错误往上传：

use std::fs;
use std::io;

// 没有 ? 的写法——每一步都要 match
fn read_username_from_file() -> Result<String, io::Error> {
    let result = fs::read_to_string("username.txt");
    match result {
        Ok(s) => Ok(s.trim().to_string()),
        Err(e) => Err(e),
    }
}

// 用 ? 的写法——简洁得多
fn read_username_from_file() -> Result<String, io::Error> {
    let s = fs::read_to_string("username.txt")?;
    Ok(s.trim().to_string())
}

? 的展开逻辑：

  • 如果是 Ok(value)：把 value 解包，继续执行
  • 如果是 Err(e)：调用 e.into() 转换错误类型，然后 return Err(e)

注意那个 into()——? 不只是语法糖，它会自动调用 From Trait 转换错误类型。这是 ? 能在不同错误类型间传播的关键。

use std::num::ParseIntError;
use std::fs;

#[derive(Debug)]
enum AppError {
    Io(std::io::Error),
    Parse(ParseIntError),
}

impl From<std::io::Error> for AppError {
    fn from(e: std::io::Error) -> Self {
        AppError::Io(e)
    }
}

impl From<ParseIntError> for AppError {
    fn from(e: ParseIntError) -> Self {
        AppError::Parse(e)
    }
}

fn read_number_from_file(path: &str) -> Result<i32, AppError> {
    let content = fs::read_to_string(path)?; // io::Error 自动转成 AppError::Io
    let number = content.trim().parse::<i32>()?; // ParseIntError 自动转成 AppError::Parse
    Ok(number)
}

? 只能用在返回 Result（或 Option）的函数里。在 main 函数里也可以用，只需要把 main 的返回类型改成 Result：

use std::fs;

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let content = fs::read_to_string("config.txt")?;
    println!("{}", content);
    Ok(())
}

自定义错误类型

在库代码中，你需要定义自己的错误类型，让调用方能区分不同类型的错误并做出相应处理。

手动实现的方式：

use std::fmt;

#[derive(Debug)]
pub enum DatabaseError {
    ConnectionFailed(String),
    QueryFailed { query: String, reason: String },
    NotFound(u64),
}

impl fmt::Display for DatabaseError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            DatabaseError::ConnectionFailed(msg) => {
                write!(f, "数据库连接失败: {}", msg)
            }
            DatabaseError::QueryFailed { query, reason } => {
                write!(f, "查询失败 [{}]: {}", query, reason)
            }
            DatabaseError::NotFound(id) => {
                write!(f, "记录不存在: id={}", id)
            }
        }
    }
}

impl std::error::Error for DatabaseError {}

实现 Display 和 std::error::Error 是错误类型的标准要求，前者给人看，后者让错误类型接入标准库的错误处理生态。

手动写这些很繁琐，这就是 thiserror 存在的原因。

thiserror：库代码的最佳实践

thiserror 是一个派生宏库，用注解代替手写的 Display 和 From 实现：

在 Cargo.toml 中添加：

[dependencies]
thiserror = "1"

然后：

use thiserror::Error;

#[derive(Debug, Error)]
pub enum DatabaseError {
    #[error("数据库连接失败: {0}")]
    ConnectionFailed(String),

    #[error("查询失败 [{query}]: {reason}")]
    QueryFailed { query: String, reason: String },

    #[error("记录不存在: id={0}")]
    NotFound(u64),

    // #[from] 自动生成 From<io::Error> for DatabaseError
    #[error("IO错误: {0}")]
    Io(#[from] std::io::Error),
}

等价于之前的手写版本，但代码量减少了一半以上，而且不容易出错。#[error("...")] 里可以用 {0} 引用第一个字段，或者用字段名 {field_name}。

thiserror 的定位：库代码用 thiserror。因为库的调用方需要知道错误的具体类型，能够 match 不同错误做不同处理。

anyhow：应用层代码的最佳实践

thiserror 适合库，但写应用程序时情况不同。应用代码往往需要把来自各处的错误——数据库错误、网络错误、解析错误——统一处理，通常只需要把错误信息记录到日志或返回给用户，不需要精确区分每种错误类型。

这时候定义一个包含所有可能错误的大 enum 就显得累赘了。anyhow 提供了一种更轻量的方式：

[dependencies]
anyhow = "1"

use anyhow::{Context, Result};
use std::fs;

// 返回类型是 anyhow::Result，等价于 Result<T, anyhow::Error>
fn load_config(path: &str) -> Result<String> {
    let content = fs::read_to_string(path)
        .with_context(|| format!("无法读取配置文件: {}", path))?;

    if content.is_empty() {
        anyhow::bail!("配置文件不能为空");
    }

    Ok(content)
}

fn main() -> Result<()> {
    let config = load_config("app.toml")?;
    println!("配置加载成功，共 {} 字节", config.len());
    Ok(())
}

anyhow 的核心能力：

  • ? 直接用：任何实现了 std::error::Error 的类型都能用 ? 转成 anyhow::Error，不需要手写 From
  • context / with_context：给错误加上上下文信息，让错误链更清晰
  • bail!：快速返回一个错误，等价于 return Err(anyhow::anyhow!("..."))
  • anyhow!：创建一个临时错误

错误链的威力在于报错时能看到完整的上下文：

Error: 无法读取配置文件: app.toml

Caused by:
    No such file or directory (os error 2)

anyhow 的定位：应用代码用 anyhow。因为应用的调用方（用户）只需要看错误信息，不需要 match 错误类型。

thiserror vs anyhow：选哪个？

这是个经常让新手困惑的问题，一张表说清楚：

thiserroranyhow
适用场景库代码应用代码
错误类型精确的自定义类型统一的 anyhow::Error
调用方能否 match能不能（类型被抹掉）
代码量中等最少
上下文信息手动内置 context

实际项目中，两者经常配合使用：底层库用 thiserror 定义精确的错误类型，应用层用 anyhow 统一收拢所有错误。

// 底层库：精确的错误类型
// lib.rs
#[derive(Debug, thiserror::Error)]
pub enum DbError {
    #[error("连接失败: {0}")]
    Connection(String),
    #[error("记录不存在: {0}")]
    NotFound(u64),
}

// 应用层：用 anyhow 统一处理
// main.rs
use anyhow::{Context, Result};

fn get_user(id: u64) -> Result<User> {
    db::find_user(id)
        .with_context(|| format!("查询用户 {} 失败", id))?;
    // ...
}

一个完整的例子

把上面的知识串起来，写一个读取并解析配置文件的场景：

use std::collections::HashMap;
use std::fs;
use thiserror::Error;

// 定义错误类型（库风格）
#[derive(Debug, Error)]
pub enum ConfigError {
    #[error("无法读取文件 {path}: {source}")]
    ReadFailed {
        path: String,
        #[source]
        source: std::io::Error,
    },

    #[error("配置格式错误，第 {line} 行: {message}")]
    ParseFailed { line: usize, message: String },

    #[error("缺少必填项: {0}")]
    MissingField(String),
}

// 解析简单的 key=value 格式
pub fn parse_config(path: &str) -> Result<HashMap<String, String>, ConfigError> {
    let content = fs::read_to_string(path).map_err(|e| ConfigError::ReadFailed {
        path: path.to_string(),
        source: e,
    })?;

    let mut map = HashMap::new();

    for (line_num, line) in content.lines().enumerate() {
        let line = line.trim();
        if line.is_empty() || line.starts_with('#') {
            continue; // 跳过空行和注释
        }

        let parts: Vec<&str> = line.splitn(2, '=').collect();
        if parts.len() != 2 {
            return Err(ConfigError::ParseFailed {
                line: line_num + 1,
                message: format!("期望 key=value 格式，实际是: {}", line),
            });
        }

        map.insert(parts[0].trim().to_string(), parts[1].trim().to_string());
    }

    // 检查必填项
    for required in &["host", "port"] {
        if !map.contains_key(*required) {
            return Err(ConfigError::MissingField(required.to_string()));
        }
    }

    Ok(map)
}

fn main() {
    match parse_config("app.conf") {
        Ok(config) => {
            println!("配置加载成功:");
            for (k, v) in &config {
                println!("  {} = {}", k, v);
            }
        }
        Err(ConfigError::ReadFailed { path, source }) => {
            eprintln!("文件读取失败 {}: {}", path, source);
        }
        Err(ConfigError::ParseFailed { line, message }) => {
            eprintln!("第 {} 行格式错误: {}", line, message);
        }
        Err(ConfigError::MissingField(field)) => {
            eprintln!("配置缺少必填项: {}", field);
        }
    }
}

这个例子展示了完整的库风格错误处理：精确的错误类型、有意义的错误消息、调用方可以 match 不同情况分别处理。

错误处理的最佳实践

经过上面的内容，几条经验法则：

1. 失败是正常情况，就用 Result

网络请求、文件操作、用户输入——这些失败是预期中的，用 Result 建模，让调用方决定怎么处理。

2. panic 只用于不变式被破坏

如果某种情况发生意味着代码逻辑有 bug（比如索引越界、空指针、配置数据在编译时就应该保证合法），用 panic 或 unwrap。

3. 库用 thiserror，应用用 anyhow

库的调用方需要精确的错误类型，用 thiserror。应用层只需要错误信息，用 anyhow。

4. 用 context 给错误加上足够的信息

// 不够好：错误信息没有上下文
let file = fs::read_to_string(path)?;

// 更好：出错时知道在做什么
let file = fs::read_to_string(path)
    .with_context(|| format!("加载用户配置失败: {}", path))?;

5. 不要把 unwrap 留在生产代码里

unwrap 是"我现在懒得处理，先跑起来再说"的标记。上线前应该逐一替换为真正的错误处理，或者至少换成 expect("说明为什么这里不可能失败")。

小结

  • panic：不可恢复错误，用于 bug、测试、原型；库代码里禁用
  • Result<T, E>：可恢复错误的标准表示，编译器强制处理
  • ? 运算符：自动调用 From 转换错误类型并提前返回，让错误传播代码简洁
  • 自定义错误类型：实现 Display + std::error::Error，或者用 thiserror 派生
  • thiserror：库代码的选择，精确的类型，调用方可以 match
  • anyhow：应用代码的选择，统一的错误类型，context 提供丰富的调试信息
  • 经验法则：失败是正常情况用 Result；逻辑 bug 用 panic；库 thiserror，应用 anyhow
