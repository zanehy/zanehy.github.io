12 | 变量与上下文共享

发表于： 2026-03-17 分类于： OpenResty
字数： 2028 阅读：≈ 5分钟 浏览：

在 OpenResty 里写业务逻辑，经常会遇到一个问题：数据在哪里存？不同阶段之间怎么传数据？多个请求之间能共享什么？

这些问题的答案，取决于你选择哪种存储方式。OpenResty 提供了几个不同作用域的存储机制，搞清楚它们的边界，能避开很多诡异的 bug。

ngx.var：读写 Nginx 变量

ngx.var 提供对 Nginx 内置变量和自定义变量的读写访问。

读取内置变量 

-- 常用内置变量
local host        = ngx.var.host           -- 请求的 Host
local uri         = ngx.var.uri            -- 请求 URI（不含 query string）
local request_uri = ngx.var.request_uri    -- 完整请求路径（含 query string）
local method      = ngx.var.request_method -- GET / POST / ...
local remote_addr = ngx.var.remote_addr    -- 客户端 IP
local scheme      = ngx.var.scheme         -- http 或 https
local server_port = ngx.var.server_port    -- 服务端口
local http_referer = ngx.var.http_referer  -- Referer 头（http_ 前缀访问请求头）
local arg_page    = ngx.var.arg_page       -- query string 中的 page 参数

http_ 前缀可以访问任意请求头：ngx.var.http_x_forwarded_for 对应 X-Forwarded-For 头（连字符换成下划线，统一小写）。

arg_ 前缀可以访问 query string 参数：ngx.var.arg_id 对应 ?id=123。

自定义变量

在 nginx.conf 里用 set 指令声明变量，Lua 里再读写：

location /api/ {
    set $user_id "";        -- 声明变量（必须先声明）
    set $auth_status "0";

    access_by_lua_block {
        -- 写入变量
        ngx.var.user_id = "12345"
        ngx.var.auth_status = "1"
    }

    content_by_lua_block {
        -- 读取变量（在同一请求的不同阶段都能访问）
        ngx.say("user: ", ngx.var.user_id)
        ngx.say("auth: ", ngx.var.auth_status)
    }
}

ngx.var 的限制

  • 只能存字符串：写入数字会被转成字符串，写入 table 会报错
  • 必须先在 nginx.conf 声明：直接在 Lua 里写一个未声明的变量会报错
  • 性能开销相对较高：每次读写都涉及 Nginx 变量哈希查找，不适合频繁操作
  • 作用域是单个请求：不同请求的 ngx.var 相互独立

ngx.ctx：请求级上下文

ngx.ctx 是一个普通的 Lua table，生命周期与当前请求绑定。同一个请求在不同处理阶段（access、content、log 等）共享同一个 ngx.ctx。

-- access_by_lua_block 阶段
access_by_lua_block {
    -- 验证 token，把用户信息写入 ctx
    local token = ngx.req.get_headers()["authorization"]
    local user = validate_token(token)

    ngx.ctx.user_id   = user.id
    ngx.ctx.user_name = user.name
    ngx.ctx.start_time = ngx.now()
}

-- content_by_lua_block 阶段
content_by_lua_block {
    -- 直接读取，不用再次验证
    local user_id = ngx.ctx.user_id
    ngx.say("Hello, user ", user_id)
}

-- log_by_lua_block 阶段
log_by_lua_block {
    local elapsed = ngx.now() - ngx.ctx.start_time
    ngx.log(ngx.INFO, "request took ", elapsed, "s, user=", ngx.ctx.user_id)
}

ngx.ctx 可以存任意 Lua 类型，包括 table、函数、布尔值——这是相比 ngx.var 的主要优势。

ngx.ctx 在子请求中的行为

子请求有自己独立的 ngx.ctx，不继承父请求的 ngx.ctx：

-- 父请求
ngx.ctx.foo = "parent"

local res = ngx.location.capture("/sub")
-- 子请求里 ngx.ctx.foo 是 nil，不是 "parent"

如果需要向子请求传数据，用 ngx.var 或者通过请求参数传递。

ngx.shared.DICT：进程级共享内存

前两种方式都是请求级别的——每个请求有自己的副本，请求结束后数据消失。ngx.shared.DICT 不一样，它是 worker 进程级别的共享内存，同一个 worker 处理的所有请求都能读写，并且数据可以持久存在（直到 worker 重启）。

配置

首先在 nginx.conf 里声明共享内存区域：

http {
    lua_shared_dict my_cache  10m;   -- 名为 my_cache，大小 10MB
    lua_shared_dict rate_limit 5m;   -- 名为 rate_limit，大小 5MB
    lua_shared_dict locks      1m;
}

基本操作 

local cache = ngx.shared.my_cache

-- 写入（带过期时间，单位秒，0 表示永不过期）
local ok, err = cache:set("key1", "value1", 60)
if not ok then
    ngx.log(ngx.ERR, "failed to set cache: ", err)
end

-- 读取
local val = cache:get("key1")
if val then
    ngx.say("cached: ", val)
end

-- 删除
cache:delete("key1")

-- 原子递增（用于计数器）
local new_val, err = cache:incr("counter", 1, 0)  -- key, delta, 初始值

原子操作

ngx.shared.DICT 的操作都是原子的，多个 worker 同时操作同一个 key 不会出现竞态条件：

-- add：只在 key 不存在时写入（相当于 setnx）
local ok, err, forcible = cache:add("lock_key", 1, 10)
if ok then
    -- 成功拿到锁
else
    -- key 已存在，锁被其他请求持有
end

-- replace：只在 key 存在时更新
local ok, err = cache:replace("existing_key", "new_value")

-- get_stale：即使 key 过期也能读到值（用于缓存降级）
local val, flags, stale = cache:get_stale("key1")
if stale then
    -- 数据过期了，但还能用（去异步更新）
end

实战：简单的本地缓存 

local function get_user(user_id)
    local cache = ngx.shared.my_cache
    local cache_key = "user:" .. user_id

    -- 先查缓存
    local cached = cache:get(cache_key)
    if cached then
        local cjson = require "cjson"
        return cjson.decode(cached)
    end

    -- 缓存未命中，查数据库（省略具体实现）
    local user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    if not user then
        return nil, "user not found"
    end

    -- 写入缓存，过期时间 5 分钟
    local cjson = require "cjson"
    cache:set(cache_key, cjson.encode(user), 300)

    return user
end

ngx.shared.DICT 的限制

  • 只能存标量：key 和 value 都只能是字符串或数字，不能直接存 table（需要序列化）
  • 容量固定：内存在 nginx 启动时就分配好了，满了之后新写入会失败（或者按 LRU 淘汰旧数据）
  • worker 间不共享：如果 nginx 有多个 worker，每个 worker 的 ngx.shared.DICT 是独立的，互相看不到对方的写入——这是最常被误解的地方

最后一点值得展开说：ngx.shared.DICT 的名字容易让人误以为是"全局共享"，但实际上它是"同一个 worker 内的所有请求共享"。如果你有 4 个 worker，同一个 key 在 4 个 worker 里可能有 4 个不同的值。

需要真正跨 worker 共享数据，得用 Redis 或其他外部存储。

三种方式的对比

特性ngx.varngx.ctxngx.shared.DICT
作用域请求级请求级worker 级
数据类型字符串任意 Lua 类型字符串/数字
请求结束后销毁销毁保留
跨阶段共享是是是
跨请求共享否否是（同 worker）
跨 worker 共享否否否
性能中高中（原子操作有锁）
需要 nginx.conf 声明是否是

选择建议：

  • 需要跨阶段传数据，优先用 ngx.ctx
  • 需要和 nginx 配置交互，用 ngx.var
  • 需要在同一 worker 的多个请求间共享（计数、缓存），用 ngx.shared.DICT
  • 需要跨 worker 或跨机器共享，用 Redis

小结

OpenResty 的数据共享机制分三层，各有边界：

  • ngx.var：桥接 Nginx 变量，只存字符串，需要先声明
  • ngx.ctx：请求级 table，可以存任意类型，最灵活
  • ngx.shared.DICT：worker 内持久共享，只存标量，操作原子，跨请求可见
