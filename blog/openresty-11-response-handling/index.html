11 | 响应处理：控制一切输出

发表于： 2026-03-16 分类于： OpenResty
字数： 1517 阅读：≈ 4分钟 浏览：

上一篇把请求输入的读取方式讲完了。这篇来看另一半——如何控制响应输出。一个请求的生命周期里，响应这一侧同样有很多细节值得梳理。

设置响应头与状态码

响应头和状态码必须在发送 body 之前设置，一旦第一次调用 ngx.say 或 ngx.print，响应头就会自动发出，之后再修改就没有效果了。

content_by_lua_block {
    -- 设置状态码
    ngx.status = 201

    -- 设置响应头
    ngx.header["Content-Type"] = "application/json; charset=utf-8"
    ngx.header["X-Request-Id"] = "abc-123"
    ngx.header["Cache-Control"] = "no-cache, no-store"

    -- 设置多值响应头
    ngx.header["Set-Cookie"] = {
        "session_id=abc123; Path=/; HttpOnly",
        "theme=dark; Path=/; Max-Age=86400"
    }

    -- 删除某个响应头
    ngx.header["X-Powered-By"] = nil

    -- 此时才发送 body（头也随之发出）
    ngx.say('{"code":0}')
}

常用状态码常量，不用背数字：

ngx.HTTP_OK                    -- 200
ngx.HTTP_CREATED               -- 201
ngx.HTTP_NO_CONTENT            -- 204
ngx.HTTP_MOVED_PERMANENTLY     -- 301
ngx.HTTP_MOVED_TEMPORARILY     -- 302
ngx.HTTP_NOT_MODIFIED          -- 304
ngx.HTTP_BAD_REQUEST           -- 400
ngx.HTTP_UNAUTHORIZED          -- 401
ngx.HTTP_FORBIDDEN             -- 403
ngx.HTTP_NOT_FOUND             -- 404
ngx.HTTP_INTERNAL_SERVER_ERROR -- 500
ngx.HTTP_BAD_GATEWAY           -- 502
ngx.HTTP_SERVICE_UNAVAILABLE   -- 503

返回 JSON 响应

JSON 是 API 接口最常见的响应格式。标准做法：

content_by_lua_block {
    local cjson = require "cjson"

    -- 统一封装响应函数
    local function json_response(code, message, data)
        ngx.header["Content-Type"] = "application/json; charset=utf-8"
        local body = {
            code    = code,
            message = message,
        }
        if data ~= nil then
            body.data = data
        end
        ngx.say(cjson.encode(body))
    end

    -- 成功响应
    json_response(0, "ok", {id = 1, name = "alice"})

    -- 错误响应
    -- ngx.status = 400
    -- json_response(400, "invalid param")
}

封装成函数后，整个项目的响应格式保持一致，维护起来也方便。

处理 cjson 的特殊类型

cjson 对某些 Lua 类型有特殊处理，需要注意：

local cjson = require "cjson"

-- 空 table 默认编码为 {}（对象），不是 []
local t = {}
cjson.encode(t)  -- "{}"

-- 如果需要编码为空数组
cjson.encode(cjson.empty_array)  -- "[]"

-- 或者用 newproxy 标记
local arr = setmetatable({}, cjson.array_mt)
cjson.encode(arr)  -- "[]"

-- null 值
cjson.encode({name = cjson.null})  -- '{"name":null}'

重定向

外部重定向 

-- 302 临时重定向（默认）
ngx.redirect("https://example.com/new-path")

-- 301 永久重定向
ngx.redirect("https://example.com/new-path", 301)

-- 带参数的跳转
local new_url = "https://example.com/login?next=" .. ngx.escape_uri(ngx.var.request_uri)
ngx.redirect(new_url)

ngx.redirect 内部调用了 ngx.exit，执行后当前 Lua 代码会立即终止。

内部跳转

内部跳转不会改变客户端看到的 URL，相当于 Nginx 的 rewrite ... last：

-- 跳转到同一个 server 下的另一个 location
ngx.exec("/internal/handler")

-- 带参数
ngx.exec("/internal/handler", {user_id = "123", action = "view"})

-- 直接传 query string
ngx.exec("/internal/handler?user_id=123&action=view")

ngx.exec 同样会立即终止当前 Lua 代码。

ngx.location.capture：子请求

子请求是 OpenResty 里一个非常实用的特性——在处理当前请求的过程中，向另一个 location 发起内部请求，拿到结果后继续处理。整个过程对客户端透明。

基本用法 

content_by_lua_block {
    -- 向 /api/user 发起子请求
    local res = ngx.location.capture("/api/user", {
        method = ngx.HTTP_GET,
        args   = { id = "123" }
    })

    ngx.say("status: ", res.status)
    ngx.say("body: ", res.body)
    ngx.say("header: ", res.header["Content-Type"])
}

子请求的结果包含：

  • res.status：HTTP 状态码
  • res.body：响应 body（字符串）
  • res.header：响应头 table

POST 子请求 

local res = ngx.location.capture("/internal/validate", {
    method = ngx.HTTP_POST,
    body   = '{"token":"abc123"}',
    args   = { source = "api" }
})

if res.status ~= 200 then
    ngx.status = 401
    ngx.say("unauthorized")
    return ngx.exit(ngx.OK)
end

并发子请求

ngx.location.capture_multi 可以同时发起多个子请求，等所有结果返回后再继续，大幅降低总耗时：

content_by_lua_block {
    -- 并发发起三个子请求
    local res_user, res_order, res_product = ngx.location.capture_multi({
        { "/api/user",    { args = { id = "123" } } },
        { "/api/order",   { args = { user_id = "123" } } },
        { "/api/product", { args = { id = "456" } } },
    })

    -- 三个请求同时执行，总耗时约等于最慢那个
    local cjson = require "cjson"
    ngx.say(cjson.encode({
        user    = cjson.decode(res_user.body),
        order   = cjson.decode(res_order.body),
        product = cjson.decode(res_product.body),
    }))
}

对应的 nginx.conf 配置，子请求的目标 location 通常标记为 internal，防止被外部直接访问：

location /api/user {
    internal;
    content_by_lua_file lua/handlers/user.lua;
}

location /api/order {
    internal;
    proxy_pass http://order-service;
}

子请求的限制

  • 子请求和父请求共享连接资源，不适合超高并发场景下大量使用
  • 子请求的 body 会完整缓存在内存，不适合大文件响应
  • 子请求不能跨 server，只能访问同一个 server 下的 location
  • 嵌套子请求深度有限制（默认 50 层）

流式响应

对于需要持续推送数据的场景（比如 SSE、进度推送），可以分批发送：

content_by_lua_block {
    ngx.header["Content-Type"] = "text/event-stream"
    ngx.header["Cache-Control"] = "no-cache"
    ngx.header["X-Accel-Buffering"] = "no"  -- 关闭 Nginx 缓冲

    for i = 1, 10 do
        ngx.say("data: chunk " .. i .. "\n")
        ngx.flush(true)
        ngx.sleep(0.5)
    end
}

一个完整的 API 响应封装

实际项目里，通常会把响应逻辑封装成一个模块：

-- lib/response.lua
local cjson = require "cjson"
local M = {}

function M.success(data, message)
    ngx.header["Content-Type"] = "application/json; charset=utf-8"
    ngx.say(cjson.encode({
        code    = 0,
        message = message or "ok",
        data    = data,
    }))
end

function M.error(code, message, http_status)
    ngx.status = http_status or 200
    ngx.header["Content-Type"] = "application/json; charset=utf-8"
    ngx.say(cjson.encode({
        code    = code,
        message = message,
    }))
end

function M.redirect(url, status)
    ngx.redirect(url, status or 302)
end

return M

使用时：

local resp = require "lib.response"

-- 正常返回
resp.success({id = 1, name = "alice"})

-- 错误返回
resp.error(401, "token expired", 401)

-- 重定向
resp.redirect("/login")

小结

响应处理的几个核心要点：

  • 状态码和响应头必须在发送 body 之前设置
  • JSON 响应记得用 cjson.safe 版本，并统一封装格式
  • ngx.redirect 用于外部跳转，ngx.exec 用于内部跳转
  • ngx.location.capture 发起单个子请求，capture_multi 并发多个
  • 子请求的目标 location 建议加 internal 限制
