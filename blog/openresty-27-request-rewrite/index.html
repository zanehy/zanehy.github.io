27 | 请求/响应改写模块

发表于： 2026-04-02 分类于： OpenResty
字数： 1540 阅读：≈ 4分钟 浏览：

网关不只是"透明转发"，它还能在请求到达后端之前和响应返回客户端之前做各种改写。这篇来实现改写模块，并用它实现一个实用的功能：灰度发布。

请求头注入

鉴权通过后，网关知道了用户信息。把这些信息注入请求头，后端服务就不需要再做一次鉴权，直接读头就能拿到用户 ID、角色等信息。

-- plugins/request_rewrite.lua
local Plugin = {}
Plugin.name     = "request_rewrite"
Plugin.priority = 30

function Plugin:access(config)
    local user = ngx.ctx.user

    -- 注入用户信息
    if user and config.inject_user ~= false then
        ngx.req.set_header("X-User-Id",    tostring(user.id or ""))
        ngx.req.set_header("X-User-Name",  user.name or "")
        ngx.req.set_header("X-User-Roles", user.roles and table.concat(user.roles, ",") or "")
    end

    -- 注入请求追踪 ID
    local trace_id = ngx.req.get_headers()["x-trace-id"]
    if not trace_id then
        trace_id = ngx.var.request_id or ngx.md5(tostring(ngx.now()) .. ngx.var.remote_addr)
    end
    ngx.req.set_header("X-Trace-Id", trace_id)
    ngx.var.trace_id = trace_id

    -- 注入网关信息
    ngx.req.set_header("X-Forwarded-By", "api-gateway")
    ngx.req.set_header("X-Real-IP",       ngx.var.remote_addr)

    -- 删除不应透传的头
    if config.remove_headers then
        for _, header in ipairs(config.remove_headers) do
            ngx.req.clear_header(header)
        end
    end

    -- 添加自定义头（从配置读取）
    if config.add_headers then
        for k, v in pairs(config.add_headers) do
            ngx.req.set_header(k, v)
        end
    end

    -- URI 重写
    if config.rewrite_uri then
        local new_uri = ngx.var.uri:gsub(config.rewrite_uri.pattern, config.rewrite_uri.replacement)
        if new_uri ~= ngx.var.uri then
            ngx.req.set_uri(new_uri)
        end
    end
end

return Plugin

响应体改写

响应体改写在 body_filter_by_lua 阶段执行，这个阶段可以访问和修改响应数据。

location / {
    body_filter_by_lua_block {
        local chunk = ngx.arg[1]
        local is_last = ngx.arg[2]

        if not chunk or chunk == "" then return end

        -- 只处理 JSON 响应
        local ct = ngx.header["Content-Type"] or ""
        if not ct:find("application/json", 1, true) then return end

        -- 把响应体收集完整后再处理
        if not ngx.ctx.resp_body then
            ngx.ctx.resp_body = ""
        end
        ngx.ctx.resp_body = ngx.ctx.resp_body .. chunk

        if not is_last then
            -- 还没收完，先清空 chunk（不发送）
            ngx.arg[1] = nil
            return
        end

        -- 收完了，做改写
        local cjson = require "cjson.safe"
        local body, err = cjson.decode(ngx.ctx.resp_body)
        if body then
            -- 统一包装响应格式
            if body.code == nil then
                body = { code = 0, data = body }
            end
            ngx.arg[1] = cjson.encode(body)
        else
            ngx.arg[1] = ngx.ctx.resp_body
        end
    }
}

注意响应体改写的代价：需要把响应缓存在内存里，直到收到最后一个 chunk 才能处理。对于大响应体（几 MB 以上），这会消耗大量内存。实际项目里要评估是否必要，并限制可改写的响应体大小。

REST 转 gRPC 思路

现代微服务很多用 gRPC 通信，但客户端（特别是浏览器、移动端）通常只会 HTTP/REST。网关可以做协议转换，对外暴露 REST 接口，对内调用 gRPC 服务。

思路：

  1. 接收客户端的 REST 请求（JSON body）
  2. 把 JSON 转为 Protobuf 二进制（需要知道 .proto 定义）
  3. 通过 HTTP/2 发送 gRPC 请求到后端
  4. 把 Protobuf 响应转回 JSON 返回给客户端

OpenResty 本身没有原生 Protobuf 支持，通常的做法是：

-- 方案一：用 grpc-gateway（Go 写的，独立进程）做转换，OpenResty 只负责路由
-- 方案二：用 lua-protobuf 库做编解码（需要预先生成 schema）
-- 方案三：让后端服务同时暴露 REST 和 gRPC 接口（最省事）

-- 如果选择方案二，大致流程：
content_by_lua_block {
    local pb = require "pb"

    -- 加载 proto schema
    pb.loadfile("user.proto")

    -- 解析 REST 请求
    ngx.req.read_body()
    local json_body = cjson.decode(ngx.req.get_body_data())

    -- 编码为 Protobuf
    local pb_data = pb.encode("UserRequest", json_body)

    -- 调用 gRPC（需要 HTTP/2 支持）
    -- OpenResty 1.19+ 支持 grpc_pass
    ngx.exec("@grpc_backend")
}

Nginx 1.13.10+ 原生支持 grpc_pass，OpenResty 也跟进支持了：

location /api/v1/user/ {
    grpc_pass grpc://user-grpc-service:50051;
}

不过这只是透传 gRPC，协议转换还是需要额外的处理。

灰度发布

灰度发布（金丝雀发布）是把新版本先发布给一小部分用户，验证没问题后再全量发布。在网关层实现，不需要修改后端服务。

按比例分流 

-- plugins/canary.lua
local Plugin = {}
Plugin.name     = "canary"
Plugin.priority = 5   -- 最先执行，决定路由到哪个 upstream

function Plugin:access(config)
    if not config.enabled then return end

    local canary_percent = config.percent or 10  -- 默认 10% 流量到新版本
    local roll = math.random(100)

    if roll <= canary_percent then
        -- 路由到新版本
        ngx.ctx.canary = true
        ngx.ctx.route.upstream = config.canary_upstream
        ngx.log(ngx.DEBUG, "canary: routing to new version")
    end
    -- 否则走默认 upstream（在 ngx.ctx.route 里已有）
end

return Plugin

路由配置：

{
  "plugins": {
    "canary": {
      "enabled": true,
      "percent": 10,
      "canary_upstream": {
        "targets": [{"host": "user-service-v2", "port": 8080}]
      }
    }
  }
}

按用户特征分流

更精细的控制——特定用户（内测用户、员工账号）固定走新版本：

function Plugin:access(config)
    if not config.enabled then return end

    local user = ngx.ctx.user

    -- 按用户 ID 尾号（稳定分流，同一用户每次结果相同）
    if user and config.user_id_modulo then
        local user_id = tonumber(user.id) or 0
        if user_id % 100 < config.user_id_modulo then
            ngx.ctx.route.upstream = config.canary_upstream
            return
        end
    end

    -- 按 Header 标记（QA 测试、内测用户）
    local canary_header = ngx.req.get_headers()["x-canary"]
    if canary_header == "true" or canary_header == "1" then
        ngx.ctx.route.upstream = config.canary_upstream
        return
    end

    -- 按 Cookie（前端 AB 测试常用）
    local cookies = ngx.req.get_headers()["cookie"] or ""
    local group = cookies:match("ab_group=([^;]+)")
    if group == "canary" then
        ngx.ctx.route.upstream = config.canary_upstream
        return
    end

    -- 按比例随机
    if config.percent and math.random(100) <= config.percent then
        ngx.ctx.route.upstream = config.canary_upstream
    end
end

记录灰度流量

在日志里标记哪些请求走了灰度，方便统计效果：

log_by_lua_block {
    if ngx.ctx.canary then
        ngx.log(ngx.INFO, "canary request: uri=", ngx.var.uri, " status=", ngx.status)
        ngx.shared.stats:incr("canary:requests", 1, 0)
        if ngx.status >= 500 then
            ngx.shared.stats:incr("canary:errors", 1, 0)
        end
    end
}

小结

改写模块覆盖了网关的三个常见需求：

  • 请求头注入：把鉴权后的用户信息传递给后端，后端直接读头，无需二次鉴权
  • 响应体改写：统一响应格式，在 body_filter_by_lua 阶段处理，注意内存消耗
  • 灰度发布：在网关层按比例或按用户特征分流，独立于业务代码，可快速回滚
