16 | Tokio 实战

发表于： 2026-04-27 分类于： Rust
字数： 3767 阅读：≈ 8分钟 浏览：

上一篇我们把异步的底层原理讲清楚了：Future 是状态机，poll 驱动执行，.await 是暂停点，Tokio 是运行时引擎。理论有了，这篇进入实战——Tokio 最常用的 API 怎么用，什么时候用，踩过哪些坑。

spawn：把任务丢给运行时

tokio::spawn 是最常用的 API，作用是把一个 async 任务提交给 Tokio 调度器，在后台运行。类比 Go 里的 go func()，或者线程池的 submit。

use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    // spawn 立刻返回一个 JoinHandle，任务在后台运行
    let handle = tokio::spawn(async {
        sleep(Duration::from_millis(100)).await;
        println!("任务完成");
        42 // 返回值
    });

    println!("主流程继续执行，不等任务");

    // 等待任务结束，拿到返回值
    let result = handle.await.unwrap();
    println!("任务返回值: {}", result);
}

JoinHandle 的两个用途：

  1. 等待任务完成（.await）
  2. 取消任务（.abort()）
let handle = tokio::spawn(async {
    loop {
        sleep(Duration::from_secs(1)).await;
        println!("还在跑...");
    }
});

sleep(Duration::from_secs(3)).await;
handle.abort(); // 取消任务
println!("任务已取消");

所有权问题： spawn 要求 Future 是 'static + Send。意思是闭包不能借用外部的非 'static 引用，需要用 move 转移所有权，或者用 Arc 共享。

let data = String::from("hello");

// 错误：data 是借用，不满足 'static
// tokio::spawn(async { println!("{}", data); });

// 正确：move 转移所有权
tokio::spawn(async move {
    println!("{}", data);
});

select!：同时等多个 Future，谁先完成处理谁

select! 是 Tokio 里最有意思的宏，没有之一。它的语义是：同时驱动多个 Future，哪个先完成就执行对应分支，其余的丢弃。

类比：你在等快递，同时在等外卖——谁先到门，你先处理谁。

use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    let task1 = sleep(Duration::from_millis(200));
    let task2 = sleep(Duration::from_millis(100));

    tokio::select! {
        _ = task1 => println!("task1 先完成"),
        _ = task2 => println!("task2 先完成"), // 这个会先触发
    }
}

实际场景：超时控制

这是 select! 最高频的用法——给任何 async 操作加超时：

use tokio::time::{sleep, timeout, Duration};

async fn fetch_data() -> String {
    sleep(Duration::from_secs(5)).await; // 模拟慢请求
    "数据".to_string()
}

#[tokio::main]
async fn main() {
    // 方式一：tokio::time::timeout 包装（推荐）
    match timeout(Duration::from_secs(2), fetch_data()).await {
        Ok(data) => println!("拿到数据: {}", data),
        Err(_) => println!("请求超时"),
    }

    // 方式二：select! + sleep
    tokio::select! {
        data = fetch_data() => println!("拿到: {}", data),
        _ = sleep(Duration::from_secs(2)) => println!("超时了"),
    }
}

实际场景：优雅关闭

配合 tokio::signal，select! 可以优雅地处理 Ctrl+C 信号：

use tokio::signal;

#[tokio::main]
async fn main() {
    tokio::select! {
        _ = main_logic() => {
            println!("主逻辑完成");
        }
        _ = signal::ctrl_c() => {
            println!("收到 Ctrl+C，开始优雅关闭...");
            // 清理资源
        }
    }
}

async fn main_logic() {
    // 主业务逻辑
    tokio::time::sleep(tokio::time::Duration::from_secs(60)).await;
}

select! 的注意事项：

select! 默认随机选择先完成的分支（当多个同时 Ready 时），这避免了饥饿问题。但要注意：未被选中的 Future 会被丢弃，如果这个 Future 内部有状态，下次重新进入 select! 会重新开始。

// 错误用法：每次循环重新创建 sleep，导致永远不会超时
loop {
    tokio::select! {
        msg = receiver.recv() => { /* 处理消息 */ }
        _ = sleep(Duration::from_secs(10)) => { /* 超时 */ } // 每次都重新计时！
    }
}

// 正确用法：在循环外创建，或用 tokio::time::interval
let sleep_future = sleep(Duration::from_secs(10));
tokio::pin!(sleep_future);
loop {
    tokio::select! {
        msg = receiver.recv() => { /* 处理消息 */ }
        _ = &mut sleep_future => { /* 真正的超时 */ }
    }
}

异步 Channel：任务间通信

线程间通信用 std::sync::mpsc，异步任务间通信用 tokio::sync 里的 Channel 家族。

mpsc：多生产者单消费者

最常用的一种，适合"多个任务往一个队列里塞消息，一个任务处理"的场景：

use tokio::sync::mpsc;

#[tokio::main]
async fn main() {
    // 创建 channel，缓冲区大小 32
    let (tx, mut rx) = mpsc::channel::<String>(32);

    // 多个生产者
    for i in 0..3 {
        let tx = tx.clone();
        tokio::spawn(async move {
            tx.send(format!("消息 {}", i)).await.unwrap();
        });
    }

    // 关闭原始发送端，让接收端知道何时结束
    drop(tx);

    // 消费者接收所有消息
    while let Some(msg) = rx.recv().await {
        println!("收到: {}", msg);
    }
    println!("所有消息处理完毕");
}

关键点： 当所有 tx（包括 clone 出来的）都被 drop 时，rx.recv() 返回 None，循环自然结束。

oneshot：一次性通信

适合"发出请求，等待一次回复"的场景，比如 RPC：

use tokio::sync::oneshot;

async fn compute(tx: oneshot::Sender<i32>) {
    let result = 42;
    tx.send(result).unwrap(); // 只能发一次
}

#[tokio::main]
async fn main() {
    let (tx, rx) = oneshot::channel();
    tokio::spawn(compute(tx));
    let result = rx.await.unwrap();
    println!("计算结果: {}", result);
}

broadcast：广播给所有订阅者 

use tokio::sync::broadcast;

#[tokio::main]
async fn main() {
    let (tx, mut rx1) = broadcast::channel::<String>(16);
    let mut rx2 = tx.subscribe();

    tx.send("广播消息".to_string()).unwrap();

    println!("rx1 收到: {}", rx1.recv().await.unwrap());
    println!("rx2 收到: {}", rx2.recv().await.unwrap());
}

Stream：异步版迭代器

Iterator 是同步的，你调用 next() 立刻拿到下一个值。Stream 是异步的，next() 返回一个 Future，可能需要等待。

类比：Iterator 像从内存数组取数据，Stream 像从网络流接收数据包——每个包什么时候到你不知道。

use tokio_stream::{self as stream, StreamExt};

#[tokio::main]
async fn main() {
    // 从 Vec 创建 Stream
    let mut s = stream::iter(vec![1, 2, 3, 4, 5]);

    while let Some(val) = s.next().await {
        println!("收到: {}", val);
    }
}

StreamExt 提供的常用方法：

use tokio_stream::{self as stream, StreamExt};

#[tokio::main]
async fn main() {
    let s = stream::iter(vec![1, 2, 3, 4, 5]);

    // map + filter + collect
    let result: Vec<i32> = s
        .filter(|x| std::future::ready(*x % 2 == 0))
        .map(|x| x * 10)
        .collect()
        .await;

    println!("{:?}", result); // [20, 40]
}

实际场景：处理 mpsc channel 作为 Stream

use tokio::sync::mpsc;
use tokio_stream::wrappers::ReceiverStream;
use tokio_stream::StreamExt;

#[tokio::main]
async fn main() {
    let (tx, rx) = mpsc::channel::<i32>(32);

    tokio::spawn(async move {
        for i in 0..5 {
            tx.send(i).await.unwrap();
            tokio::time::sleep(tokio::time::Duration::from_millis(100)).await;
        }
    });

    // 把 receiver 包装成 Stream
    let mut stream = ReceiverStream::new(rx);
    while let Some(val) = stream.next().await {
        println!("Stream 收到: {}", val);
    }
}

spawn_blocking：处理阻塞操作

这是一个经常被忽视但非常重要的 API。

Tokio 的工作线程数量有限（默认等于 CPU 核数），如果你在 async 函数里做了一个阻塞操作（比如调用同步库、做复杂计算），这个线程就被占住了，其他 async 任务无法调度，整个运行时会卡住。

// 错误：在 async 函数里做阻塞计算
async fn bad_example() {
    let result = (0..100_000_000_u64).sum::<u64>(); // 阻塞几百毫秒
    println!("{}", result);
    // 这期间，这个工作线程上的所有其他任务都被卡住了
}

spawn_blocking 把阻塞任务扔到专用的阻塞线程池（与 async 工作线程分开），不占用 async 调度资源：

#[tokio::main]
async fn main() {
    // 正确：把阻塞操作移到阻塞线程池
    let result = tokio::task::spawn_blocking(|| {
        // 这里可以安全地做阻塞操作
        (0..100_000_000_u64).sum::<u64>()
    })
    .await
    .unwrap();

    println!("计算结果: {}", result);
}

什么时候用 spawn_blocking？

  • 调用同步 I/O（老式文件操作、某些数据库驱动）
  • CPU 密集型计算（加密、压缩、图像处理）
  • 调用不支持 async 的第三方库
  • std::sync::Mutex 的加锁（如果持锁时间长）

决策树：

是否需要等待 I/O？
  ├─ 是 → 用 tokio 的异步 I/O（tokio::fs, tokio::net）
  └─ 否（CPU 计算 or 同步阻塞）→ 用 spawn_blocking

结构化并发：让任务的生命周期有边界

“结构化并发”（Structured Concurrency）是一个设计思想：任务的生命周期应该被明确的边界约束，不应该有游离的后台任务。

Go 语言的 goroutine 是"自由的"——你可以随时 go func() 启动一个任务，然后完全忘掉它，这会导致泄漏和竞争。Tokio 的 JoinSet 提供了结构化并发的基础设施：

use tokio::task::JoinSet;

#[tokio::main]
async fn main() {
    let mut set = JoinSet::new();

    // 启动多个任务
    for i in 0..5 {
        set.spawn(async move {
            tokio::time::sleep(tokio::time::Duration::from_millis(i * 100)).await;
            format!("任务 {} 完成", i)
        });
    }

    // 按完成顺序收集结果
    while let Some(result) = set.join_next().await {
        println!("{}", result.unwrap());
    }
    println!("所有任务完成");
}

JoinSet 的优势：

  • 当 JoinSet 被 drop 时，所有未完成的任务自动取消——不会泄漏
  • 可以方便地等待"所有完成"或"任意一个完成"
  • 任务数量动态可控

结合超时的完整模式：

use tokio::task::JoinSet;
use tokio::time::{timeout, Duration};

async fn fetch(id: u32) -> String {
    tokio::time::sleep(Duration::from_millis(id as u64 * 200)).await;
    format!("数据 {}", id)
}

#[tokio::main]
async fn main() {
    let mut set = JoinSet::new();

    for i in 1..=5 {
        set.spawn(async move {
            timeout(Duration::from_millis(500), fetch(i)).await
        });
    }

    while let Some(result) = set.join_next().await {
        match result.unwrap() {
            Ok(data) => println!("成功: {}", data),
            Err(_) => println!("超时"),
        }
    }
}

综合示例：一个完整的异步任务调度器

把本篇所有知识点串起来，实现一个简单的任务调度器：支持并发执行、超时控制、优雅关闭。

use tokio::sync::mpsc;
use tokio::task::JoinSet;
use tokio::time::{timeout, Duration};
use tokio::signal;

#[derive(Debug)]
struct Task {
    id: u32,
    duration_ms: u64,
}

async fn execute_task(task: Task) -> String {
    tokio::time::sleep(Duration::from_millis(task.duration_ms)).await;
    format!("任务 {} 执行完毕（耗时 {}ms）", task.id, task.duration_ms)
}

#[tokio::main]
async fn main() {
    let (tx, mut rx) = mpsc::channel::<Task>(100);

    // 生产者：提交任务
    let producer = tokio::spawn(async move {
        let tasks = vec![
            Task { id: 1, duration_ms: 300 },
            Task { id: 2, duration_ms: 100 },
            Task { id: 3, duration_ms: 800 }, // 这个会超时
            Task { id: 4, duration_ms: 200 },
        ];
        for task in tasks {
            tx.send(task).await.unwrap();
        }
    });

    // 消费者：并发执行，每个任务超时 500ms
    let consumer = tokio::spawn(async move {
        let mut set = JoinSet::new();

        while let Some(task) = rx.recv().await {
            set.spawn(async move {
                timeout(Duration::from_millis(500), execute_task(task)).await
            });
        }

        // 等待所有任务
        while let Some(result) = set.join_next().await {
            match result.unwrap() {
                Ok(msg) => println!("[完成] {}", msg),
                Err(_)  => println!("[超时] 任务超时被取消"),
            }
        }
    });

    // select!：等业务完成，或收到 Ctrl+C
    tokio::select! {
        _ = async { producer.await.unwrap(); consumer.await.unwrap(); } => {
            println!("所有任务处理完毕，正常退出");
        }
        _ = signal::ctrl_c() => {
            println!("收到中断信号，优雅退出");
        }
    }
}

小结

  • spawn：把 async 任务提交给运行时后台执行，返回 JoinHandle，可等待、可取消；需要 'static + Send，借用要改为 move 或 Arc
  • select!：同时等多个 Future，谁先完成处理谁；最常用于超时控制和优雅关闭；注意循环内 Future 重新创建的陷阱
  • Channel：mpsc 多生产者单消费者，oneshot 一次性回复，broadcast 广播；Channel 关闭由发送端全部 drop 触发
  • Stream：异步版迭代器，StreamExt 提供 map/filter/collect 等方法；ReceiverStream 可将 mpsc receiver 包装成 Stream
  • spawn_blocking：阻塞操作必须移到专用线程池，否则占用 async 工作线程导致运行时卡顿
  • JoinSet：结构化并发的基础设施，任务生命周期有明确边界，JoinSet drop 时自动取消所有子任务
