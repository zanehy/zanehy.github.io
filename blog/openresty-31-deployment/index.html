31 | 部署与运维

发表于： 2026-04-08 分类于： OpenResty
字数： 1711 阅读：≈ 4分钟 浏览：

代码写完、测试通过、调优完毕，最后一步是部署到生产环境。这篇来讲容器化部署、优雅重启和监控接入，以及一份生产环境检查清单。

Docker 容器化部署

Dockerfile 

FROM openresty/openresty:1.25.3-alpine

# 安装 OPM（OpenResty 包管理器）
RUN opm get ledgetech/lua-resty-http \
    && opm get SkyLothar/lua-resty-jwt \
    && opm get doujiang24/lua-resty-kafka

# 复制配置和代码
COPY conf/nginx.conf         /usr/local/openresty/nginx/conf/nginx.conf
COPY lua/                    /usr/local/openresty/nginx/lua/

# 创建日志目录
RUN mkdir -p /var/log/nginx

# 健康检查
HEALTHCHECK --interval=10s --timeout=3s --start-period=5s --retries=3 \
    CMD curl -sf http://localhost:8080/health || exit 1

EXPOSE 8080

CMD ["/usr/local/openresty/nginx/sbin/nginx", "-g", "daemon off;"]

健康检查接口 

location /health {
    access_log off;
    content_by_lua_block {
        -- 检查关键依赖
        local redis = require "resty.redis"
        local red = redis:new()
        red:set_timeout(500)
        local ok = red:connect("redis", 6379)

        if not ok then
            ngx.status = 503
            ngx.say('{"status":"unhealthy","redis":"down"}')
            return
        end
        red:set_keepalive(10000, 5)
        ngx.say('{"status":"healthy"}')
    }
}

docker-compose 完整部署 

# docker-compose.yml
version: '3.8'

services:
  gateway:
    build: .
    ports:
      - "80:8080"
    environment:
      - JWT_SECRET=${JWT_SECRET}
      - REDIS_HOST=redis
    depends_on:
      redis:
        condition: service_healthy
    restart: unless-stopped
    volumes:
      - ./logs:/var/log/nginx
    deploy:
      replicas: 2
      resources:
        limits:
          cpus: '2'
          memory: 512M

  redis:
    image: redis:7-alpine
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 3
    volumes:
      - redis_data:/data

volumes:
  redis_data:

优雅重启与热更新

信号控制 

# 查看 nginx master 进程 PID
cat /usr/local/openresty/nginx/logs/nginx.pid

# 优雅重载配置（不中断现有连接）
nginx -s reload
# 或
kill -HUP $(cat logs/nginx.pid)

# 优雅停止（等待现有连接处理完）
nginx -s quit
# 或
kill -QUIT $(cat logs/nginx.pid)

# 强制停止
nginx -s stop
kill -TERM $(cat logs/nginx.pid)

# 重新打开日志文件（日志切割后用）
nginx -s reopen
kill -USR1 $(cat logs/nginx.pid)

热更新流程

场景一：修改了 nginx.conf

# 1. 检查配置语法
nginx -t

# 2. 优雅重载
nginx -s reload

# 重载过程：master 进程 fork 出新 worker，新请求由新 worker 处理
# 旧 worker 处理完现有请求后退出，整个过程无请求丢失

场景二：只修改了 Lua 代码

如果 Lua 代码通过 require 加载，修改后需要 reload 才能生效（因为 require 有缓存）。

但如果用了 content_by_lua_file，Nginx 可以配置检测文件变化：

# 开发环境：每次请求都检查文件是否变化（有性能开销，生产环境不用）
lua_code_cache off;

生产环境还是要 nginx -s reload。

场景三：只更新了路由配置（Redis）

通过管理接口触发热更新，无需 reload：

curl http://gateway/admin/reload-config
# 路由配置立即生效，零停机

滚动更新（Kubernetes） 

# k8s deployment
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0    # 更新期间始终保持全部副本可用
      maxSurge: 1          # 最多多起 1 个新副本
  template:
    spec:
      containers:
      - name: gateway
        lifecycle:
          preStop:
            exec:
              # 收到 SIGTERM 前先等 5 秒，让 LB 完成摘流
              command: ["/bin/sh", "-c", "sleep 5 && nginx -s quit"]
      terminationGracePeriodSeconds: 30

监控接入

Prometheus 指标

用 lua-resty-prometheus 暴露指标：

opm get knyar/nginx-lua-prometheus

http {
    lua_shared_dict prometheus_metrics 10m;

    init_by_lua_block {
        prometheus = require("prometheus").init("prometheus_metrics")

        -- 定义指标
        metric_requests = prometheus:counter(
            "nginx_http_requests_total",
            "HTTP requests total",
            {"host", "status", "route"}
        )
        metric_latency = prometheus:histogram(
            "nginx_http_request_duration_seconds",
            "HTTP request duration",
            {"host", "route"},
            {0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5}
        )
        metric_connections = prometheus:gauge(
            "nginx_connections_active",
            "Active connections"
        )
    }

    log_by_lua_block {
        local route_id = (ngx.ctx.route and ngx.ctx.route.id) or "unknown"
        metric_requests:inc(1, {
            ngx.var.server_name,
            tostring(ngx.status),
            route_id,
        })
        metric_latency:observe(
            tonumber(ngx.var.request_time),
            {ngx.var.server_name, route_id}
        )
    }

    server {
        # Prometheus 抓取接口（内网访问）
        location /metrics {
            allow 10.0.0.0/8;
            deny  all;
            content_by_lua_block {
                metric_connections:set(ngx.var.connections_active)
                prometheus:collect()
            }
        }
    }
}

Grafana Dashboard

导入 Nginx / OpenResty 的社区 Dashboard（ID: 10187 或搜索 “nginx lua”），或者自建关键面板：

  • 请求量（QPS）趋势图
  • 响应延迟百分位（P50/P95/P99）
  • HTTP 状态码分布（2xx/4xx/5xx）
  • 每条路由的请求量和错误率
  • 活跃连接数

告警规则示例（Prometheus AlertManager）：

groups:
- name: gateway
  rules:
  - alert: HighErrorRate
    expr: |
      sum(rate(nginx_http_requests_total{status=~"5.."}[5m]))
      / sum(rate(nginx_http_requests_total[5m])) > 0.05      
    for: 2m
    labels:
      severity: warning
    annotations:
      summary: "Gateway error rate > 5%"

  - alert: HighLatency
    expr: |
      histogram_quantile(0.99,
        rate(nginx_http_request_duration_seconds_bucket[5m])
      ) > 1      
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "P99 latency > 1s"

生产环境配置 Checklist

上线前逐一检查：

安全

  • JWT secret 通过环境变量注入，不写在代码里
  • 管理接口（/admin/*、/metrics）限制内网访问
  • 关闭 server_tokens（不暴露 Nginx 版本）：server_tokens off;
  • 设置合理的 client_max_body_size，防止超大请求
  • HTTPS 配置（证书、HSTS、TLS 版本限制）

性能

  • worker_processes auto + worker_cpu_affinity auto
  • worker_connections 和系统 nofile 对齐
  • 所有外部调用都有超时设置
  • 连接池大小合理，不超过下游服务的连接数上限
  • lua_code_cache on（生产环境必须开启）

可观测性

  • 结构化 JSON 日志，包含 trace_id
  • Prometheus 指标已接入
  • 告警规则已配置（错误率、延迟、连接数）
  • 日志轮转已配置（logrotate）

稳定性

  • 所有插件有熔断保护
  • 限流配置合理
  • 健康检查接口可用
  • 优雅停机已测试（nginx -s quit 无请求丢失）
  • 配置文件语法检查：nginx -t

运维

  • 部署脚本可重复执行
  • 回滚方案已准备
  • 关键配置（Redis 地址、JWT secret）走配置中心或环境变量

小结

生产部署的三件套：

  • 容器化：Dockerfile + docker-compose，环境一致，部署可重复
  • 优雅重启：配置变更用 nginx -s reload，Lua 路由配置通过 Redis 热更新
  • 监控：Prometheus 采集指标，Grafana 可视化，AlertManager 告警
