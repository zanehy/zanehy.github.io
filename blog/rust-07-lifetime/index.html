07 | 生命周期——引用的有效期限

发表于： 2026-04-21 分类于： Rust
字数： 3441 阅读：≈ 7分钟 浏览：

上一篇我们学了借用与引用，知道 &T 可以让你"看"数据而不转移所有权。但有个问题留着没讲：

引用到底能用多久？

如果数据已经被释放，但引用还指向那块地址，会发生什么？Rust 是怎么在编译期发现这个问题的？

这就是**生命周期（Lifetime）**要解决的事。

生命周期是 Rust 里最让初学者头疼的概念，主要原因是语法看起来很陌生：'a、'b、'static……像某种暗语。但它的核心思想其实很朴素，类比一下就清楚了。

租约类比：引用就是一份有期限的借据

你向朋友借一本书，写了一张借据：“我借了这本书，2025年底还。”

这张借据有一个有效期。如果你的朋友在2025年底之前把书卖掉了，你的借据就失效了——书不存在了，借据还指向那本书，是没有意义的。

Rust 的引用和生命周期，就是这套逻辑：

  • 数据（书）有它自己的生命周期——它在内存里存活的时间段
  • 引用（借据）也有一个有效期
  • 规则：引用的有效期不能超过它所指向的数据的生命周期

编译器的借用检查器，就是在帮你检查"借据是否会比书活得更久"。

悬空引用：借据还在，书已消失

最直接的例子，C/C++ 里的经典 bug：

// C 代码（危险！）
int* get_number() {
    int x = 42;       // x 在栈上
    return &x;        // 返回 x 的地址
}                     // 函数返回，x 被销毁——地址指向垃圾数据

int main() {
    int* p = get_number();
    printf("%d\n", *p);  // 未定义行为，可能崩溃
}

用借据类比：书已经烧掉了，你还拿着借据去找书，找到的是灰烬。

Rust 在编译期就拒绝了这种写法：

fn get_number() -> &i32 {  // 编译错误！
    let x = 42;
    &x  // 想返回 x 的引用
}   // x 在这里被 drop，引用将指向无效内存

编译器报错：returns a reference to data owned by the current function

修复很简单——把值本身返回出去，而不是引用：

fn get_number() -> i32 {  // 返回值，转移所有权
    let x = 42;
    x  // 所有权转移给调用者，x 的数据不会消失
}

生命周期标注：不是指定，而是描述

好，简单的情况编译器能自动推断。但有时候引用关系复杂，编译器推断不了，就需要你手动标注。

关键点：生命周期标注不会改变任何东西，它只是在告诉编译器几个引用之间的关系。

来看一个场景：写一个函数，接收两个字符串引用，返回其中较长的那个。

// 这个函数不能通过编译
fn longest(x: &str, y: &str) -> &str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}

编译器报错：missing lifetime specifier

为什么？因为编译器不知道返回的 &str 和哪个参数有关系。返回的引用是来自 x 还是 y？它们的生命周期可能不同，编译器无法保证返回值是安全的。

加上生命周期标注：

fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}

'a 是一个生命周期参数，你可以把它读作"某个生命周期"。这段标注的意思是：

“返回值的生命周期，和 x、y 中较短的那个相同。”

不是说你"指定"了生命周期的长度，而是在"描述"输入和输出之间的关系。

调用时就能看出为什么需要这个约束：

fn main() {
    let s1 = String::from("long string");
    let result;
    {
        let s2 = String::from("xy");
        result = longest(s1.as_str(), s2.as_str());
        println!("{}", result);  // 在 s2 的作用域内使用，没问题
    }
    // println!("{}", result);  // 编译错误！s2 已经被 drop，result 可能指向 s2
}

编译器通过生命周期标注，推断出 result 的有效期不能超过 s2，阻止了悬空引用。

生命周期标注的语法

生命周期参数以单引号开头，名字通常用小写字母，最常见的是 'a、'b。

&i32         // 普通引用（编译器能推断时不需要标注）
&'a i32      // 有生命周期标注的引用
&'a mut i32  // 有生命周期标注的可变引用

生命周期参数写在函数名后面的尖括号里：

fn foo<'a, 'b>(x: &'a str, y: &'b str) -> &'a str {
    // 返回值的生命周期和 x 相同，与 y 无关
    x
}

这里的 'a 和 'b 是两个不同的生命周期，说明 x 和 y 可以来自不同生命周期的数据。返回值标注 'a 表示它和 x 的生命周期绑定。

结构体中的生命周期

如果结构体里存放了引用，也需要生命周期标注——因为编译器需要知道：结构体能活多久，引用就最多活多久。

// 存放了引用的结构体，必须标注生命周期
struct Excerpt<'a> {
    text: &'a str,
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");

    // 取第一句话的引用
    let first_sentence = novel.split('.').next().unwrap();

    let excerpt = Excerpt {
        text: first_sentence,  // text 引用了 novel 中的数据
    };

    println!("{}", excerpt.text);
    // excerpt 不能活得比 novel 更久——novel 消失了，excerpt.text 就失效了
}

用租约类比：Excerpt 相当于一份"摘录卡片"，卡片上贴着原书的引用。原书还在，卡片才有意义；原书卖掉了，卡片上的内容就无效了。

结构体的方法里也可以用生命周期：

impl<'a> Excerpt<'a> {
    fn level(&self) -> i32 {
        3  // 返回值不是引用，不需要生命周期标注
    }

    fn announce(&self, announcement: &str) -> &str {
        println!("Attention: {}", announcement);
        self.text  // 返回 self 中的引用，生命周期和 self 相同
    }
}

生命周期省略规则：大多数时候不用写

你可能注意到，前几篇的代码里有大量的引用，但没有一处写了 'a。这是因为 Rust 有一套生命周期省略规则（Lifetime Elision Rules），在模式明确的情况下，编译器会自动补全。

规则一共有三条，理解了之后你就知道什么时候需要写、什么时候不需要：

规则一：每个引用参数都有自己独立的生命周期。

// 你写的
fn foo(x: &str, y: &str) -> i32 { ... }

// 编译器理解为
fn foo<'a, 'b>(x: &'a str, y: &'b str) -> i32 { ... }

规则二：如果只有一个引用参数，返回值的生命周期和它相同。

// 你写的
fn first_word(s: &str) -> &str { ... }

// 编译器理解为
fn first_word<'a>(s: &'a str) -> &'a str { ... }

这就是为什么 first_word 这种函数不需要写生命周期标注——只有一个输入引用，返回值显然和它相关。

规则三：如果有多个引用参数，但其中一个是 &self 或 &mut self，返回值的生命周期和 self 相同。

// 你写的
impl Excerpt {
    fn announce(&self, text: &str) -> &str { ... }
}

// 编译器理解为
impl<'a> Excerpt<'a> {
    fn announce<'b>(&'a self, text: &'b str) -> &'a str { ... }
}

这三条规则覆盖了绝大多数日常代码。只有在编译器应用完三条规则后仍然无法确定返回值生命周期时，才需要你手动标注。

实际写代码时的感受：大部分函数不需要写生命周期标注。当编译器报错说"missing lifetime specifier"时，再想清楚输入输出的关系，加上标注即可。

‘static：活得和程序一样久

'static 是一个特殊的生命周期，表示"整个程序运行期间都有效"。

最常见的 'static 是字符串字面量：

let s: &'static str = "I am a literal";

字符串字面量被编译进程序的只读数据段，程序不结束它就不消失，所以是 'static。

常见误解：'static 不代表"放在堆上"或"不会被释放"，只代表"能活到程序结束"。

有时候你会在错误信息里看到这样的提示：

error: ... has lifetime 'static, but it needs to satisfy a shorter lifetime requirement

或者反过来，编译器要求你的类型满足 'static：

fn spawn_thread<F>(f: F)
where
    F: Fn() + Send + 'static,  // 闭包必须是 'static——线程可能活得比当前作用域更久
{
    std::thread::spawn(f);
}

为什么线程的闭包需要 'static？因为线程可能在当前函数返回之后才执行，如果闭包引用了局部变量，那个变量可能已经消失了——借据比书活得更久，又是悬空引用。'static 约束确保闭包里捕获的数据能活到程序结束，不会悬空。

新手常见误区

误区一：以为生命周期标注能"延长"数据的生命周期

fn get_str<'a>() -> &'a str {
    let s = String::from("hello");
    &s  // 编译错误！生命周期标注改变不了 s 在函数结束时被 drop 这个事实
}

生命周期标注只是描述关系，不是魔法。s 在函数结束时依然会被释放，标注 'a 并不能阻止这件事。

误区二：结构体存引用时忘记标注

// 编译错误！
struct Config {
    host: &str,  // 编译器：这个引用能活多久？我不知道。
    port: u16,
}

// 正确写法
struct Config<'a> {
    host: &'a str,
    port: u16,
}

结构体里只要有引用，就必须告诉编译器这个引用的生命周期约束。

误区三：搞不清楚什么时候返回引用、什么时候返回值

一个简单的判断原则：

  • 返回的数据来自参数——可以返回引用（生命周期和参数相关）
  • 返回的数据在函数内部创建——必须返回值（转移所有权），不能返回引用
// 来自参数，可以返回引用
fn first<'a>(items: &'a [String]) -> &'a String {
    &items[0]
}

// 在函数内创建，必须返回值
fn make_greeting(name: &str) -> String {
    format!("Hello, {}!", name)  // 返回 String，不是 &String
}

小结

生命周期是 Rust 在编译期追踪引用安全的机制，核心逻辑只有一条：

引用的有效期不能超过它所指向数据的生命周期。

这一篇的关键点：

  • 悬空引用（引用指向已释放的数据）是 Rust 在编译期绝对拒绝的
  • 生命周期标注（'a）不会改变任何运行时行为，只是描述引用之间的关系
  • 函数里有多个引用参数且返回引用时，需要标注说明返回值和哪个参数的生命周期绑定
  • 结构体存放引用时，必须标注生命周期
  • 三条省略规则覆盖了大部分场景，大多数时候不需要手写标注
  • 'static 表示"活到程序结束"，常见于字符串字面量和跨线程的闭包

生命周期、所有权、借用这三个概念合在一起，构成了 Rust 内存安全的完整拼图。掌握了这三篇，你已经过了 Rust 最难的门槛。
