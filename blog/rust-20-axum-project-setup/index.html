20 | 项目搭建与路由设计：用 axum 构建任务管理 API

发表于： 2026-04-29 分类于： Rust
字数： 3202 阅读：≈ 7分钟 浏览：

前面 19 篇，我们把 Rust 的核心概念逐一拆解：所有权、生命周期、Trait、异步、宏、unsafe……现在到了检验的时候——把这些东西放进一个真实项目，看它们如何协同工作。

接下来三篇，我们会从零构建一个任务管理 REST API：

  • 第 20 篇（本篇）：项目搭建、路由设计、请求/响应处理
  • 第 21 篇：统一错误处理、Tower 中间件、优雅关闭
  • 第 22 篇：测试、性能调优、Docker 部署

项目不复杂，但足够覆盖生产级 Rust Web 服务的核心模式。技术栈：axum + tokio + serde + thiserror。

为什么选 axum

Rust 的 Web 框架不少：actix-web、axum、warp、poem……为什么选 axum？

axum 是 tokio 团队出品，和 tokio 生态天然契合。它的设计哲学很 Rust：把 HTTP 处理拆成组合式的小块，而不是用宏魔法生成代码。路由、提取器、响应——每一层都是普通的 Rust 类型，编译器全程参与检查。

另一个优势是基于 Tower——一个成熟的中间件生态。日志、超时、限流、CORS，都有现成的 Tower 中间件可以直接插入。

项目结构

先来规划项目结构。一个清晰的结构在项目生长时会省很多麻烦：

task-api/
├── Cargo.toml
└── src/
    ├── main.rs          # 程序入口，启动服务
    ├── routes.rs        # 路由注册
    ├── handlers.rs      # 请求处理函数
    ├── models.rs        # 数据模型（Task 结构体等）
    ├── store.rs         # 内存存储层
    └── errors.rs        # 统一错误类型（第 21 篇展开）

这个结构遵循"按职责分层"的原则：数据、业务逻辑、HTTP 层各自独立，便于测试和替换。

初始化项目 

cargo new task-api
cd task-api

Cargo.toml 添加依赖：

[package]
name = "task-api"
version = "0.1.0"
edition = "2021"

[dependencies]
axum = "0.7"
tokio = { version = "1", features = ["full"] }
serde = { version = "1", features = ["derive"] }
serde_json = "1"
uuid = { version = "1", features = ["v4"] }
chrono = { version = "0.4", features = ["serde"] }
thiserror = "1"
tower = "0.4"
tower-http = { version = "0.5", features = ["cors", "trace"] }
tracing = "0.1"
tracing-subscriber = { version = "0.3", features = ["env-filter"] }

各个依赖的职责：

  • axum：Web 框架本体
  • tokio：异步运行时，features = ["full"] 开启所有特性
  • serde + serde_json：JSON 序列化/反序列化
  • uuid：生成任务 ID
  • chrono：时间戳处理
  • thiserror：定义错误类型（第 21 篇用）
  • tower-http：HTTP 中间件（CORS、日志等）
  • tracing + tracing-subscriber：结构化日志

数据模型

先定义核心数据结构，src/models.rs：

use chrono::{DateTime, Utc};
use serde::{Deserialize, Serialize};
use uuid::Uuid;

/// 任务状态
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
#[serde(rename_all = "lowercase")]
pub enum TaskStatus {
    Todo,
    InProgress,
    Done,
}

/// 任务实体
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Task {
    pub id: Uuid,
    pub title: String,
    pub description: Option<String>,
    pub status: TaskStatus,
    pub created_at: DateTime<Utc>,
    pub updated_at: DateTime<Utc>,
}

impl Task {
    pub fn new(title: String, description: Option<String>) -> Self {
        let now = Utc::now();
        Task {
            id: Uuid::new_v4(),
            title,
            description,
            status: TaskStatus::Todo,
            created_at: now,
            updated_at: now,
        }
    }
}

/// 创建任务的请求体
#[derive(Debug, Deserialize)]
pub struct CreateTaskRequest {
    pub title: String,
    pub description: Option<String>,
}

/// 更新任务的请求体（所有字段可选）
#[derive(Debug, Deserialize)]
pub struct UpdateTaskRequest {
    pub title: Option<String>,
    pub description: Option<String>,
    pub status: Option<TaskStatus>,
}

几个值得注意的地方：

#[serde(rename_all = "lowercase")] 让枚举序列化为小写字符串，JSON 里 "todo" 而不是 "Todo"，更符合 REST API 惯例。

UpdateTaskRequest 所有字段都是 Option，这是 REST PATCH 语义的标准做法：只传要修改的字段，不传的保持原值。

DateTime<Utc> 配合 chrono 的 serde feature，序列化为 ISO 8601 格式（"2026-04-28T10:00:00Z"），前端友好。

内存存储层

生产项目会用数据库，但这三篇的重点是 Rust Web 框架的用法，所以用内存存储保持简单。src/store.rs：

use std::collections::HashMap;
use std::sync::{Arc, RwLock};
use uuid::Uuid;

use crate::models::{Task, UpdateTaskRequest};

/// 共享状态：用 Arc<RwLock<T>> 在多个 Handler 间安全共享
pub type SharedStore = Arc<RwLock<TaskStore>>;

pub struct TaskStore {
    tasks: HashMap<Uuid, Task>,
}

impl TaskStore {
    pub fn new() -> Self {
        TaskStore {
            tasks: HashMap::new(),
        }
    }

    pub fn list(&self) -> Vec<Task> {
        let mut tasks: Vec<Task> = self.tasks.values().cloned().collect();
        // 按创建时间排序，返回结果稳定
        tasks.sort_by(|a, b| a.created_at.cmp(&b.created_at));
        tasks
    }

    pub fn get(&self, id: &Uuid) -> Option<Task> {
        self.tasks.get(id).cloned()
    }

    pub fn create(&mut self, task: Task) -> Task {
        let task_clone = task.clone();
        self.tasks.insert(task.id, task);
        task_clone
    }

    pub fn update(&mut self, id: &Uuid, req: UpdateTaskRequest) -> Option<Task> {
        let task = self.tasks.get_mut(id)?;
        if let Some(title) = req.title {
            task.title = title;
        }
        if let Some(desc) = req.description {
            task.description = Some(desc);
        }
        if let Some(status) = req.status {
            task.status = status;
        }
        task.updated_at = chrono::Utc::now();
        Some(task.clone())
    }

    pub fn delete(&mut self, id: &Uuid) -> bool {
        self.tasks.remove(id).is_some()
    }
}

这里的 SharedStore 是 Arc<RwLock<TaskStore>> 的类型别名。这是 axum 共享状态的标准模式：

  • Arc：让多个 Handler 持有同一份数据的所有权
  • RwLock：允许多个并发读，写时独占。任务列表读多写少，比 Mutex 更合适

update 方法用了 ? 运算符——如果 id 不存在，tasks.get_mut(id) 返回 None，? 直接返回 None，不需要额外的 if let。

Handler 函数

Handler 是 axum 中处理 HTTP 请求的函数。src/handlers.rs：

use axum::{
    extract::{Path, State},
    http::StatusCode,
    Json,
};
use uuid::Uuid;

use crate::models::{CreateTaskRequest, Task, UpdateTaskRequest};
use crate::store::SharedStore;

/// GET /tasks — 获取所有任务
pub async fn list_tasks(
    State(store): State<SharedStore>,
) -> Json<Vec<Task>> {
    let store = store.read().unwrap();
    Json(store.list())
}

/// GET /tasks/:id — 获取单个任务
pub async fn get_task(
    State(store): State<SharedStore>,
    Path(id): Path<Uuid>,
) -> Result<Json<Task>, StatusCode> {
    let store = store.read().unwrap();
    store
        .get(&id)
        .map(Json)
        .ok_or(StatusCode::NOT_FOUND)
}

/// POST /tasks — 创建任务
pub async fn create_task(
    State(store): State<SharedStore>,
    Json(req): Json<CreateTaskRequest>,
) -> (StatusCode, Json<Task>) {
    let task = Task::new(req.title, req.description);
    let mut store = store.write().unwrap();
    let created = store.create(task);
    (StatusCode::CREATED, Json(created))
}

/// PATCH /tasks/:id — 更新任务
pub async fn update_task(
    State(store): State<SharedStore>,
    Path(id): Path<Uuid>,
    Json(req): Json<UpdateTaskRequest>,
) -> Result<Json<Task>, StatusCode> {
    let mut store = store.write().unwrap();
    store
        .update(&id, req)
        .map(Json)
        .ok_or(StatusCode::NOT_FOUND)
}

/// DELETE /tasks/:id — 删除任务
pub async fn delete_task(
    State(store): State<SharedStore>,
    Path(id): Path<Uuid>,
) -> StatusCode {
    let mut store = store.write().unwrap();
    if store.delete(&id) {
        StatusCode::NO_CONTENT
    } else {
        StatusCode::NOT_FOUND
    }
}

这里有几个 axum 的核心概念值得展开：

提取器（Extractor）

axum 通过提取器从 HTTP 请求中提取数据，用函数参数的类型来声明需要什么：

  • State(store): State<SharedStore>：从应用状态中提取共享存储
  • Path(id): Path<Uuid>：从 URL 路径中提取并解析参数
  • Json(req): Json<CreateTaskRequest>：从请求体中解析 JSON

提取器的魔法在于：它们是组合式的，可以在一个 Handler 里声明多个，axum 自动按顺序执行。如果某个提取器失败（比如 JSON 解析出错、路径参数格式不对），axum 自动返回对应的错误响应，不需要你手写校验逻辑。

返回类型的灵活性

axum 的 Handler 返回类型很灵活，任何实现了 IntoResponse 的类型都可以：

  • Json<T>：序列化为 JSON 响应
  • StatusCode：只返回状态码
  • (StatusCode, Json<T>)：元组，状态码 + 响应体
  • Result<T, E>：Ok 返回成功响应，Err 返回错误响应

get_task 里的 .map(Json).ok_or(StatusCode::NOT_FOUND) 是一个惯用写法：把 Option<Task> 转成 Result<Json<Task>, StatusCode>，找不到就 404。

关于代码里的 .unwrap()：这里获取锁后直接 unwrap，是因为 RwLock 的"中毒"（poison）场景——只有在持锁期间发生 panic 才会触发，而我们的 Handler 逻辑不会 panic，所以这里 unwrap 是安全的。生产代码里若想更严谨，可以用 .unwrap_or_else(|e| e.into_inner()) 恢复中毒的锁，或者统一用第 21 篇的错误处理层来处理。

路由注册

Handler 写好了，还需要把它们挂到路由上。src/routes.rs：

use axum::{
    routing::{delete, get, patch, post},
    Router,
};

use crate::handlers::{create_task, delete_task, get_task, list_tasks, update_task};
use crate::store::SharedStore;

pub fn create_router(store: SharedStore) -> Router {
    Router::new()
        .route("/tasks", get(list_tasks).post(create_task))
        .route(
            "/tasks/:id",
            get(get_task).patch(update_task).delete(delete_task),
        )
        .with_state(store)
}

axum 的路由设计很直观：route(path, method_handler) 把路径和 HTTP 方法绑定在一起。同一个路径可以链式挂多个方法——get(list_tasks).post(create_task) 在 /tasks 上同时注册 GET 和 POST。

.with_state(store) 把共享存储注入到路由中，之后所有 Handler 都可以通过 State 提取器拿到它。注意：with_state 要在路由树的根调用，或者在需要共享状态的子路由上调用，否则 Handler 拿不到。

启动服务

最后是程序入口，src/main.rs：

mod errors;
mod handlers;
mod models;
mod routes;
mod store;

use std::sync::{Arc, RwLock};
use store::TaskStore;
use tracing::info;
use tracing_subscriber::EnvFilter;

#[tokio::main]
async fn main() {
    // 初始化日志，支持 RUST_LOG 环境变量控制级别
    tracing_subscriber::fmt()
        .with_env_filter(EnvFilter::from_default_env().add_directive("task_api=debug".parse().unwrap()))
        .init();

    // 初始化共享存储
    let store = Arc::new(RwLock::new(TaskStore::new()));

    // 构建路由
    let app = routes::create_router(store);

    // 绑定端口并启动
    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000")
        .await
        .expect("Failed to bind port 3000");

    info!("Server listening on http://0.0.0.0:3000");
    axum::serve(listener, app)
        .await
        .expect("Server error");
}

#[tokio::main] 是第 15 篇讲过的宏——它把 async fn main 包裹在 tokio 运行时里，让整个程序跑在异步上下文中。

TcpListener::bind + axum::serve 是 axum 0.7 的标准启动方式（旧版用 Server::bind，0.7 改成了这个更底层、更灵活的写法）。

启动后，用 curl 快速验证一下：

# 创建任务
curl -X POST http://localhost:3000/tasks \
  -H "Content-Type: application/json" \
  -d '{"title": "写第 21 篇", "description": "错误处理和中间件"}'

# 列出所有任务
curl http://localhost:3000/tasks

# 更新状态
curl -X PATCH http://localhost:3000/tasks/<id> \
  -H "Content-Type: application/json" \
  -d '{"status": "in_progress"}'

# 删除任务
curl -X DELETE http://localhost:3000/tasks/<id>

小结

这篇搭起了整个项目的骨架：

  • 数据模型：Task 结构体和请求/响应类型，用 serde 处理序列化
  • 存储层：Arc<RwLock<TaskStore>> 实现线程安全的内存存储
  • Handler：用提取器声明式地解析请求，返回类型灵活组合
  • 路由：链式注册，.with_state 注入共享状态
  • 启动：tokio 运行时 + axum::serve，几行代码拉起一个 HTTP 服务

目前的实现有一个明显不足：错误处理太粗糙——找不到资源直接返回裸 StatusCode，没有结构化的错误信息，也没有统一的响应格式。下一篇，我们来解决这个问题：自定义错误类型、统一响应结构、Tower 中间件，以及优雅关闭。
