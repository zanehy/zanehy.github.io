02 | 工具链与项目结构：Rust 的工程基础

发表于： 2026-04-17 分类于： Rust
字数： 2582 阅读：≈ 6分钟 浏览：

在正式写 Rust 代码之前，先把工具链搞清楚。工具用顺手了，后面的学习会省很多力气。

Rust 的工具链设计得相当完整，安装好之后你会得到一整套开箱即用的工具：编译器、包管理器、格式化工具、代码检查工具——它们都来自官方，不需要到处找第三方替代品。

rustup：管理 Rust 本身

rustup 是 Rust 的工具链管理器，类似 Python 的 pyenv 或 Node 的 nvm。它负责安装、更新和切换不同版本的 Rust。

安装 Rust：

curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

安装完成后，rustup 会在你的 PATH 里加入 ~/.cargo/bin，所有工具都放在这里。

常用命令：

# 更新到最新稳定版
rustup update

# 查看当前版本
rustc --version
cargo --version

# 安装 nightly 版本（某些实验性特性需要）
rustup install nightly

# 为当前项目切换到 nightly
rustup override set nightly

# 查看已安装的工具链
rustup toolchain list

三个发布频道：

  • stable：稳定版，每六周发布一次，生产环境用这个
  • beta：下一个 stable 的预览版
  • nightly：每日构建，包含最新的实验性特性，某些库（比如早期的 async trait）依赖它

绝大多数情况下用 stable 就够了。

cargo：Rust 的核心工具

cargo 是 Rust 的构建系统和包管理器，它把编译、测试、依赖管理、发布全包了。和 Go 的 go 命令类似，但功能更丰富。

创建项目：

# 创建二进制项目（可执行程序）
cargo new my-app

# 创建库项目
cargo new my-lib --lib

# 在已有目录里初始化
cargo init

新建项目后的目录结构：

my-app/
├── Cargo.toml      # 项目配置文件（类似 package.json / go.mod）
├── Cargo.lock      # 依赖锁定文件（自动生成）
└── src/
    └── main.rs     # 程序入口

常用命令：

# 编译（debug 模式，快但慢运行）
cargo build

# 编译 + 运行
cargo run

# 发布编译（优化，慢但快运行）
cargo build --release

# 运行测试
cargo test

# 检查代码是否能编译（不生成产物，比 build 快很多）
cargo check

# 添加依赖
cargo add serde
cargo add tokio --features full

# 查看依赖树
cargo tree

# 清理构建产物
cargo clean

cargo check 是日常开发中最常用的命令之一——它只做类型检查和借用检查，不生成二进制，速度比 cargo build 快几倍，适合边写边检查。

Cargo.toml 详解

Cargo.toml 是项目的核心配置文件，用 TOML 格式编写。

[package]
name = "my-app"          # 包名
version = "0.1.0"        # 版本号（语义化版本）
edition = "2021"         # Rust edition，推荐用 2021
authors = ["Your Name <you@example.com>"]
description = "A brief description"

[dependencies]
# 来自 crates.io 的依赖
serde = { version = "1", features = ["derive"] }
tokio = { version = "1", features = ["full"] }
anyhow = "1"

# 来自 Git 的依赖
# some-lib = { git = "https://github.com/user/repo", branch = "main" }

# 来自本地路径的依赖（monorepo 常用）
# my-utils = { path = "../my-utils" }

[dev-dependencies]
# 只在测试和 benchmark 中用的依赖
mockito = "1"

[build-dependencies]
# 构建脚本用的依赖
# cc = "1"

[profile.release]
# 发布版本的优化配置
opt-level = 3
lto = true              # Link Time Optimization，减小二进制体积

关于 edition：

Rust 有 2015、2018、2021 三个 edition，每个 edition 会引入一些语法和语义上的改进。不同 edition 的包可以互相依赖，没有兼容性问题。新项目统一用 2021。

版本号规则：

依赖的版本号遵循语义化版本（SemVer），cargo 有一套版本解析规则：

serde = "1"          # 等价于 >=1.0.0, <2.0.0
serde = "1.0"        # 等价于 >=1.0.0, <2.0.0
serde = "=1.0.130"   # 精确版本
serde = ">=1.0, <2"  # 手动范围

Workspace：多包项目

当项目变大，或者需要把代码拆成多个独立的 crate 时，用 workspace 来组织。

my-project/
├── Cargo.toml          # workspace 根配置
├── Cargo.lock          # 整个 workspace 共享一个 lock 文件
├── api/                # HTTP 服务
│   ├── Cargo.toml
│   └── src/
├── core/               # 核心业务逻辑
│   ├── Cargo.toml
│   └── src/
└── utils/              # 工具库
    ├── Cargo.toml
    └── src/

根目录的 Cargo.toml：

[workspace]
members = [
    "api",
    "core",
    "utils",
]
# workspace 级别的依赖，各成员可以引用，避免版本不一致
[workspace.dependencies]
serde = { version = "1", features = ["derive"] }
tokio = { version = "1", features = ["full"] }

成员 crate 引用 workspace 依赖：

# api/Cargo.toml
[dependencies]
serde = { workspace = true }   # 使用 workspace 里定义的版本
core = { path = "../core" }    # 引用同 workspace 的其他成员

Workspace 的好处是所有成员共享一个 Cargo.lock，保证依赖版本一致，构建缓存也是共享的。

模块系统：mod / use / pub

Rust 的模块系统初学者容易搞混，理清楚几个概念就好了。

模块（mod）：

模块是代码的组织单位，用 mod 关键字声明。

// src/main.rs

mod math {
    // 模块内的东西默认是私有的
    fn add(a: i32, b: i32) -> i32 {
        a + b
    }

    // 用 pub 暴露给外部
    pub fn multiply(a: i32, b: i32) -> i32 {
        a * b
    }
}

fn main() {
    // 用路径访问模块内的公开函数
    let result = math::multiply(3, 4);
    println!("{}", result); // 12

    // 这行会报错：add 是私有的
    // math::add(1, 2);
}

文件即模块：

更常见的做法是把每个模块放到单独的文件里。

src/
├── main.rs
├── math.rs       # 对应 mod math
└── utils/
    ├── mod.rs    # 对应 mod utils（旧风格）
    └── string.rs # 对应 mod utils::string

或者用新风格（推荐）：

src/
├── main.rs
├── math.rs
└── utils.rs      # 对应 mod utils

在 main.rs 里声明模块：

// src/main.rs
mod math;    // 对应 src/math.rs
mod utils;   // 对应 src/utils.rs 或 src/utils/mod.rs

fn main() {
    math::some_function();
}

use：引入路径

每次都写全路径太麻烦，用 use 引入：

use std::collections::HashMap;
use std::io::{self, Read, Write}; // 一次引入多个

fn main() {
    let mut map = HashMap::new(); // 不用写 std::collections::HashMap
    map.insert("key", "value");
}

pub 的可见性：

pub fn public_fn() {}           // 任何地方都能访问
pub(crate) fn crate_fn() {}     // 只有当前 crate 内能访问
pub(super) fn parent_fn() {}    // 只有父模块能访问
fn private_fn() {}              // 只有当前模块内能访问

结构体的字段也遵循同样的规则：

pub struct User {
    pub name: String,    // 公开
    age: u32,            // 私有，外部无法直接访问
}

rustfmt：统一代码风格

rustfmt 是官方的代码格式化工具，安装 Rust 时自带。

# 格式化当前项目
cargo fmt

# 只检查，不修改（CI 里用）
cargo fmt -- --check

在项目根目录创建 rustfmt.toml 可以自定义格式规则，但绝大多数团队直接用默认配置，不折腾。Rust 社区对代码风格的统一程度远高于 Go 和 Python——基本上所有开源项目都跑 cargo fmt。

clippy：超级 lint

clippy 是 Rust 的静态分析工具，比编译器的警告更进一步，会指出代码里可以改进的地方。

cargo clippy

clippy 的提示分几类：

// clippy 会提示：这个可以简化为 .is_empty()
if vec.len() == 0 { }

// clippy 会提示：用 ..= 更清晰
if x >= 0 && x <= 100 { }

// clippy 会提示：这个 clone 是不必要的
let s = some_string.clone().to_uppercase();

clippy 的建议几乎都是有道理的，养成习惯在提交代码前跑一次 cargo clippy，代码质量会有明显提升。

一个完整的工作流示例

把上面的工具串起来，典型的日常开发流程是这样的：

# 1. 创建项目
cargo new my-service
cd my-service

# 2. 添加依赖
cargo add tokio --features full
cargo add serde --features derive
cargo add anyhow

# 3. 开发阶段：边写边检查（不生成二进制，最快）
cargo check

# 4. 运行看效果
cargo run

# 5. 跑测试
cargo test

# 6. 提交前整理
cargo fmt
cargo clippy

# 7. 发布构建
cargo build --release

这套流程配合编辑器的 rust-analyzer 插件（VSCode、Neovim、IntelliJ 都支持），开发体验相当顺畅。rust-analyzer 会实时做类型检查、自动补全、inline 报错提示，基本上等价于一个实时运行的 cargo check。

小结

Rust 的工具链设计是它工程体验好的重要原因之一——官方统一维护，不用在生态里挑选，开箱即用。

几个关键点记住：

  • rustup 管理 Rust 版本，日常用 stable
  • cargo 是核心，几乎所有操作都通过它完成
  • Cargo.toml 是项目的配置中心，workspace 用来组织多包项目
  • 模块系统用 mod 声明、pub 控制可见性、use 引入路径
  • cargo fmt + cargo clippy 是提交代码前的标配
