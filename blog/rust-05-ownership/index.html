05 | 所有权——Rust 最重要的概念

发表于： 2026-04-19 更新于： 2026-04-20 分类于： Rust
字数： 4805 阅读：≈ 10分钟 浏览：

如果你用 Go 或 Python 写过代码，大概从来没想过一个问题：这段数据，到底谁负责把它从内存里删掉？

在 Go 里，你 new 一个对象，用完了不管，GC 会帮你清。在 Python 里更是如此，变量一多，引用计数器自动追踪，没人用了就回收。这套机制很舒服，但代价是你对内存的生命周期几乎没有控制权，而且 GC 随时可能暂停程序做垃圾回收（Stop-The-World），高性能场景下是个大麻烦。

Rust 的答案是：既不让你手动 free，也不搞 GC，而是在编译期就把内存的生命周期推断清楚。这套机制叫做所有权（Ownership）。

理解所有权，是从"会写 Go/Python"到"会写 Rust"的关键一跳。

一套房子只能有一个产权人

在讲代码之前，先建立一个直觉。

假设你有一套房子，这套房子有一本产权证。产权证上只能登记一个人的名字——这个人就是这套房子的所有者（Owner）。

  • 产权人可以住在这套房子里（使用这块内存）
  • 产权人可以把房子转让给别人（Move，所有权转移）
  • 产权人搬走、消失时，这套房子就会被收回（内存释放）
  • 同一时刻，不能有两个人都拿着这套房子的产权证——否则都消失时，房子会被收回两次（double-free，程序崩溃）

这就是 Rust 所有权的核心逻辑。

从内存问题出发：GC 解决了什么，又带来了什么

写 Go 时你可能遇到过这种情况：

// Go 代码
func newUser() *User {
    u := &User{Name: "Alice"}
    return u  // 返回局部变量的指针，Go 的 GC 会保证它活着
}

这在 Go 里完全没问题。GC 在后台追踪所有对象，只要还有人引用它，就不释放。

但 GC 有两个代价：

  1. Stop-The-World（STW）停顿：GC 运行时需要暂停程序，哪怕只是几毫秒，对实时性要求高的系统（游戏引擎、交易系统）也不能接受。
  2. 无法精确控制释放时机：你无法保证一个资源（比如数据库连接、文件句柄）在某个精确的时刻被释放，只能用 defer 或 context 来模拟。

C/C++ 让程序员手动 malloc/free，没有 GC 停顿，但极容易出错：

  • use-after-free：内存释放后继续用，读到垃圾数据甚至崩溃
  • double-free：同一块内存释放两次，堆结构损坏
  • 内存泄漏：忘记释放，内存一直涨

Rust 走的是第三条路：在编译期，通过所有权规则静态推断每块内存的生命周期。没有运行时 GC，也不需要手动 free，错误在编译时就被拦住。

堆与栈：桌面 vs 仓库

理解所有权之前，要先搞清楚内存的两个区域。

栈（Stack）——桌面：

  • 你桌面上放的东西，大小是固定的，放之前就知道占多少地方
  • 用完了（函数返回），东西自动清掉
  • 操作极快，就是移动一个指针

堆（Heap）——仓库：

  • 仓库可以存大件、可变大小的东西
  • 但要预约（向操作系统申请内存）、用完要归还（释放）
  • 生命周期和函数调用无关，必须显式管理

当你写 String::from("hello")，发生了什么？

栈（桌面）上存放 s 变量：
┌─────────────────────────┐
│  ptr  ──────────────────┼──→  堆（仓库）上的实际数据：
│  len: 5                 │     h e l l o
│  cap: 5                 │
└─────────────────────────┘

s 变量本身住在栈上，只有 3 个字段：指向堆的指针、当前长度、分配容量。真正的字节数据 hello 在堆上。

整数 let x = 5 就简单多了，直接住在栈上，没有堆数据。

这个区别很关键：有堆数据的类型，需要所有权来管理它的释放；只在栈上的类型，函数结束自动清。

所有权三条规则

Rust 的所有权建立在三条规则上：

  1. 每个值都有一个所有者（Owner）
  2. 同一时刻，一个值只能有一个所有者
  3. 所有者离开作用域时，值被自动丢弃（dropped）

结合房产证类比：

  • 每套房子（每个值）只有一本产权证（一个 Owner）
  • 产权证同一时刻只能有一个人持有
  • 持有人消失（离开作用域），房子被收回（内存释放）
fn main() {
    {
        let s = String::from("hello"); // s 获得了这块内存的产权证
        println!("{}", s);             // s 在有效范围内，可以使用
    } // s 离开作用域，Rust 自动调用 drop，堆内存归还操作系统

    // println!("{}", s); // 编译错误：s 已超出作用域
}

注意：这里没有任何 GC 在后台运行，也没有 defer。是编译器在编译期就确定了 s 在哪里消失，然后在那个位置插入释放代码。

Move 语义：Go 开发者最容易踩的坑

这是从 Go 切换到 Rust 时最大的思维跳跃。

在 Go 里，变量赋值是复制：

// Go 代码
s1 := "hello"
s2 := s1  // s1 的内容被复制给 s2
fmt.Println(s1, s2)  // 两个都可以用，完全没问题

在 Rust 里，对于持有堆数据的类型，赋值是"搬家"，不是"复印"：

fn main() {
    let s1 = String::from("hello");
    let s2 = s1; // 所有权从 s1"搬"到了 s2

    // println!("{}", s1); // 编译错误！s1 已经无效
    println!("{}", s2);    // 只有 s2 有效
}

用搬家来理解：你把家具从旧房子搬到新房子，旧房子就空了。你不能再用旧地址去取东西——那里什么都没有了。

为什么 Rust 要这样设计？

Move 之前：
s1 → [ptr ──→ heap: "hello", len:5, cap:5]
s2 → (未初始化)

如果允许"复制"（浅拷贝）：
s1 → [ptr ──→ heap: "hello", len:5, cap:5]
                      ↑
s2 → [ptr ──────────────┘     len:5, cap:5]

问题：s1 和 s2 的作用域结束时，两个都会尝试释放同一块堆内存 → double-free → 崩溃

Move 之后：
s1 → (无效，编译器标记为已移动)
s2 → [ptr ──→ heap: "hello", len:5, cap:5]

只有 s2 会在作用域结束时释放内存，问题消失。

新手最常犯的错误——在循环里 Move：

fn print_str(s: String) {  // 参数接收所有权
    println!("{}", s);
}  // s 在这里被 drop

fn main() {
    let messages = vec![
        String::from("hello"),
        String::from("world"),
    ];

    for msg in messages {  // msg 逐个获得 Vec 元素的所有权
        print_str(msg);    // 所有权再次移入函数
        // msg 已无效
    }
    // 这里 messages 也已无效（所有权在 for 循环里被移走了）

    // println!("{:?}", messages); // 编译错误！
}

Go 开发者看到这段代码会很困惑：我只是遍历了一下，怎么 messages 就没了？这就是 Move 语义——遍历时所有权被转移了。解决方案是借用（下一篇讲）。

Copy 语义：有些类型赋值就是复印

不是所有类型都遵循 Move。只存在栈上、大小固定、复制代价极低的类型使用 Copy 语义——赋值就是复印，两份都能用。

fn main() {
    let x = 5;
    let y = x;  // x 被完整复制给 y，不是 Move

    println!("x = {}, y = {}", x, y);  // 两个都有效，没问题
}

整数就像一张纸上的数字，复印一份，两份都是完整的，互不影响。

Copy 类型 vs 非 Copy 类型对照表：

类型Copy?原因
i32, u64, f64 等数值是纯栈数据，复制极快
bool是纯栈数据
char是纯栈数据
(i32, bool) 元组是所有字段都是 Copy
[i32; 5] 数组是所有元素都是 Copy
String否持有堆内存
Vec<T>否持有堆内存
Box<T>否持有堆内存
含 String 字段的结构体否间接持有堆内存

判断规则只有一条：如果一个类型持有堆上的数据，它就不是 Copy。

所有权在函数调用中的流动

把值传给函数，和赋值一样，会发生 Move（或 Copy）。

fn takes_ownership(s: String) {  // s 接收所有权（Move 进来）
    println!("拿到了：{}", s);
}  // s 离开作用域，String 被释放

fn makes_copy(x: i32) {  // x 是 Copy，不发生 Move
    println!("复制到：{}", x);
}  // x 离开作用域，但没有堆内存需要释放

fn main() {
    let s = String::from("hello");
    takes_ownership(s);    // s 的所有权移入函数
    // println!("{}", s);  // 编译错误！s 已无效

    let x = 5;
    makes_copy(x);         // x 被复制传入
    println!("{}", x);     // 没问题，x 仍然有效
}

这就是"临时委托"的反面教材：你把房子的产权证直接给了中介，中介用完就烧掉了，你再也拿不回来。

函数可以通过返回值"归还"所有权：

fn borrow_and_return(s: String) -> String {
    println!("看了一眼：{}", s);
    s  // 把所有权原封不动还回去
}

fn main() {
    let s1 = String::from("hello");
    let s2 = borrow_and_return(s1);  // s1 移入，s2 获得返回值
    println!("{}", s2);              // 可以继续用
}

但这样写很麻烦——每次调用函数都要把值传进去再传回来。这正是 Rust 引入**借用（borrowing）**机制的原因，下一篇详细讲。

clone()：我就是要两份，显式说出来

如果确实需要两个独立的副本，调用 clone()：

fn main() {
    let s1 = String::from("hello");
    let s2 = s1.clone();  // 深拷贝：复制堆上的数据，创建新的堆分配

    println!("s1 = {}, s2 = {}", s1, s2);  // 两个都有效
}

clone() 会完整复制堆上的数据，有内存分配和数据复制的代价。Rust 的设计哲学是：昂贵的操作要显式写出来，不能默默发生。

对比 Go：

// Go 代码
type User struct {
    Name string
    Tags []string  // 含指针字段
}

u1 := User{Name: "Alice", Tags: []string{"admin"}}
u2 := u1  // 看起来是复制，实际是浅拷贝
u2.Tags[0] = "guest"
fmt.Println(u1.Tags[0])  // "guest"！u1 也被改了！

Go 的结构体赋值是浅拷贝，含指针字段时两个变量共享底层数据，这是很多 Go bug 的来源。Rust 要求你用 clone() 明确表达"我要深拷贝"，不让这种隐性共享悄悄发生。

Drop 与 RAII：租约到期自动退房

当一个值离开作用域，Rust 自动调用它的 drop 方法。这不是 GC，是编译器在你代码里确定性地插入释放逻辑。

类比：你租了一套房子，租约写明"离开时自动退租，押金原路退回"。你不需要记得打电话给房东，合约到期自动执行。

struct DbConnection {
    name: String,
}

impl Drop for DbConnection {
    fn drop(&mut self) {
        println!("关闭连接：{}", self.name);
        // 真实场景：这里断开网络连接、释放资源
    }
}

fn main() {
    let conn1 = DbConnection { name: String::from("主库") };
    let conn2 = DbConnection { name: String::from("从库") };
    println!("两个连接已建立");
}  // conn2 先 drop，conn1 后 drop（后进先出，像栈一样）

输出：

两个连接已建立
关闭连接：从库
关闭连接：主库

和 Go/Python 的对比：

Go 用 defer 来模拟：

conn := openDB()
defer conn.Close()  // 手动写，忘了就泄漏

Python 用 with 语句：

with open("file.txt") as f:
    # f 在 with 块结束时关闭
    pass
# 但 with 之外的对象，只能靠 GC 的不确定时机

Rust 的 Drop 是语言级别的、确定性的：不需要 defer，不需要 with，作用域结束就释放，100% 可靠，不依赖 GC 时机。

为什么不能直接调用 .drop()？

fn main() {
    let s = String::from("hello");
    // s.drop();  // 编译错误！不能直接调用
}
// 如果能直接调用，作用域结束时还会再 drop 一次 → double-free

如果确实需要提前释放，用标准库提供的 std::mem::drop()：

fn main() {
    let conn = DbConnection { name: String::from("临时连接") };
    // ... 用完了
    std::mem::drop(conn);  // 提前释放，conn 在这之后无效
    println!("连接已关闭，继续做别的事");
}

std::mem::drop() 接收所有权，函数返回时触发 Drop，之后 conn 已被 move 走，作用域结束时不会再次释放。

新手常见误区

误区一：以为赋值是复制（Go 习惯带来的）

// 错误代码
let s1 = String::from("hello");
let s2 = s1;
println!("{}", s1);  // 编译错误：value borrowed here after move

原因：String 是 Move 语义，s1 赋给 s2 后，s1 失效。

修复：如果需要两份，用 clone()；如果只是想"看"，用借用（&s1，下一篇讲）。

let s1 = String::from("hello");
let s2 = s1.clone();  // 深拷贝
println!("{} {}", s1, s2);  // 两个都有效

误区二：传给函数后还想继续用

// 错误代码
fn print_name(name: String) {
    println!("{}", name);
}

fn main() {
    let name = String::from("Alice");
    print_name(name);
    println!("name is: {}", name);  // 编译错误：name 已被 move 进函数
}

原因：name 的所有权移入 print_name，函数结束后被释放。

修复：让函数接收引用而不是值（下一篇详解），或让函数返回所有权。

fn print_name(name: &String) {  // 接收引用，不转移所有权
    println!("{}", name);
}

fn main() {
    let name = String::from("Alice");
    print_name(&name);         // 传引用
    println!("name is: {}", name);  // 没问题，name 仍然有效
}

误区三：在循环里 Move 导致集合失效

// 错误代码
let names = vec![String::from("Alice"), String::from("Bob")];
for name in names {  // names 的所有权被 for 循环消耗
    println!("{}", name);
}
println!("{:?}", names);  // 编译错误：names 已被 move

原因：for name in names 会消耗 names 的所有权，逐个 move 出元素。

修复：遍历引用，不消耗所有权。

let names = vec![String::from("Alice"), String::from("Bob")];
for name in &names {  // 遍历引用，names 所有权不转移
    println!("{}", name);
}
println!("{:?}", names);  // 没问题，names 仍然有效

小结

Rust 所有权的核心逻辑只有一句话：

每块内存只有一个主人，主人消失时内存自动释放。

这一篇的关键点：

  • Go/Python 用 GC 管内存，有 STW 停顿和不确定的释放时机；Rust 用所有权在编译期解决，零运行时开销
  • 持有堆数据的类型（String、Vec 等）遵循 Move 语义：赋值或传参后原变量失效
  • 只在栈上的类型（整数、bool 等）遵循 Copy 语义：赋值是复制，两份都活着
  • 函数调用会转移所有权，返回值可以把所有权还回来
  • clone() 是显式深拷贝，代价明确，不允许隐性发生
  • Drop trait 实现确定性析构，比 defer/with 更可靠，是 RAII 的语言级保障

但现在你可能已经感受到了：每次传参都要转移所有权太麻烦了，很多时候我只是想"让别人看一眼"，并不想把产权证交出去。
