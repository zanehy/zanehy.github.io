17 | 并发模式实战

发表于： 2026-04-27 分类于： Rust
字数： 3473 阅读：≈ 7分钟 浏览：

上一篇我们把 Tokio 的核心 API 过了一遍：spawn、select!、Channel、JoinSet、spawn_blocking。这篇进入真正的实战——用这些积木搭出生产级的并发模式。

这就像学会了锤子、螺丝刀、锯子之后，开始学怎么造家具。单独的工具你都懂了，但组合起来还需要经验和套路。

模式一：生产者 / 消费者

这是最经典的并发模式，几乎所有涉及异步处理的系统都在用：任务不断涌入（生产者），一批 worker 负责处理（消费者），中间用 Channel 解耦。

基础版：单消费者 

use tokio::sync::mpsc;
use tokio::time::{sleep, Duration};

#[derive(Debug)]
struct Job {
    id: u32,
    payload: String,
}

async fn process_job(job: Job) -> String {
    sleep(Duration::from_millis(100)).await;
    format!("processed: {} - {}", job.id, job.payload)
}

#[tokio::main]
async fn main() {
    let (tx, mut rx) = mpsc::channel::<Job>(100);

    // 生产者
    let producer = tokio::spawn(async move {
        for i in 0..10 {
            tx.send(Job { id: i, payload: format!("data_{}", i) }).await.unwrap();
            println!("生产者: 提交 job {}", i);
        }
        // tx 在这里 drop，消费者会收到 None 并自动退出
    });

    // 消费者
    let consumer = tokio::spawn(async move {
        while let Some(job) = rx.recv().await {
            let result = process_job(job).await;
            println!("消费者: {}", result);
        }
        println!("消费者: channel 已关闭，退出");
    });

    tokio::join!(producer, consumer).0.unwrap();
}

关键设计：tx 在 producer 闭包里，producer 结束时 tx 自动 drop，rx.recv() 返回 None，消费者循环自然退出。不需要额外的停止信号。

进阶版：多消费者

标准库的 mpsc Receiver 不能 clone，多个 worker 共享需要额外处理。最简洁的方案是用 async-channel crate，它原生支持多消费者：

// Cargo.toml: async-channel = "2"
use async_channel;
use tokio::task::JoinSet;

#[tokio::main]
async fn main() {
    let (tx, rx) = async_channel::bounded::<u32>(100);

    // 生产者
    tokio::spawn(async move {
        for i in 0..20 {
            tx.send(i).await.unwrap();
        }
        // tx drop 时 channel 关闭，所有消费者退出
    });

    // 4 个 worker 共享同一个 rx（async_channel 的 receiver 可以 clone）
    let mut set = JoinSet::new();
    for worker_id in 0..4 {
        let rx = rx.clone();
        set.spawn(async move {
            while let Ok(job) = rx.recv().await {
                println!("Worker {}: 处理 {}", worker_id, job);
                tokio::time::sleep(tokio::time::Duration::from_millis(50)).await;
            }
            println!("Worker {} 退出", worker_id);
        });
    }

    while let Some(res) = set.join_next().await {
        res.unwrap();
    }
    println!("所有 worker 完成");
}

不引入额外依赖的话，用 Arc<Mutex<Receiver>> 也可以：

use tokio::sync::{mpsc, Mutex};
use std::sync::Arc;
use tokio::task::JoinSet;

#[tokio::main]
async fn main() {
    let (tx, rx) = mpsc::channel::<u32>(100);
    let rx = Arc::new(Mutex::new(rx));

    tokio::spawn(async move {
        for i in 0..20 { tx.send(i).await.unwrap(); }
    });

    let mut set = JoinSet::new();
    for worker_id in 0..4 {
        let rx = Arc::clone(&rx);
        set.spawn(async move {
            loop {
                // 注意：lock 的作用域要尽量短，拿到任务后立刻释放锁
                let job = { rx.lock().await.recv().await };
                match job {
                    Some(val) => {
                        println!("Worker {}: 处理 {}", worker_id, val);
                        tokio::time::sleep(tokio::time::Duration::from_millis(50)).await;
                    }
                    None => { println!("Worker {} 退出", worker_id); break; }
                }
            }
        });
    }

    while let Some(res) = set.join_next().await { res.unwrap(); }
}

这里有个容易犯的错：把锁持有到任务处理完再释放——那同一时刻只有一个 worker 在工作，完全失去了多消费者的意义。锁要在 recv() 之后立刻释放。

模式二：有界并发（控制并发上限）

生产者/消费者用固定数量的 worker 控制并发。但有时候任务是动态的，你有一批任务要跑，但不想一次性把所有任务都 spawn 出去（内存压力、连接池耗尽等）。

JoinSet 配合长度检查，可以优雅地实现有界并发：

use tokio::task::JoinSet;
use tokio::time::{sleep, Duration};

async fn run_task(id: u32) {
    sleep(Duration::from_millis(200)).await;
    println!("任务 {} 完成", id);
}

#[tokio::main]
async fn main() {
    let mut set = JoinSet::new();
    let max_concurrent = 5; // 最多同时跑 5 个

    for i in 0..20 {
        // 超过上限时，等一个完成再提交
        if set.len() >= max_concurrent {
            set.join_next().await;
        }
        set.spawn(run_task(i));
    }

    // 等待剩余任务
    while let Some(res) = set.join_next().await {
        res.unwrap();
    }
    println!("全部完成");
}

这个模式比固定 worker 数的工作池更灵活：任务是动态提交的，每个任务直接 spawn，不需要 Channel 中转，代码也更简洁。

模式三：优雅关闭（Graceful Shutdown）

进程收到终止信号时，不能直接退出，要：停止接收新请求 → 等待进行中的任务完成 → 释放资源 → 退出。

方案一：CancellationToken（推荐）

tokio-util 提供的 CancellationToken 是最干净的方案：

// Cargo.toml: tokio-util = { version = "0.7", features = ["rt"] }
use tokio_util::sync::CancellationToken;
use tokio::signal;
use tokio::task::JoinSet;

#[tokio::main]
async fn main() {
    let token = CancellationToken::new();
    let mut set = JoinSet::new();

    for i in 0..3 {
        let child = token.child_token(); // 父 token 取消时子 token 也取消
        set.spawn(async move { worker(i, child).await });
    }

    signal::ctrl_c().await.unwrap();
    println!("\n收到停止信号，开始优雅关闭...");
    token.cancel(); // 一次调用，通知所有 worker

    while let Some(res) = set.join_next().await { res.unwrap(); }
    println!("优雅关闭完成");
}

async fn worker(id: usize, token: CancellationToken) {
    loop {
        tokio::select! {
            _ = token.cancelled() => {
                // 在这里做清理：关闭连接、刷新缓冲区等
                println!("Worker {}: 正在清理，退出", id);
                break;
            }
            _ = do_work(id) => {} // 正常处理一轮
        }
    }
    println!("Worker {} 已退出", id);
}

async fn do_work(id: usize) {
    println!("Worker {}: 处理任务中...", id);
    tokio::time::sleep(tokio::time::Duration::from_millis(500)).await;
}

方案二：broadcast Channel（不依赖额外 crate） 

use tokio::sync::broadcast;
use tokio::signal;
use tokio::task::JoinSet;

#[tokio::main]
async fn main() {
    let (shutdown_tx, _) = broadcast::channel::<()>(1);
    let mut set = JoinSet::new();

    for i in 0..3 {
        let mut rx = shutdown_tx.subscribe();
        set.spawn(async move {
            loop {
                tokio::select! {
                    _ = rx.recv() => {
                        println!("Worker {}: 收到关闭信号，退出", i);
                        break;
                    }
                    _ = do_work(i) => {}
                }
            }
        });
    }

    signal::ctrl_c().await.unwrap();
    println!("\n开始关闭...");
    let _ = shutdown_tx.send(()); // 广播给所有订阅者

    while let Some(res) = set.join_next().await { res.unwrap(); }
    println!("关闭完成");
}

async fn do_work(id: usize) {
    tokio::time::sleep(tokio::time::Duration::from_millis(500)).await;
    println!("Worker {}: 任务处理完成", id);
}

两种方案效果相同，CancellationToken 语义更清晰，broadcast 不需要额外依赖。实际项目里选一个就好。

模式四：Actor 模式

Actor 模式是消息传递的最佳实践形态：每个 Actor 是一个独立的 async 任务，独占自己的状态，外部只能通过消息与它交互，完全不需要锁。

这个模式在 Erlang/Elixir 里大量使用，Rust 用 Channel 实现同样自然。

use tokio::sync::{mpsc, oneshot};
use std::collections::HashMap;

// 用枚举表示所有可能的命令
enum Command {
    Get {
        key: String,
        respond_to: oneshot::Sender<Option<String>>,
    },
    Set {
        key: String,
        value: String,
    },
    Delete {
        key: String,
    },
}

// Actor 本体：独占 HashMap，通过命令交互，完全不需要 Mutex
async fn cache_actor(mut rx: mpsc::Receiver<Command>) {
    let mut store: HashMap<String, String> = HashMap::new();

    while let Some(cmd) = rx.recv().await {
        match cmd {
            Command::Get { key, respond_to } => {
                let _ = respond_to.send(store.get(&key).cloned());
            }
            Command::Set { key, value } => {
                store.insert(key, value);
            }
            Command::Delete { key } => {
                store.remove(&key);
            }
        }
    }
}

// 封装一个句柄，让调用方不需要关心 Channel 细节
#[derive(Clone)]
struct CacheHandle {
    tx: mpsc::Sender<Command>,
}

impl CacheHandle {
    fn new() -> Self {
        let (tx, rx) = mpsc::channel(100);
        tokio::spawn(cache_actor(rx));
        CacheHandle { tx }
    }

    async fn get(&self, key: &str) -> Option<String> {
        let (resp_tx, resp_rx) = oneshot::channel();
        self.tx.send(Command::Get {
            key: key.to_string(),
            respond_to: resp_tx,
        }).await.unwrap();
        resp_rx.await.unwrap()
    }

    async fn set(&self, key: &str, value: &str) {
        self.tx.send(Command::Set {
            key: key.to_string(),
            value: value.to_string(),
        }).await.unwrap();
    }
}

#[tokio::main]
async fn main() {
    let cache = CacheHandle::new();

    cache.set("name", "Rust").await;
    cache.set("version", "1.78").await;

    println!("{:?}", cache.get("name").await);    // Some("Rust")
    println!("{:?}", cache.get("missing").await); // None

    // CacheHandle 可以 clone，多个地方共享同一个 Actor
    let cache2 = cache.clone();
    tokio::spawn(async move {
        println!("{:?}", cache2.get("version").await); // Some("1.78")
    }).await.unwrap();
}

Actor 模式的优势：状态完全由 Actor 内部管理，外部拿不到引用，不可能有竞争条件，不需要任何锁。代价是所有操作都要走 Channel，有一定延迟和样板代码。

适合用 Actor 的场景：

  • 需要复杂事务性操作（读-改-写原子性）
  • 状态访问模式复杂，锁粒度难以控制
  • 需要序列化所有操作（比如数据库连接池）

共享状态 vs 消息传递：怎么选

这是高频问题，没有绝对的答案，只有适用场景。

共享状态（Arc<RwLock<T>>）适合：

  • 读多写少的数据（缓存、配置）
  • 简单的计数器、标志位
  • 数据结构本身复杂，但操作简单
use std::sync::Arc;
use tokio::sync::RwLock;
use std::collections::HashMap;

type SharedConfig = Arc<RwLock<HashMap<String, String>>>;

async fn read(cfg: &SharedConfig, key: &str) -> Option<String> {
    cfg.read().await.get(key).cloned() // 多个读者可以并发
}

async fn update(cfg: &SharedConfig, key: String, val: String) {
    cfg.write().await.insert(key, val); // 写者独占
}

消息传递（Channel / Actor）适合：

  • 写密集或操作复杂（容易产生锁竞争）
  • 生产者消费者天然解耦
  • 需要背压（Channel 满时自动阻塞生产者，防止过载）
  • 状态访问需要序列化保证

一个实用的经验法则：如果你发现自己在写 Arc<Mutex<...>> 并且锁的持有时间越来越长，或者加锁逻辑越来越复杂，那就是应该换 Actor 模式的信号。

常见陷阱

死锁

Rust 类型系统防不住死锁，运行时才会暴露。最常见的原因是加锁顺序不一致：

// 任务 1：先锁 A，再锁 B
// 任务 2：先锁 B，再锁 A
// → 互相等待，永远不会完成

// 解法：固定加锁顺序，全局约定总是先 A 后 B

另一个常见死锁场景：在持有 std::sync::Mutex 锁的情况下 .await：

use std::sync::Mutex;

async fn bad_example(lock: &Mutex<Vec<i32>>) {
    let mut guard = lock.lock().unwrap(); // 持有同步 Mutex 锁
    some_async_op().await;               // await 期间锁没有释放！
    guard.push(1);                       // 可能导致死锁
}

// 解法：换用 tokio::sync::Mutex，或者在 await 之前释放锁
async fn good_example(lock: &tokio::sync::Mutex<Vec<i32>>) {
    let mut guard = lock.lock().await; // tokio Mutex 可以安全跨 await
    some_async_op().await;
    guard.push(1);
}

Channel 背压

Channel 满时，send().await 会阻塞生产者，这是有意为之的背压机制——消费者跟不上时自动给生产者踩刹车。

但如果你不想阻塞，可以用 try_send：

match tx.try_send(job) {
    Ok(()) => {} // 发送成功
    Err(mpsc::error::TrySendError::Full(_)) => {
        // channel 满了，可以丢弃、记录、或换用其他策略
        println!("队列已满，丢弃任务");
    }
    Err(mpsc::error::TrySendError::Closed(_)) => {
        println!("接收端已关闭");
    }
}

任务泄漏

tokio::spawn 返回的 JoinHandle 被 drop 后，任务并不会停止，它会继续在后台跑直到完成或程序退出：

// handle 被 drop，但任务仍在后台运行！
let _ = tokio::spawn(async {
    loop { tokio::time::sleep(Duration::from_secs(1)).await; }
});

用 JoinSet 管理任务生命周期可以避免这个问题——JoinSet drop 时，所有未完成的任务自动取消。

小结

  • 生产者/消费者：mpsc::channel 解耦，发送端全部 drop 即为关闭信号；多消费者用 async-channel 或 Arc<Mutex<Receiver>>，注意锁的粒度要尽量小
  • 有界并发：用 JoinSet::len() 检查并发数上限，动态提交任务比固定 worker 数更灵活
  • 优雅关闭：CancellationToken 语义最清晰，broadcast 不需要额外依赖；核心思路是给 worker 发停止信号，等它完成当前任务再退出
  • Actor 模式：任务独占状态，外部只通过 Channel 交互，完全免锁；CacheHandle 这类封装可以让调用方无感知底层 Channel 细节
  • 选型原则：读多写少用共享状态；操作复杂/写密集/需要解耦用消息传递；发现锁逻辑越来越复杂时，考虑切换到 Actor
