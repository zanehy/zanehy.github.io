为什么是 Rust？它到底解决了什么问题

发表于： 2026-04-17 分类于： Rust
字数： 2735 阅读：≈ 6分钟 浏览：

在学一门语言之前，先搞明白它为什么存在，比死记语法有用得多。

Rust 不是凭空冒出来的。它的诞生，是软件行业几十年来一个没有被彻底解决的矛盾逼出来的。

那个悬而未决的矛盾

写程序，从来都有两个方向可以走。

第一条路：追求极致性能，自己管内存。

C 和 C++ 走的就是这条路。你需要什么内存，自己 malloc，用完自己 free。操作系统把控制权完全交给你，没有任何额外开销，代码跑得飞快。Linux 内核、数据库引擎、浏览器的渲染器——这类软件都选这条路。

但代价是什么？内存管理是出了名的容易出错：

  • 忘了 free，内存泄漏，程序越跑越慢，最终 OOM
  • free 了之后还在用，悬空指针（use-after-free），直接崩溃，或者更糟，被攻击者利用
  • 用了数组越界，栈溢出，缓冲区溢出——这类漏洞至今仍是安全漏洞的主要来源

微软曾统计过，其安全补丁里大约 70% 是内存安全相关的 bug。Google 的 Chrome 团队给出了类似的数字。这些 bug 不是因为工程师水平差，而是因为手动内存管理本身就是一件很难做对的事。

第二条路：让运行时帮你管内存。

Java、Go、Python 走的是这条路。有垃圾回收器（GC），你分配内存，GC 自动回收，不需要操心悬空指针，不会内存泄漏（通常）。

代价也是明摆着的：

  • GC 有暂停（Stop-the-World），哪怕是 Go 这种已经优化得很好的并发 GC，在某些场景下仍然会有延迟抖动
  • 运行时本身有开销，内存占用更高
  • 无法精细控制内存布局，某些底层需求做不到

所以这两条路，长期以来是一个二选一的问题：要性能，就手动管内存冒风险；要安全，就用 GC 付出代价。

Rust 的答案

Rust 说：这两条路不是非此即彼，有第三种选择。

Rust 的核心思路是：把内存管理的正确性放到编译期来保证。 不靠 GC，不靠程序员的自律，而是让编译器通过一套严格的规则，在你的代码进入运行时之前就拒绝掉所有可能的内存错误。

这套规则的核心就是所有权系统（Ownership），我们后面会专门用几篇文章来讲它。但在这里，你只需要知道一个结论：

Rust 的编译器能够在不引入 GC 的前提下，从静态分析层面保证内存安全。

这意味着：

  • 没有悬空指针——编译器不允许你持有一个已经被释放的值的引用
  • 没有数据竞争——编译器不允许多个线程同时读写同一块内存（在没有同步原语的情况下）
  • 没有空指针——Rust 没有 null，可能缺失的值用 Option<T> 表达

这些保证不是运行时检查，是编译期保证。如果你的代码违反了这些规则，它根本编译不过去。

零成本抽象

Rust 还有另一个核心原则：零成本抽象（Zero-Cost Abstraction）。

这个词是从 C++ 那边继承来的，含义是：你使用高层抽象（迭代器、泛型、闭包）写出来的代码，编译后的性能等价于手写的底层循环，没有额外的运行时开销。

来看一个对比：

// 用迭代器写
let sum: i32 = (1..=100).filter(|x| x % 2 == 0).sum();

// 手写循环
let mut sum = 0i32;
let mut i = 1i32;
while i <= 100 {
    if i % 2 == 0 {
        sum += i;
    }
    i += 1;
}

Rust 的迭代器在编译后会被优化成和手写循环等价的代码，甚至会利用 SIMD 等 CPU 指令进一步加速。你写的是高层抽象，得到的是底层性能。

这就是 Rust 性能能和 C/C++ 比肩的原因。

和 Go/Python 的本质差异

如果你有 Go 或 Python 背景，有几个思维上的转变值得提前说清楚。

关于内存：

Go 和 Python 都有 GC，你从来不需要思考"这个值现在在哪里，谁拥有它，什么时候该释放"。Rust 没有 GC，这些问题你必须通过所有权规则来明确表达——但好消息是，编译器会帮你检查，不是靠自律。

关于类型系统：

Python 是动态类型，运行时才知道类型。Go 是静态类型但比较简单。Rust 的类型系统是三者里最强的：泛型、Trait、生命周期参数，组合起来能表达非常复杂的约束，很多在其他语言里需要运行时检查的错误，Rust 在编译期就能发现。

关于并发：

Go 的 goroutine 模型很友好，并发写起来顺手，但数据竞争依然可能在运行时出现（go vet 和 race detector 是运行时工具）。Rust 的并发安全是编译期保证的——如果代码编译过了，就不存在数据竞争，这就是所谓的"无畏并发"。

关于错误处理：

Python 靠异常，Go 靠多返回值（err != nil 那套），Rust 靠 Result<T, E> 类型。Rust 的设计哲学是：错误是值，必须显式处理，不能被忽略。编译器会强迫你面对每一个可能失败的操作。

关于学习曲线：

说实话，Rust 的学习曲线是陡峭的。编译器会拒绝你在其他语言里习以为常的写法，你会和借用检查器"斗争"一段时间。但这个过程恰恰是 Rust 在训练你以一种更精确的方式思考内存和数据流向。多数人的反馈是：过了这道坎之后，代码质量和对底层的理解都会有一个台阶式的提升。

Rust 的适用场景

Rust 不是适合所有场景的。在选择之前，先搞清楚它在哪里发光。

非常适合 Rust 的场景：

  • 系统编程：操作系统组件、驱动程序、嵌入式开发——需要直接控制硬件和内存
  • 高性能服务：对延迟敏感、需要低 GC 暂停的网络服务，比如消息队列、代理服务器
  • 命令行工具：Rust 的二进制文件体积小、启动快，ripgrep、fd、bat 都是用 Rust 写的
  • WebAssembly：Rust 是 WASM 生态里最成熟的语言之一
  • 安全敏感代码：加密库、解析器——内存安全的编译期保证让这类代码更可靠

相对不适合的场景：

  • 快速原型和脚本：如果需要快速验证想法，Python 的开发效率仍然无可替代
  • CRUD 业务后端：如果你的瓶颈不在计算和内存，而在数据库 I/O，Rust 的复杂度可能得不偿失
  • 团队里没有 Rust 经验：Rust 的学习成本是真实的，引入时要考虑团队的承受能力

真实世界里的 Rust

Rust 在工业界的落地情况已经相当可观：

  • Linux 内核：自 6.1 版本起，Linux 内核正式引入 Rust 作为第二语言，已有驱动程序用 Rust 编写
  • Windows：微软在逐步用 Rust 重写部分 Windows 内核组件
  • Android：Google 在 Android 操作系统中引入 Rust，用于编写新的底层组件
  • AWS：Firecracker（Lambda 和 Fargate 背后的虚拟机监控程序）用 Rust 编写
  • Cloudflare：边缘网络的核心组件大量使用 Rust
  • Discord：将部分服务从 Go 迁移到 Rust，延迟显著降低

Rust 连续多年在 Stack Overflow 开发者调查中被评为"最受喜爱的编程语言"（2023 年改为"最受期待学习的语言"，依然排名靠前）——这不只是因为它性能好，更是因为它的编程体验在某种意义上让开发者觉得"被保护了"。

小结

Rust 的存在是为了解决一个长期悬而未决的矛盾：在不引入 GC 的前提下实现内存安全。它通过编译期的所有权系统做到了这一点，同时通过零成本抽象保证了性能不打折扣。

和 Go/Python 相比，Rust 的核心差异不在语法，而在思维模型：你需要明确表达每一个值的归属、借用和生命周期。这是学习曲线陡峭的原因，也是它能给你带来其他语言给不了的保证的原因。
