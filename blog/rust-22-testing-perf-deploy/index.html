22 | 测试、性能与部署：把 API 送上生产

发表于： 2026-04-29 分类于： Rust
字数： 2564 阅读：≈ 6分钟 浏览：

三篇实战项目的最后一篇。前两篇把 task-api 从骨架搭到具备生产级错误处理和中间件，这一篇补上最后三块拼图：测试、性能调优和Docker 部署。

测试

Rust 的测试体系是内置的，不需要额外的测试框架。cargo test 直接运行，编译器帮你检查测试代码的类型安全。

单元测试

单元测试放在被测模块的同一文件里，用 #[cfg(test)] 模块包裹：

// src/store.rs 末尾追加

#[cfg(test)]
mod tests {
    use super::*;
    use crate::models::{Task, TaskStatus, UpdateTaskRequest};

    fn make_store() -> TaskStore {
        TaskStore::new()
    }

    #[test]
    fn test_create_and_get() {
        let mut store = make_store();
        let task = Task::new("写测试".to_string(), None);
        let id = task.id;
        store.create(task);

        let found = store.get(&id).unwrap();
        assert_eq!(found.title, "写测试");
        assert_eq!(found.status, TaskStatus::Todo);
    }

    #[test]
    fn test_update_task() {
        let mut store = make_store();
        let task = Task::new("原标题".to_string(), None);
        let id = task.id;
        store.create(task);

        let req = UpdateTaskRequest {
            title: Some("新标题".to_string()),
            description: None,
            status: Some(TaskStatus::Done),
        };
        let updated = store.update(&id, req).unwrap();
        assert_eq!(updated.title, "新标题");
        assert_eq!(updated.status, TaskStatus::Done);
    }

    #[test]
    fn test_delete_task() {
        let mut store = make_store();
        let task = Task::new("待删除".to_string(), None);
        let id = task.id;
        store.create(task);

        assert!(store.delete(&id));
        assert!(store.get(&id).is_none());
        // 再次删除应返回 false
        assert!(!store.delete(&id));
    }

    #[test]
    fn test_list_sorted_by_created_at() {
        let mut store = make_store();
        // 快速插入两个任务，chrono 精度足够区分
        store.create(Task::new("第一个".to_string(), None));
        std::thread::sleep(std::time::Duration::from_millis(5));
        store.create(Task::new("第二个".to_string(), None));

        let tasks = store.list();
        assert_eq!(tasks[0].title, "第一个");
        assert_eq!(tasks[1].title, "第二个");
    }
}

#[cfg(test)] 让这段代码只在 cargo test 时编译，不会进入生产二进制。make_store() 是测试辅助函数，避免每个测试重复初始化。

集成测试

集成测试放在 tests/ 目录下，测试的是对外暴露的 HTTP 接口，更接近真实使用场景。axum 提供了 axum::body::to_bytes 和 tower::ServiceExt 来在不启动真实 HTTP server 的情况下测试路由。

集成测试通过 crate 名引用项目模块，因此需要先把公共模块暴露为库。在 src/lib.rs 里添加：

// src/lib.rs
pub mod errors;
pub mod handlers;
pub mod models;
pub mod routes;
pub mod store;

main.rs 里把原来的 mod 声明全部改成 use task_api::* 风格引用即可，或者直接保留 mod 声明——Rust 允许 main.rs 和 lib.rs 并存，两者共享同一套源文件，互不冲突。

然后新建 tests/api_test.rs：

// tests/api_test.rs

use axum::{
    body::Body,
    http::{Request, StatusCode},
};
use serde_json::{json, Value};
use std::sync::{Arc, RwLock};
use task_api::{routes, store::TaskStore};
use tower::ServiceExt; // 提供 .oneshot() 方法

fn test_app() -> axum::Router {
    let store = Arc::new(RwLock::new(TaskStore::new()));
    routes::create_router(store)
}

#[tokio::test]
async fn test_create_and_list_tasks() {
    let app = test_app();

    // 创建任务
    let create_resp = app
        .clone()
        .oneshot(
            Request::builder()
                .method("POST")
                .uri("/tasks")
                .header("content-type", "application/json")
                .body(Body::from(
                    serde_json::to_string(&json!({"title": "集成测试任务"})).unwrap(),
                ))
                .unwrap(),
        )
        .await
        .unwrap();

    assert_eq!(create_resp.status(), StatusCode::CREATED);

    let body = axum::body::to_bytes(create_resp.into_body(), usize::MAX)
        .await
        .unwrap();
    let task: Value = serde_json::from_slice(&body).unwrap();
    assert_eq!(task["title"], "集成测试任务");
    assert_eq!(task["status"], "todo");

    // 列出任务
    let list_resp = test_app()
        .oneshot(Request::builder().uri("/tasks").body(Body::empty()).unwrap())
        .await
        .unwrap();

    assert_eq!(list_resp.status(), StatusCode::OK);
    let body = axum::body::to_bytes(list_resp.into_body(), usize::MAX)
        .await
        .unwrap();
    let tasks: Value = serde_json::from_slice(&body).unwrap();
    assert!(tasks.as_array().unwrap().len() >= 1);
}

#[tokio::test]
async fn test_get_nonexistent_task() {
    let app = test_app();

    let resp = app
        .oneshot(
            Request::builder()
                .uri("/tasks/00000000-0000-0000-0000-000000000000")
                .body(Body::empty())
                .unwrap(),
        )
        .await
        .unwrap();

    assert_eq!(resp.status(), StatusCode::NOT_FOUND);

    let body = axum::body::to_bytes(resp.into_body(), usize::MAX)
        .await
        .unwrap();
    let error: Value = serde_json::from_slice(&body).unwrap();
    assert_eq!(error["code"], "NOT_FOUND");
}

.oneshot() 把一个请求发进路由，得到响应，整个过程在内存里完成，速度极快，也不需要绑定端口。这是测试 axum 应用的标准方式。

运行测试：

cargo test                    # 跑所有测试
cargo test test_create        # 只跑名字含 test_create 的测试
cargo test -- --nocapture     # 显示测试中的 println! 输出

编译优化

Rust 的 cargo build 默认是 debug 模式——编译快，运行慢（未优化，带调试信息）。发布时要用 --release：

cargo build --release

release 二进制在 target/release/ 下，体积和性能都与 debug 差异显著。可以在 Cargo.toml 里进一步调整 release profile：

[profile.release]
opt-level = 3        # 最高优化级别（默认就是 3）
lto = true           # 链接时优化，跨 crate 内联，显著减小体积
codegen-units = 1    # 单个代码生成单元，允许更激进的优化（编译慢但结果更优）
strip = true         # 剥离符号表，进一步减小二进制体积
panic = "abort"      # panic 时直接终止而不展开栈，减小体积

lto = true 对 Rust 项目效果明显——axum 这类框架有大量跨 crate 的小函数，LTO 可以把它们内联掉，同时消除死代码。实测开启 LTO 后，简单 axum 项目的二进制可以从 ~8MB 缩减到 ~3MB。

基准测试

cargo bench 是内置的基准测试工具，需要配合 #[bench] 属性（目前稳定版只支持 criterion 这类第三方库）。加入 criterion：

[dev-dependencies]
criterion = { version = "0.5", features = ["html_reports"] }

[[bench]]
name = "store_bench"
harness = false

benches/store_bench.rs：

use criterion::{black_box, criterion_group, criterion_main, Criterion};
use task_api::{models::Task, store::TaskStore};

fn bench_create_tasks(c: &mut Criterion) {
    c.bench_function("store_create_1000", |b| {
        b.iter(|| {
            let mut store = TaskStore::new();
            for i in 0..1000 {
                let task = Task::new(format!("任务 {}", i), None);
                store.create(black_box(task));
            }
        })
    });
}

fn bench_list_tasks(c: &mut Criterion) {
    // store 在 bench_function 外初始化，测的是固定 1000 条数据的 list 性能
    // 注意：如果你想测 create 的性能，必须在 b.iter 里面创建新 store
    // 否则每次迭代都是往越来越大的 HashMap 插入，测到的是不断增长的性能曲线
    let mut store = TaskStore::new();
    for i in 0..1000 {
        store.create(Task::new(format!("任务 {}", i), None));
    }

    c.bench_function("store_list_1000", |b| {
        b.iter(|| {
            black_box(store.list());
        })
    });
}

criterion_group!(benches, bench_create_tasks, bench_list_tasks);
criterion_main!(benches);

cargo bench

criterion 会多次运行并统计均值、标准差，生成 HTML 报告。black_box 防止编译器把基准测试的计算优化掉，确保结果真实。

Docker 多阶段构建

Rust 的编译镜像很大（官方 rust:latest 约 1.5GB），但运行时只需要一个静态链接的二进制。多阶段构建就是为了解决这个问题：编译阶段用大镜像，运行阶段用最小镜像。

Dockerfile：

# ---- 构建阶段 ----
FROM rust:1.77-slim AS builder

WORKDIR /app

# 先复制依赖文件，利用 Docker 层缓存
# 只有 Cargo.toml/Cargo.lock 变化时才重新下载依赖
COPY Cargo.toml Cargo.lock ./

# 创建空 main，让依赖先编译缓存
RUN mkdir src && echo "fn main() {}" > src/main.rs
RUN cargo build --release
RUN rm -f target/release/deps/task_api*

# 复制真正的源码并编译
COPY src ./src
RUN cargo build --release

# ---- 运行阶段 ----
FROM debian:bookworm-slim AS runtime

# 安装运行时依赖（动态链接的 glibc、SSL）
RUN apt-get update && apt-get install -y \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# 只从构建阶段复制二进制
COPY --from=builder /app/target/release/task-api .

EXPOSE 3000

ENV RUST_LOG=info

CMD ["./task-api"]

这个 Dockerfile 有几个值得注意的技巧：

依赖缓存层：先复制 Cargo.toml 和 Cargo.lock，用空 main.rs 触发依赖编译。依赖不变时，这一层会命中缓存，只重新编译你的源码，大幅缩短 CI 时间。

最终镜像用 debian:bookworm-slim：比 scratch（完全空镜像）更实用——有 shell 方便调试，有 glibc 兼容大多数动态链接。如果想要更小，可以用 musl target 编译静态二进制再配 scratch：

FROM rust:1.77-slim AS builder
RUN rustup target add x86_64-unknown-linux-musl
RUN apt-get update && apt-get install -y musl-tools
# ...
RUN cargo build --release --target x86_64-unknown-linux-musl

FROM scratch AS runtime
COPY --from=builder /app/target/x86_64-unknown-linux-musl/release/task-api /task-api
EXPOSE 3000
CMD ["/task-api"]

scratch 镜像只有二进制本身，最终镜像可以压缩到 10MB 以内。

构建和运行：

docker build -t task-api .
docker run -p 3000:3000 -e RUST_LOG=debug task-api

小结

这三篇走完了一个 Rust Web 服务从零到上线的完整路径：

篇内容
第 20 篇项目结构、数据模型、Handler、路由注册、服务启动
第 21 篇统一错误处理、Tower 中间件、优雅关闭
