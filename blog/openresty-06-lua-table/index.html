06 | Lua Table 深度解析

发表于： 2026-03-10 分类于： OpenResty
字数： 2292 阅读：≈ 5分钟 浏览：

在前两篇文章里，我们把 Lua 的数据类型和流程控制都梳理了一遍。但有一个类型只是顺带提了一下——table。

这篇文章专门来聊它，因为 table 在 Lua 里的地位实在特殊：它是 Lua 里唯一的数据结构。数组是它，字典是它，模块是它，对象也是它。搞懂 table，相当于搞懂了 Lua 的半壁江山。

table 的双重身份

Lua 里没有内置的"数组类型"或"字典类型"，只有一个 table，但它能同时扮演这两个角色。

作为数组 

local fruits = {"apple", "banana", "orange"}

print(fruits[1])  -- apple（注意：Lua 数组下标从 1 开始！）
print(fruits[2])  -- banana
print(#fruits)    -- 3（# 获取长度）

-- 遍历数组
for i, v in ipairs(fruits) do
    print(i, v)
end

注意那个 [1]——Lua 的数组下标从 1 开始，不是 0。这是很多从 C / Python / JavaScript 转过来的开发者最先踩的坑。

ipairs 是专门用来遍历数组部分的迭代器，它从下标 1 开始，遇到第一个 nil 就停下来。

作为字典 

local config = {
    host = "127.0.0.1",
    port = 6379,
    timeout = 1000,
}

print(config["host"])  -- 127.0.0.1
print(config.port)     -- 6379（两种写法等价）

-- 遍历字典
for k, v in pairs(config) do
    print(k, v)
end

pairs 遍历 table 里所有的键值对，包括数组部分和字典部分，但顺序不固定。

同时作为两种

table 完全可以既有数组部分又有字典部分：

local user = {
    "alice",       -- [1]
    "bob",         -- [2]
    name = "alice",
    age = 28,
}

print(user[1])     -- alice
print(user.name)   -- alice
print(#user)       -- 2（# 只统计连续整数键的部分）

不过混用通常是坏习惯，代码可读性会很差。实际项目里建议只用其中一种形式。

操作 table 的常用方法

Lua 标准库提供了 table 模块，包含一些常用操作：

local arr = {10, 20, 30}

-- 追加元素
table.insert(arr, 40)           -- 末尾追加
table.insert(arr, 2, 15)        -- 在位置 2 插入

-- 删除元素
local removed = table.remove(arr, 1)  -- 删除位置 1 的元素并返回

-- 拼接数组为字符串
local str = table.concat(arr, ", ")   -- "15, 20, 30, 40"

-- 排序
table.sort(arr)
table.sort(arr, function(a, b) return a > b end)  -- 自定义比较

table 的内存共享陷阱

table 是引用类型。赋值时复制的是引用，不是内容。

local a = {1, 2, 3}
local b = a       -- b 和 a 指向同一个 table

b[1] = 99
print(a[1])       -- 99（a 也变了！）

如果需要独立的副本，要手动拷贝：

-- 浅拷贝
local function shallow_copy(t)
    local copy = {}
    for k, v in pairs(t) do
        copy[k] = v
    end
    return copy
end

-- 深拷贝（处理嵌套 table）
local function deep_copy(t)
    if type(t) ~= "table" then return t end
    local copy = {}
    for k, v in pairs(t) do
        copy[deep_copy(k)] = deep_copy(v)
    end
    return setmetatable(copy, getmetatable(t))
end

这个问题在 OpenResty 里特别容易踩——当你把一个配置 table 传给多个函数，其中某个函数悄悄修改了它，另一个函数读到的就是被污染的值。

元表：让 table 变得更聪明

元表（metatable）是 Lua 里最有意思的特性之一。通过给 table 设置元表，可以改变它的默认行为——比如两个 table 相加时发生什么，访问不存在的键时返回什么。

设置元表 

local t = {}
local mt = {}

setmetatable(t, mt)  -- 给 t 设置元表 mt
getmetatable(t)      -- 返回 mt

__index：访问不存在的键

这是最常用的元方法。当你访问一个 table 里不存在的键时，Lua 会检查是否有 __index 元方法：

local defaults = {color = "red", size = 10}

local obj = setmetatable({}, {
    __index = defaults
})

print(obj.color)   -- red（从 defaults 里找到的）
print(obj.size)    -- 10
print(obj.name)    -- nil（defaults 里也没有）

__index 也可以是一个函数：

local obj = setmetatable({}, {
    __index = function(t, key)
        return "default_" .. key
    end
})

print(obj.anything)  -- default_anything

__newindex：拦截赋值操作

当你给 table 中不存在的键赋值时，会触发 __newindex：

local readonly = setmetatable({}, {
    __newindex = function(t, key, value)
        error("this table is read-only!")
    end
})

readonly.x = 1  -- 触发 error

__tostring：自定义打印格式 

local point = setmetatable({x = 3, y = 4}, {
    __tostring = function(t)
        return string.format("Point(%d, %d)", t.x, t.y)
    end
})

print(tostring(point))  -- Point(3, 4)
print(point)            -- Point(3, 4)（print 内部调用 tostring）

__len：自定义 # 运算符 

local custom = setmetatable({}, {
    __len = function(t)
        return 42
    end
})

print(#custom)  -- 42

运算符重载

Lua 支持重载大部分运算符：

元方法对应操作
__add+
__sub-
__mul*
__div/
__mod%
__eq==
__lt<
__le<=
__concat..
__callobj() 当作函数调用

举个向量加法的例子：

local Vector = {}
Vector.__index = Vector

function Vector.new(x, y)
    return setmetatable({x = x, y = y}, Vector)
end

function Vector.__add(a, b)
    return Vector.new(a.x + b.x, a.y + b.y)
end

function Vector:__tostring()
    return string.format("Vector(%g, %g)", self.x, self.y)
end

local v1 = Vector.new(1, 2)
local v2 = Vector.new(3, 4)
local v3 = v1 + v2

print(tostring(v3))  -- Vector(4, 6)

用元表模拟面向对象

Lua 没有原生的 class 关键字，但借助元表的 __index，可以实现继承和方法调用——这就是 Lua 面向对象的核心技巧。

基本写法 

-- 定义"类"
local Animal = {}
Animal.__index = Animal

-- 构造函数
function Animal.new(name, sound)
    local self = setmetatable({}, Animal)
    self.name = name
    self.sound = sound
    return self
end

-- 方法
function Animal:speak()
    print(self.name .. " says " .. self.sound)
end

function Animal:getName()
    return self.name
end

-- 使用
local cat = Animal.new("Cat", "meow")
local dog = Animal.new("Dog", "woof")

cat:speak()   -- Cat says meow
dog:speak()   -- Dog says woof

这里有个关键点：Animal.__index = Animal 这一行是整个模式的核心。

当你访问 cat.speak 时，cat 这个 table 里并没有 speak 键，Lua 就查找它的元表，发现 __index = Animal，于是去 Animal 里找 speak，找到了！

继承 

local Dog = setmetatable({}, {__index = Animal})
Dog.__index = Dog

function Dog.new(name)
    local self = Animal.new(name, "woof")
    return setmetatable(self, Dog)
end

function Dog:fetch(item)
    print(self.name .. " fetches the " .. item)
end

local rex = Dog.new("Rex")
rex:speak()        -- Rex says woof（继承自 Animal）
rex:fetch("ball")  -- Rex fetches the ball

Dog 的元表 __index 指向 Animal，所以当 Dog 的实例找不到某个方法时，会继续向上查找 Animal——这就是原型链。

在 OpenResty 中的实际应用

OpenResty 的很多第三方库都用这种模式，比如 lua-resty-redis：

local redis = require "resty.redis"

local red = redis:new()  -- redis 就是个 table，new 是它的方法
red:set_timeout(1000)

local ok, err = red:connect("127.0.0.1", 6379)
-- red 的实例方法（get / set / del 等）都通过这种方式挂载

理解了元表，你就知道这些库为什么能这样调用，出问题时也更容易定位。

几个常见陷阱

陷阱一：# 的结果不可靠

local t = {1, nil, 3}
print(#t)  -- 可能是 1，也可能是 3，行为未定义

Lua 的 # 运算符要求数组部分是连续的整数键。中间有 nil 的话结果不可预期。如果需要准确长度，只能自己用 pairs 计数。

陷阱二：table 作为 key 时的比较

local t1 = {1, 2, 3}
local t2 = {1, 2, 3}
local map = {}

map[t1] = "first"
print(map[t2])  -- nil（t1 和 t2 是不同的引用）

table 作为 key 时，比较的是引用地址，不是内容。

陷阱三：全局 table 污染

function init()
    config = {}    -- 忘记写 local，config 变成全局变量
    config.host = "127.0.0.1"
end

在 OpenResty 里，全局变量在同一个 worker 的所有请求间共享，而且修改全局 table 不是线程安全的。永远记得加 local。

小结

table 是 Lua 的核心，也是 OpenResty 代码里最常见的数据结构。归纳一下本篇的要点：

  • table 既是数组又是字典，数组下标从 1 开始
  • table 是引用类型，赋值不会自动复制
  • 元表让 table 具备了自定义行为的能力，__index 是最重要的元方法
  • 通过 __index 可以实现继承，这是 Lua 面向对象的基础
  • # 对非连续数组不可靠；全局 table 要谨慎
