13 | 协程与非阻塞 I/O

发表于： 2026-03-18 分类于： OpenResty
字数： 2111 阅读：≈ 5分钟 浏览：

很多人用 OpenResty 一段时间后会有一个困惑：明明代码里写了连接 Redis、发起 HTTP 请求这些操作，但性能还是很高，没有感受到明显的阻塞。这是为什么？

答案就在这篇文章里：OpenResty 的协程模型和 cosocket 机制。理解这个模型，你才能真正知道 OpenResty 的高并发是怎么实现的，以及哪些操作会破坏这个模型。

OpenResty 的协程模型

先从 Nginx 说起。Nginx 用事件驱动模型处理连接：一个 worker 进程可以同时维护成千上万条连接，靠的是 epoll（Linux）这样的 I/O 多路复用机制。当某个连接的数据还没到时，worker 不等待，去处理其他连接，等内核通知"数据到了"再回来处理。

OpenResty 在这个基础上加了 LuaJIT 和协程。每个 HTTP 请求对应一个 Lua 协程。当协程需要做 I/O（比如查 Redis、发 HTTP 请求），它会：

  1. 发出 I/O 请求
  2. 把自己挂起，把控制权交还给 Nginx 的事件循环
  3. Nginx 继续处理其他请求
  4. I/O 完成后，Nginx 事件循环唤醒这个协程
  5. 协程从挂起点继续执行

从你的代码角度看，这些都是透明的——你写的是同步风格的代码，但底层实际上是异步非阻塞的。这就是 OpenResty 的核心魔法。

-- 看起来是同步代码，实际是非阻塞的
local red = redis:new()
red:connect("127.0.0.1", 6379)   -- 挂起，等连接建立
local val = red:get("key")        -- 挂起，等 Redis 返回
ngx.say(val)

cosocket：协程感知的 socket

cosocket 是 OpenResty 自己实现的网络 socket，名字里的 “co” 来自 coroutine（协程）。它的特别之处在于：当需要等待网络 I/O 时，它会自动挂起当前协程，不占用 CPU，等数据到了再自动恢复。

所有 lua-resty-* 库（resty.redis、resty.http 等）底层都用了 cosocket。

手动使用 cosocket

虽然大部分时候你用的是封装好的库，但了解底层接口有助于理解原理：

content_by_lua_block {
    local sock = ngx.socket.tcp()

    -- 设置超时（连接超时、读超时、发送超时，单位毫秒）
    sock:settimeouts(1000, 5000, 5000)

    -- 连接（这里会挂起协程，等待 TCP 握手完成）
    local ok, err = sock:connect("httpbin.org", 80)
    if not ok then
        ngx.log(ngx.ERR, "connect failed: ", err)
        return ngx.exit(500)
    end

    -- 发送请求
    local req = "GET /get HTTP/1.1\r\nHost: httpbin.org\r\nConnection: close\r\n\r\n"
    sock:send(req)

    -- 接收响应（挂起协程，等数据到达）
    local data, err = sock:receive("*a")

    sock:close()
    ngx.say(data)
}

cosocket 的限制

cosocket 只能在特定阶段使用：

  • rewrite_by_lua
  • access_by_lua
  • content_by_lua
  • ngx.timer.* 回调
  • ssl_certificate_by_lua

不能用的阶段：

  • init_by_lua（master 进程，没有事件循环）
  • init_worker_by_lua（有限制，部分操作可以）
  • header_filter_by_lua
  • body_filter_by_lua
  • log_by_lua

在这些禁止的阶段使用 cosocket 会报错：cosocket API disabled。

ngx.thread.spawn：轻线程

有时候需要并发执行多个任务——比如同时查 Redis 和调用外部 API，然后合并结果。ngx.thread.spawn 可以在当前请求内创建"轻线程"（light thread），也叫用户级线程。

content_by_lua_block {
    local function fetch_user(user_id)
        local red = redis:new()
        red:connect("127.0.0.1", 6379)
        local user_json = red:get("user:" .. user_id)
        return user_json
    end

    local function fetch_orders(user_id)
        -- 调用订单服务
        local httpc = require("resty.http").new()
        local res = httpc:request_uri("http://order-service/orders?user_id=" .. user_id)
        return res.body
    end

    local user_id = ngx.var.arg_id

    -- 并发启动两个轻线程
    local t1 = ngx.thread.spawn(fetch_user, user_id)
    local t2 = ngx.thread.spawn(fetch_orders, user_id)

    -- 等待两个线程都完成
    local ok1, user_data  = ngx.thread.wait(t1)
    local ok2, order_data = ngx.thread.wait(t2)

    local cjson = require "cjson"
    ngx.say(cjson.encode({
        user   = cjson.decode(user_data or "null"),
        orders = cjson.decode(order_data or "[]"),
    }))
}

ngx.thread.spawn 和 ngx.location.capture_multi 都能做并发，区别在于：

  • capture_multi：通过子请求，需要对应的 location 配置，更适合复用现有的 location 逻辑
  • thread.spawn：直接运行 Lua 函数，更灵活，不需要额外的 location

等待和终止轻线程 

-- 等待某个线程完成，获取返回值
local ok, result = ngx.thread.wait(t)

-- 等待第一个完成的线程（竞争模式）
local ok, result = ngx.thread.wait(t1, t2, t3)

-- 终止某个线程（如果它还在运行）
ngx.thread.kill(t)

什么操作会阻塞 worker

这是 OpenResty 开发里最重要的一条规则：不能在请求处理过程中调用任何阻塞操作。

一旦有阻塞，当前 worker 进程就被卡住了——它处理不了其他请求，直到阻塞操作完成。如果有 4 个 worker、所有 worker 都被阻塞，整个服务就卡死了。

常见的阻塞操作

文件 I/O（同步）：

-- 危险！io.open 是阻塞的
local f = io.open("/some/file", "r")
local content = f:read("*a")
f:close()

文件读写会阻塞内核，在文件读取期间 worker 无法响应其他请求。如果必须读文件，在 init_by_lua 阶段（启动时）读取并缓存到 ngx.shared.DICT。

标准库的网络操作：

-- 危险！luasocket 是阻塞的，不是 cosocket
local socket = require "socket"
local tcp = socket.tcp()
tcp:connect("redis", 6379)   -- 阻塞整个 worker！

所有网络操作必须通过 lua-resty-* 库（基于 cosocket），不能用 luasocket 或其他阻塞 socket 库。

os.execute / io.popen：

-- 危险！调用外部命令，可能阻塞很久
os.execute("curl https://example.com")
local handle = io.popen("ls /tmp")

ngx.sleep(0) 的误用：

-- 这是非阻塞的（协程 yield），OK
ngx.sleep(0)

-- 下面这个是阻塞的！
require("socket").sleep(1)

DNS 解析（同步）：

-- 危险！标准 DNS 查询是阻塞的
local ip = socket.dns.toip("redis.internal")

OpenResty 提供了 resty.dns.resolver 做非阻塞 DNS 解析，或者直接配置 resolver 指令让 nginx 负责解析。

怎么判断某个操作是否阻塞

一个简单的判断方法：如果这个操作底层依赖了 Linux 的 read/write/connect 等系统调用，并且不是通过 cosocket 发起的，那就是阻塞的。

实际开发中，坚守一条原则：所有 I/O 操作只用 lua-resty-* 系列库，不引入其他会做 I/O 的 Lua 库。

一个常见的陷阱：在 log 阶段用 cosocket 

log_by_lua_block {
    -- 想在日志阶段上报数据到 Kafka
    local producer = require "resty.kafka.producer"
    local p = producer:new(...)
    p:send("logs", nil, log_data)   -- 报错！log 阶段禁用 cosocket
}

log_by_lua 阶段不能用 cosocket。正确的做法是用定时器把上报逻辑推到后台：

log_by_lua_block {
    local log_data = {
        uri    = ngx.var.uri,
        status = ngx.status,
        time   = ngx.now(),
    }

    -- 用 ngx.timer.at 异步上报，0 秒后触发
    ngx.timer.at(0, function(premature, data)
        if premature then return end
        -- 这里可以用 cosocket
        local producer = require "resty.kafka.producer"
        -- ...
    end, log_data)
}

小结

OpenResty 的高并发来自协程 + cosocket 的组合：

  • 每个请求对应一个协程，I/O 等待时自动挂起，不占 CPU
  • cosocket 是协程感知的 socket，所有 lua-resty-* 库都基于它
  • ngx.thread.spawn 可以在一个请求内并发执行多个任务
  • 绝对不能在请求处理中调用阻塞操作（同步文件 I/O、luasocket、os.execute 等）
