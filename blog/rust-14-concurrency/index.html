发表于： 2026-04-26 分类于： Rust
字数： 4700 阅读：≈ 10分钟 浏览：

Go 有一句名言：“不要通过共享内存来通信，而要通过通信来共享内存。“这句话出现的原因，是因为共享内存并发太容易出错——数据竞争、死锁、忘记加锁，这些 bug 在 C/C++ 里困扰了程序员几十年。

Rust 的答案更激进：在编译期就消灭数据竞争。不是靠运行时检测，不是靠程序员自律，而是让编译器拒绝编译任何可能产生数据竞争的代码。

这一切的底气来自所有权系统——那两个神秘的 Trait：Send 和 Sync。

Send 和 Sync：并发安全的编译期保证

Send 和 Sync 是 Rust 标准库里最简单也最重要的两个 Trait，它们没有任何方法，纯粹是标记：

// 标记 Trait，没有任何方法
pub unsafe auto trait Send {}
pub unsafe auto trait Sync {}

Send：类型的所有权可以在线程间转移。如果 T: Send，就可以把 T 的值从一个线程移到另一个线程。

Sync：类型可以被多个线程同时引用。如果 T: Sync，就可以在多个线程里同时持有 &T（等价于 &T: Send）。

绝大多数类型都自动实现了这两个 Trait——i32、String、Vec<T>（当 T: Send）等等。编译器会自动推导：只要类型的所有字段都是 Send，这个类型就是 Send；字段都是 Sync，这个类型就是 Sync。

哪些类型不是 Send/Sync？

use std::rc::Rc;
use std::cell::RefCell;

// Rc<T>：不是 Send，也不是 Sync
// 原因：Rc 的引用计数用的是普通整数，不是原子操作，多线程访问会出问题
let rc = Rc::new(42);
// thread::spawn(move || println!("{}", rc)); // 编译错误！Rc 不是 Send

// RefCell<T>：是 Send（如果 T: Send），但不是 Sync
// 原因：RefCell 的运行时借用检查不是线程安全的
let cell = RefCell::new(42);
// 可以 move 到另一个线程，但不能同时被多个线程引用

对应的线程安全版本：Rc → Arc，RefCell → Mutex/RwLock。

这就是 Rust 并发安全的核心：编译器通过 Send/Sync 标记，在你试图跨线程传递不安全类型时直接报错，根本不给数据竞争留下发生的机会。

thread::spawn：创建线程

Rust 的线程 API 很直接：

use std::thread;
use std::time::Duration;

fn main() {
    let handle = thread::spawn(|| {
        for i in 1..=5 {
            println!("子线程: {}", i);
            thread::sleep(Duration::from_millis(50));
        }
    });

    for i in 1..=3 {
        println!("主线程: {}", i);
        thread::sleep(Duration::from_millis(80));
    }

    // 等待子线程结束，不调用 join 的话主线程结束时子线程会被强制终止
    handle.join().unwrap();
    println!("子线程结束");
}

thread::spawn 返回一个 JoinHandle<T>，其中 T 是闭包的返回值类型。调用 .join() 会阻塞当前线程直到目标线程结束，返回 Result<T, Box<dyn Any>>——Ok(T) 是线程的返回值，Err 是线程 panic 时的错误。

向线程传递数据

上一篇提到了，向线程传数据必须用 move 闭包：

use std::thread;

fn main() {
    let numbers = vec![1, 2, 3, 4, 5];

    let handle = thread::spawn(move || {
        let sum: i32 = numbers.iter().sum();
        println!("总和: {}", sum);
        sum // 闭包返回值
    });

    // numbers 已经被移走，这里不能再用
    let result = handle.join().unwrap();
    println!("线程返回值: {}", result);
}

同时启动多个线程 

use std::thread;

fn main() {
    let handles: Vec<_> = (0..5)
        .map(|i| {
            thread::spawn(move || {
                println!("线程 {} 启动", i);
                i * i // 返回平方值
            })
        })
        .collect();

    // 等所有线程结束，收集返回值
    let results: Vec<i32> = handles
        .into_iter()
        .map(|h| h.join().unwrap())
        .collect();

    println!("结果: {:?}", results); // [0, 1, 4, 9, 16]
}

Mutex：共享可变状态

当多个线程需要访问同一份数据时，需要互斥锁。Rust 的 Mutex<T> 把锁和数据绑定在一起——不拿锁就拿不到数据，从根本上避免了"忘记加锁"的问题。

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    // Arc：原子引用计数，让多个线程共享所有权
    // Mutex：互斥锁，保护内部数据
    let counter = Arc::new(Mutex::new(0));

    let handles: Vec<_> = (0..10)
        .map(|_| {
            let counter = Arc::clone(&counter); // 克隆 Arc，引用计数 +1
            thread::spawn(move || {
                let mut num = counter.lock().unwrap(); // 获取锁，返回 MutexGuard
                *num += 1;
                // MutexGuard 在这里 drop，锁自动释放
            })
        })
        .collect();

    for handle in handles {
        handle.join().unwrap();
    }

    println!("最终计数: {}", *counter.lock().unwrap()); // 10
}

几个关键点：

为什么要 Arc？ 单个 Mutex 只有一个所有者，但我们需要多个线程都能访问它。Arc（Atomic Reference Counting）用原子操作维护引用计数，让多个所有者安全地共享数据。Rc 不行，因为它不是 Send。

lock() 返回什么？ 返回 Result<MutexGuard<T>, PoisonError>。MutexGuard 是一个智能指针，实现了 Deref，可以直接操作内部数据；当 MutexGuard 被 drop 时，锁自动释放——不需要手动 unlock，RAII 机制保证了锁一定会被释放。

什么是 PoisonError？ 如果持有锁的线程 panic 了，锁会进入"中毒”（poisoned）状态，后续的 lock() 调用会返回 Err。.unwrap() 在这种情况下会 panic，很多场景下这是合理的——数据可能处于不一致状态，与其继续用，不如直接崩溃。

死锁：最常见的陷阱

Mutex 最大的风险是死锁。一个线程试图对已经持有的锁再次加锁，就会永远等待：

use std::sync::Mutex;

fn main() {
    let lock = Mutex::new(0);

    let _guard1 = lock.lock().unwrap(); // 获取锁
    // let _guard2 = lock.lock().unwrap(); // 死锁！同一线程再次请求同一把锁
}

更常见的是两把锁的循环等待：线程 A 持有锁 1 等锁 2，线程 B 持有锁 2 等锁 1。预防办法是保持一致的加锁顺序，或者尽量缩小持锁的范围。

锁的作用域越小越好——尽早 drop MutexGuard：

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));

    let data_clone = Arc::clone(&data);
    let handle = thread::spawn(move || {
        // 用花括号限定锁的作用域，操作完立即释放
        {
            let mut v = data_clone.lock().unwrap();
            v.push(4);
        } // MutexGuard 在这里 drop，锁释放

        // 锁已经释放，可以做其他事情
        println!("线程完成数据操作");
    });

    handle.join().unwrap();
    println!("{:?}", *data.lock().unwrap()); // [1, 2, 3, 4]
}

RwLock：读多写少的优化

Mutex 是排他锁：不管读还是写，同一时刻只有一个线程能访问数据。如果读操作远多于写操作，这就浪费了并发性能——多个线程同时读是完全安全的，没必要互相阻塞。

RwLock<T> 解决这个问题：允许多个读者同时持锁，但写者独占：

use std::sync::{Arc, RwLock};
use std::thread;

fn main() {
    let config = Arc::new(RwLock::new(String::from("initial")));

    // 启动多个读线程
    let readers: Vec<_> = (0..3)
        .map(|i| {
            let config = Arc::clone(&config);
            thread::spawn(move || {
                let value = config.read().unwrap(); // 获取读锁，多个读者可以同时持有
                println!("读线程 {}: {}", i, *value);
            })
        })
        .collect();

    // 写线程
    let config_write = Arc::clone(&config);
    let writer = thread::spawn(move || {
        let mut value = config_write.write().unwrap(); // 获取写锁，独占
        *value = String::from("updated");
        println!("写入完成");
    });

    for r in readers { r.join().unwrap(); }
    writer.join().unwrap();

    println!("最终值: {}", *config.read().unwrap());
}

什么时候用 RwLock 而不是 Mutex？

  • 读操作远多于写操作（比如配置、缓存）
  • 读操作耗时较长（否则锁的开销比收益还大）
  • 写操作不频繁——如果写很频繁，RwLock 可能因为写者饥饿问题反而更慢

实践中，先用 Mutex，有性能瓶颈再考虑换 RwLock。

Channel：通过通信共享数据

Go 的 goroutine + channel 模型非常流行。Rust 标准库也有 channel，语义类似，但类型系统更严格。

标准库提供的是 mpsc（multiple producer, single consumer）——多个发送端，一个接收端：

use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    // tx: 发送端（transmitter），rx: 接收端（receiver）
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        let messages = vec!["hello", "from", "thread"];
        for msg in messages {
            tx.send(msg).unwrap();
            thread::sleep(Duration::from_millis(100));
        }
        // tx 在这里 drop，channel 关闭
    });

    // recv() 阻塞等待，channel 关闭后返回 Err，for 循环自动退出
    for received in rx {
        println!("收到: {}", received);
    }

    println!("channel 已关闭，所有消息处理完毕");
}

多个发送端

mpsc 支持多个发送端，通过 clone 复制发送端：

use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    // 多个生产者线程
    let handles: Vec<_> = (0..3)
        .map(|i| {
            let tx = tx.clone(); // 克隆发送端
            thread::spawn(move || {
                tx.send(format!("来自线程 {}", i)).unwrap();
            })
        })
        .collect();

    // 必须 drop 原始 tx，否则 rx 不知道什么时候所有发送端都关闭了
    drop(tx);

    for msg in rx {
        println!("{}", msg);
    }

    for h in handles { h.join().unwrap(); }
}

注意那个 drop(tx)。原始 tx 没有被 move 到任何线程里，如果不手动 drop，rx 的 for 循环会一直等待，永远不结束——因为还有一个活着的发送端没有关闭。

同步 channel：背压控制

标准库的 channel 是无界的——发送端可以无限发送，不管接收端处不处理得过来。这在生产者远快于消费者时会导致内存暴增。

sync_channel 提供有界 channel，发送端在队列满时会阻塞：

use std::sync::mpsc;
use std::thread;

fn main() {
    // 缓冲区大小为 2，队列满时 send 会阻塞
    let (tx, rx) = mpsc::sync_channel(2);

    let handle = thread::spawn(move || {
        for i in 0..5 {
            println!("发送 {}", i);
            tx.send(i).unwrap(); // 队列满时会等待
        }
    });

    // 慢速消费
    for received in rx {
        thread::sleep(std::time::Duration::from_millis(100));
        println!("处理 {}", received);
    }

    handle.join().unwrap();
}

共享状态 vs 消息传递：怎么选？

这是并发设计中最常见的问题，没有绝对答案，但有一些经验法则：

优先用消息传递（channel）：

  • 任务天然可以分解为独立的工作单元
  • 生产者和消费者职责清晰，不需要互相感知
  • 数据所有权需要跨线程转移（而不是共享）
  • 典型场景：工作队列、流水线、事件驱动

用共享状态（Mutex/RwLock）：

  • 多个线程需要访问和修改同一份数据
  • 数据需要长期存在，不适合在消息里传递
  • 需要随时读取最新状态（而不是等消息到达）
  • 典型场景：计数器、缓存、配置中心、连接池

实际项目中两者经常混用：用 channel 分发任务，用 Arc<Mutex<T>> 存储共享状态。

一个典型的模式：

use std::sync::{Arc, Mutex};
use std::sync::mpsc;
use std::thread;

// 共享的统计数据
#[derive(Default)]
struct Stats {
    processed: usize,
    errors: usize,
}

fn main() {
    let stats = Arc::new(Mutex::new(Stats::default()));
    let (tx, rx) = mpsc::channel::<String>();

    // 工作线程：从 channel 取任务，更新共享统计
    let stats_clone = Arc::clone(&stats);
    let worker = thread::spawn(move || {
        for task in rx {
            // 处理任务...
            let success = !task.contains("error");

            let mut s = stats_clone.lock().unwrap();
            if success {
                s.processed += 1;
            } else {
                s.errors += 1;
            }
        }
    });

    // 发送任务
    tx.send("task-1".to_string()).unwrap();
    tx.send("task-error".to_string()).unwrap();
    tx.send("task-2".to_string()).unwrap();
    drop(tx); // 关闭 channel，工作线程会退出循环

    worker.join().unwrap();

    let s = stats.lock().unwrap();
    println!("处理成功: {}, 错误: {}", s.processed, s.errors);
}

线程本地存储

有时候每个线程需要自己独立的数据副本，不需要共享。thread_local! 宏提供线程本地存储：

use std::cell::RefCell;
use std::thread;

thread_local! {
    // 每个线程有自己独立的 counter，互不干扰
    static COUNTER: RefCell<u32> = RefCell::new(0);
}

fn increment() -> u32 {
    COUNTER.with(|c| {
        *c.borrow_mut() += 1;
        *c.borrow()
    })
}

fn main() {
    let handles: Vec<_> = (0..3)
        .map(|_| {
            thread::spawn(|| {
                println!("{}", increment()); // 1
                println!("{}", increment()); // 2
                // 每个线程从 0 开始，互不影响
            })
        })
        .collect();

    for h in handles { h.join().unwrap(); }

    // 主线程的 counter 也是独立的
    println!("主线程: {}", increment()); // 1
}

线程本地存储用 RefCell 而不是 Mutex——因为根本就没有共享，不需要锁，访问速度极快。适合用来存储线程级别的上下文，比如请求 ID、日志上下文、数据库连接等。

原子操作：无锁并发的基础

对于简单的数字计数，用 Mutex<u64> 开销有点大。标准库提供了原子类型，利用 CPU 的原子指令实现无锁并发：

use std::sync::Arc;
use std::sync::atomic::{AtomicUsize, Ordering};
use std::thread;

fn main() {
    let counter = Arc::new(AtomicUsize::new(0));

    let handles: Vec<_> = (0..10)
        .map(|_| {
            let counter = Arc::clone(&counter);
            thread::spawn(move || {
                // fetch_add 是原子操作：读取、加法、写入一气呵成
                counter.fetch_add(1, Ordering::SeqCst);
            })
        })
        .collect();

    for h in handles { h.join().unwrap(); }
    println!("最终计数: {}", counter.load(Ordering::SeqCst)); // 10
}

Ordering 参数控制内存顺序语义，从宽松到严格：

  • Relaxed：只保证原子性，不保证顺序——适合纯计数，性能最好
  • Acquire/Release：配对使用，建立 happens-before 关系——适合构建锁
  • SeqCst：全序一致，最严格——适合不确定时的安全选择，性能最差

实践建议：计数器用 Relaxed，其他情况用 SeqCst 直到你真的需要优化。内存顺序的坑很深，不确定时选最严格的。

标准库提供了 AtomicBool、AtomicI8/AtomicU8 等一系列原子类型，但没有原子的 String 或复合类型——复杂类型还是要用 Mutex。

小结

  • Send/Sync：编译期的并发安全保证；Send 允许所有权跨线程转移，Sync 允许引用跨线程共享；不满足约束的代码直接编译报错
  • thread::spawn：创建线程，必须用 move 闭包转移所有权；JoinHandle::join 等待线程结束并取回返回值
  • Arc<Mutex<T>>：多线程共享可变数据的标准组合；Arc 共享所有权，Mutex 保护数据，RAII 确保锁自动释放
  • RwLock<T>：读多写少时的优化选择，允许多个读者并发
  • mpsc::channel：多生产者单消费者 channel；sync_channel 有背压，无界 channel 无背压
  • 线程本地存储：每个线程独立数据，无需锁，适合线程级上下文
  • 原子类型：无锁的简单计数和标志位，比 Mutex 轻量；Ordering 控制内存顺序
  • 选择原则：任务分解用 channel，共享状态用 Mutex；实践中两者混用
