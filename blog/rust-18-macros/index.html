18 | 宏系统精讲

发表于： 2026-04-28 分类于： Rust
字数： 3589 阅读：≈ 8分钟 浏览：

前几篇我们一直在用宏，却没有正面解释它：println!、vec!、tokio::select!、#[derive(Debug)]……这些都是宏。这篇我们正面拆解宏系统——它是什么，怎么写，什么时候该用。

Rust 的宏和 C 的 #define 完全不同。C 的宏是文本替换，编译器看不见，调试地狱。Rust 的宏在语法树层面操作，编译器全程参与，类型安全，可调试。

为什么需要宏

先说宏解决的是什么问题。

有些代码模式用函数无法优雅表达：

  • 可变参数：println! 接受任意数量的参数，普通函数做不到
  • 语法扩展：vec![1, 2, 3] 创建 Vec，这是一种更自然的语法，函数调用做不到这个字面量感觉
  • 代码生成：#[derive(Serialize)] 自动实现序列化，手写要几十行；宏帮你生成

本质上，宏是在编译期生成代码的工具。

声明宏：macro_rules!

声明宏是最常用的宏形式，用模式匹配的方式定义"输入长什么样，展开成什么代码"。

最简单的宏 

macro_rules! say_hello {
    () => {
        println!("Hello, Rust!");
    };
}

fn main() {
    say_hello!(); // 展开为 println!("Hello, Rust!");
}

macro_rules! 后面是宏的名字，花括号里是一组规则，每条规则是 (模式) => { 展开内容 }。

带参数的宏 

macro_rules! greet {
    ($name:expr) => {
        println!("Hello, {}!", $name);
    };
}

fn main() {
    greet!("World");
    greet!(42);
}

$name:expr 是模式片段，$name 是捕获变量名，expr 是片段类型（fragment specifier），表示"匹配任意表达式"。

常用片段类型

片段类型匹配内容示例
expr表达式1 + 2、"hello"、foo()
ident标识符变量名、函数名
ty类型i32、Vec<String>
pat模式Some(x)、(a, b)
stmt语句let x = 1;
block代码块{ ... }
item条目函数、结构体定义
literal字面量42、"hello"
ttToken 树（万能匹配）任何 token

多规则匹配

宏可以有多条规则，类似函数重载：

macro_rules! log {
    // 只有消息
    ($msg:expr) => {
        println!("[LOG] {}", $msg);
    };
    // 有级别和消息
    ($level:ident, $msg:expr) => {
        println!("[{}] {}", stringify!($level), $msg);
    };
    // 有级别、消息和额外参数（格式化字符串）
    ($level:ident, $fmt:expr, $($arg:expr),*) => {
        println!(concat!("[{}] ", $fmt), stringify!($level), $($arg),*);
    };
}

fn main() {
    log!("系统启动");
    log!(INFO, "用户登录");
    log!(ERROR, "请求失败，错误码: {}", 404);
}

重复匹配：$(...)* 和 $(...)+

这是声明宏最强大的特性——匹配任意数量的重复内容，这也是 vec! 的实现原理：

// vec! 的简化实现
macro_rules! my_vec {
    // 空 vec
    () => {
        Vec::new()
    };
    // 一个或多个元素，用逗号分隔
    ($($elem:expr),+ $(,)?) => {
        {
            let mut v = Vec::new();
            $(v.push($elem);)+  // 对每个 elem 重复展开
            v
        }
    };
}

fn main() {
    let v1: Vec<i32> = my_vec![];
    let v2 = my_vec![1, 2, 3];
    let v3 = my_vec![1, 2, 3,]; // 尾部逗号也支持，$(,)? 的作用
    println!("{:?} {:?} {:?}", v1, v2, v3);
}

$($elem:expr),+ 的含义：

  • $(...) 表示重复分组
  • $elem:expr 是分组内的捕获
  • , 是分隔符
  • + 表示一次或多次（* 是零次或多次）

展开时，$($expr)* 中的每个 $expr 都会被替换成对应的捕获内容，有多少个捕获就展开多少次。

一个实用示例：测试辅助宏 

macro_rules! assert_approx_eq {
    ($left:expr, $right:expr, $epsilon:expr) => {
        let diff = ($left - $right).abs();
        if diff > $epsilon {
            panic!(
                "断言失败: {} ≈ {} 不成立（差值 {} 超过容差 {}）",
                $left, $right, diff, $epsilon
            );
        }
    };
    // 默认容差
    ($left:expr, $right:expr) => {
        assert_approx_eq!($left, $right, 1e-6);
    };
}

fn main() {
    assert_approx_eq!(0.1 + 0.2, 0.3, 1e-10); // 会 panic，因为浮点误差
    assert_approx_eq!(0.1_f64 + 0.2_f64, 0.3_f64, 1e-9); // 通过
}

过程宏：在语法树层面操作

声明宏是"模式替换"，过程宏是真正的代码生成——它接收 Rust 语法树作为输入，输出新的语法树。功能更强大，但写起来也更复杂。

过程宏分三种：

  • Derive 宏：#[derive(MyTrait)]，为结构体/枚举自动实现 trait
  • 属性宏：#[my_attribute]，修改或替换被标注的条目
  • 函数式宏：my_macro!(...)，和声明宏调用方式相同，但实现方式不同

过程宏必须放在独立的 crate 里（proc-macro = true），这是编译器的要求。

Derive 宏：最常用的过程宏

你已经用过很多 derive 宏：

#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
struct User {
    name: String,
    age: u32,
}

每个 derive 背后都是一个过程宏。Debug 是标准库提供的，Serialize/Deserialize 是 serde 提供的。

让我们写一个简单的自定义 derive 宏，理解它的工作原理。创建一个新 crate：

# Cargo.toml（过程宏 crate）
[lib]
proc-macro = true

[dependencies]
syn = { version = "2", features = ["full"] }
quote = "1"
proc-macro2 = "1"

// src/lib.rs（过程宏 crate）
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, DeriveInput};

#[proc_macro_derive(HelloMacro)]
pub fn hello_macro_derive(input: TokenStream) -> TokenStream {
    // 1. 把输入 token 流解析成语法树
    let ast = parse_macro_input!(input as DeriveInput);
    let name = &ast.ident; // 结构体名字

    // 2. 用 quote! 生成新代码
    let gen = quote! {
        impl HelloMacro for #name {
            fn hello() {
                println!("Hello from {}!", stringify!(#name));
            }
        }
    };

    // 3. 转换回 token 流返回
    gen.into()
}

使用方：

// 在另一个 crate 中使用
use hello_macro::HelloMacro;

trait HelloMacro {
    fn hello();
}

#[derive(HelloMacro)]
struct Pancakes;

#[derive(HelloMacro)]
struct Waffles;

fn main() {
    Pancakes::hello(); // 输出: Hello from Pancakes!
    Waffles::hello();  // 输出: Hello from Waffles!
}

宏帮你为每个类型生成了 impl HelloMacro 的实现，不需要手写。

serde 的 Derive 宏：最值得深入的案例

serde 是 Rust 生态里用得最广的序列化框架，它的 #[derive(Serialize, Deserialize)] 几乎是每个项目的标配。理解它背后发生了什么，能帮你更好地使用它，也能在出错时快速定位问题。

展开后的代码长什么样 

use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize, Debug)]
struct User {
    name: String,
    age: u32,
    #[serde(skip_serializing_if = "Option::is_none")]
    email: Option<String>,
}

#[derive(Serialize)] 展开后，大致等价于：

impl serde::Serialize for User {
    fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error>
    where
        S: serde::Serializer,
    {
        use serde::ser::SerializeStruct;
        let mut state = serializer.serialize_struct("User", 3)?;
        state.serialize_field("name", &self.name)?;
        state.serialize_field("age", &self.age)?;
        // skip_serializing_if 属性影响了这里的逻辑
        if !Option::is_none(&self.email) {
            state.serialize_field("email", &self.email)?;
        }
        state.end()
    }
}

宏读取了 #[serde(skip_serializing_if = "Option::is_none")] 这个属性，把它翻译成了 if 判断。这就是属性宏的强大之处——不只是生成代码，还能根据标注修改生成逻辑。

serde 常用属性 

#[derive(Serialize, Deserialize)]
#[serde(rename_all = "camelCase")] // 字段名转 camelCase
struct ApiResponse {
    user_name: String,    // 序列化为 "userName"
    created_at: u64,      // 序列化为 "createdAt"

    #[serde(rename = "id")] // 单独重命名
    user_id: u32,

    #[serde(skip)]          // 完全跳过（序列化和反序列化都忽略）
    internal_flag: bool,

    #[serde(default)]       // 反序列化时如果字段缺失，使用 Default::default()
    optional_count: u32,

    #[serde(flatten)]       // 把嵌套结构体的字段提升到同一层
    metadata: Metadata,
}

#[derive(Serialize, Deserialize)]
struct Metadata {
    version: String,
    source: String,
}

#[serde(flatten)] 的效果：

// 没有 flatten：
{ "user_name": "Alice", "metadata": { "version": "1.0", "source": "api" } }

// 有 flatten：
{ "user_name": "Alice", "version": "1.0", "source": "api" }

自定义序列化

当默认的 derive 不够用时，可以手动实现，或用 serialize_with：

use serde::{Serialize, Deserialize, Serializer, Deserializer};
use chrono::{DateTime, Utc};

#[derive(Serialize, Deserialize)]
struct Event {
    name: String,

    // 用自定义函数控制时间的序列化格式
    #[serde(
        serialize_with = "serialize_datetime",
        deserialize_with = "deserialize_datetime"
    )]
    created_at: DateTime<Utc>,
}

fn serialize_datetime<S>(dt: &DateTime<Utc>, s: S) -> Result<S::Ok, S::Error>
where
    S: Serializer,
{
    s.serialize_str(&dt.format("%Y-%m-%d %H:%M:%S").to_string())
}

fn deserialize_datetime<'de, D>(d: D) -> Result<DateTime<Utc>, D::Error>
where
    D: Deserializer<'de>,
{
    use serde::de::Error;
    let s = String::deserialize(d)?;
    DateTime::parse_from_str(&s, "%Y-%m-%d %H:%M:%S")
        .map(|dt| dt.with_timezone(&Utc))
        .map_err(Error::custom)
}

属性宏和函数式过程宏

属性宏

属性宏可以完全替换被标注的条目。axum 里没有这类宏，但 actix-web 的路由就是典型例子：

// actix-web 风格（属性宏）
#[get("/users/{id}")]
async fn get_user(path: web::Path<u32>) -> impl Responder {
    format!("User {}", path.into_inner())
}

#[get("/users/{id}")] 这个宏接收整个函数，给它包装上路由注册逻辑，返回修改后的代码。

实现属性宏的签名：

#[proc_macro_attribute]
pub fn my_attribute(attr: TokenStream, item: TokenStream) -> TokenStream {
    // attr: 属性参数，即 #[my_attribute(这里的内容)]
    // item: 被标注的条目（函数、结构体等）
    // 返回替换后的代码
    item // 最简单的情况：原样返回，不做修改
}

函数式过程宏

和声明宏调用方式相同（my_macro!(...)），但用过程宏实现，可以做更复杂的解析。tokio::select! 就是这种方式实现的。

#[proc_macro]
pub fn sql(input: TokenStream) -> TokenStream {
    // 可以在这里解析 SQL 语法，做编译期检查
    // 这就是 sqlx 的 query! 宏的思路
    todo!()
}

// 使用
let result = sql!(SELECT * FROM users WHERE id = $1);

sqlx 的 query! 宏会在编译期连接数据库，验证 SQL 语句是否合法，并自动推导返回类型——这在其他语言里几乎不可能做到。

调试宏展开

宏出错时，报错信息往往指向展开后的代码，看起来非常困惑。有几个工具可以帮助调试：

cargo expand 

# 安装
cargo install cargo-expand

# 查看展开结果
cargo expand

# 只展开特定模块
cargo expand my_module

举例，对这个代码：

#[derive(Debug, Clone)]
struct Point {
    x: f64,
    y: f64,
}

cargo expand 输出大致为：

struct Point {
    x: f64,
    y: f64,
}
impl std::fmt::Debug for Point {
    fn fmt(&self, f: &mut std::fmt::Formatter) -> std::fmt::Result {
        f.debug_struct("Point")
            .field("x", &self.x)
            .field("y", &self.y)
            .finish()
    }
}
impl Clone for Point {
    fn clone(&self) -> Self {
        Point { x: self.x.clone(), y: self.y.clone() }
    }
}

看到展开后的代码，宏生成了什么一目了然。

dbg! 宏本身就是调试工具

dbg! 是标准库提供的调试宏，打印表达式的值和位置，并返回该值，可以嵌入到表达式中间：

fn main() {
    let a = 2;
    let b = dbg!(a * 2) + 1; // 打印: [src/main.rs:3] a * 2 = 4
    dbg!(b);                  // 打印: [src/main.rs:4] b = 5
}

比 println! 方便的地方：不需要手写变量名，位置信息自动包含，且不影响表达式的值。

什么时候该写宏

宏是强大的工具，但不是所有问题都需要宏来解决。

该用宏的场景：

  1. 可变参数：函数无法接受不确定数量的参数，宏可以
  2. 代码生成：为大量相似类型重复实现 trait，手写太枯燥，出错概率高
  3. DSL（领域特定语言）：像 tokio::select!、sqlx::query!，提供更自然的语法
  4. 编译期计算：利用宏在编译期完成一些工作（验证、生成查找表等）

不该用宏的场景：

  1. 函数能解决的问题：别为了"酷"去写宏，泛型和 trait 往往更合适
  2. 逻辑复杂的封装：宏调试困难，错误信息不友好；如果逻辑复杂，用函数更清晰
  3. 团队不熟悉宏：宏增加了代码的认知负担，要评估收益是否值得

一个简单的判断：如果你可以用函数、trait 或泛型解决，就不要用宏。宏应该是最后的手段，而不是第一反应。

小结

  • 声明宏（macro_rules!）：模式匹配式的代码生成，适合简单的代码复用和可变参数；$(...)* / $(...)+ 处理重复是核心特性
  • 过程宏：在语法树层面操作，功能更强大；分 derive 宏、属性宏、函数式宏三种形式；需要 syn + quote + proc-macro2 工具链
  • serde derive：最值得深入的宏案例；#[serde(...)] 属性控制生成逻辑；cargo expand 可以看到展开后的完整代码
  • 调试工具：cargo expand 查看宏展开结果；dbg! 宏嵌入表达式调试，比 println! 方便
  • 选用原则：函数/trait/泛型能解决的问题不用宏；宏适合可变参数、批量代码生成、DSL 场景
