03 | OpenResty 执行生命周期

发表于： 2026-03-06 分类于： OpenResty
字数： 4701 阅读：≈ 10分钟 浏览：

上一篇我们搭好了环境，跑通了第一个 Hello World。但当时埋了一个伏笔——content_by_lua_block 里的 content 到底是什么意思？为什么不叫 run_lua_block 或者 exec_lua_block？

这篇文章就来解答这个问题。搞懂 OpenResty 的执行生命周期，是真正理解 OpenResty 编程模型的关键一步。

从一个比喻开始

把处理一个 HTTP 请求想象成在餐厅点餐的全过程。

顾客（客户端）走进餐厅，服务员（Nginx）接待。整个流程大致是：门口迎宾检查着装 → 带位 → 点单 → 后厨做菜 → 上菜 → 结账离开 → 收拾桌子写日志。

每个环节都有自己的职责。迎宾不负责炒菜，后厨不负责结账，收银员不管收拾桌子。如果你在"带位"阶段突然要求"炒一道菜给我"，服务员会一脸懵——这不是你该在这个阶段提的要求。

Nginx 处理请求也是这个逻辑，它把一次请求的生命周期切成了若干个阶段，每个阶段有明确的职责，不能随意跨阶段操作。

Nginx 请求处理的 11 个阶段

Nginx 内部将 HTTP 请求处理划分为以下 11 个阶段：

POST_READ       → 读取请求头之后，最早能介入的阶段
SERVER_REWRITE  → server 块级别的 rewrite
FIND_CONFIG     → 查找匹配的 location 配置
REWRITE         → location 块内的 rewrite
POST_REWRITE    → rewrite 完成后的收尾
PREACCESS       → access 之前的限流、限速等
ACCESS          → 权限检查（认证、鉴权）
POST_ACCESS     → access 结果处理
PRECONTENT      → 内容生成前的准备（try_files 在这里）
CONTENT         → 生成响应内容（核心阶段）
LOG             → 请求结束后记录日志

OpenResty 在这 11 个阶段的基础上，提供了一系列 *_by_lua* 指令，让你可以在对应阶段嵌入 Lua 代码。常用的对应关系如下：

init_by_lua_block         → worker 启动时执行（全局初始化）
init_worker_by_lua_block  → 每个 worker 进程启动时执行
rewrite_by_lua_block      → REWRITE 阶段
access_by_lua_block       → ACCESS 阶段
content_by_lua_block      → CONTENT 阶段
header_filter_by_lua_block → 响应头过滤
body_filter_by_lua_block  → 响应体过滤
log_by_lua_block          → LOG 阶段

接下来逐个讲解最常用的几个阶段，并配上实际可以跑的代码示例。

init_by_lua_block：在服务启动时做准备

这个阶段在 Nginx master 进程启动时执行，且只执行一次。所有 worker 进程 fork 出来之前，这段代码就已经跑完了。

适合做什么：

  • 预加载 Lua 模块（缓存到全局，避免每次请求重复 require）
  • 初始化全局配置、常量
  • 预编译正则表达式

不能做什么：

  • 不能访问 ngx.req、ngx.resp 这类请求相关的 API，因为此时根本没有请求
  • 不能操作连接（数据库、Redis），worker 进程还没起来，连接池没有意义

示例：预加载模块和初始化全局配置

http {
    init_by_lua_block {
        -- 预加载常用模块，避免每次请求都 require
        cjson   = require "cjson"
        resty_redis = require "resty.redis"

        -- 初始化全局配置
        REDIS_HOST = "127.0.0.1"
        REDIS_PORT = 6379
        REDIS_TIMEOUT = 1000  -- 毫秒

        ngx.log(ngx.INFO, "init_by_lua_block executed, global config loaded")
    }

    server {
        listen 80;

        location /test {
            content_by_lua_block {
                -- 直接使用前面初始化的全局变量
                local info = {
                    redis_host = REDIS_HOST,
                    redis_port = REDIS_PORT
                }
                ngx.say(cjson.encode(info))
            }
        }
    }
}

注意全局变量的使用要谨慎，滥用全局变量会导致内存泄漏和逻辑混乱。通常做法是把配置存在 Lua table 里，而不是到处散落的裸变量。

init_worker_by_lua_block：每个 worker 的私有初始化

Master 进程 fork 出 worker 之后，每个 worker 启动时会执行这段代码。

适合做什么：

  • 启动定时任务（ngx.timer.at 或 ngx.timer.every）
  • 每个 worker 各自维护自己的本地状态

示例：在 worker 启动时创建一个周期性心跳日志

init_worker_by_lua_block {
    local function heartbeat(premature)
        if premature then
            -- premature=true 表示 worker 正在关闭，定时器提前触发
            return
        end

        ngx.log(ngx.INFO, "worker ", ngx.worker.id(), " is alive at ", ngx.now())

        -- 创建下一次定时器（循环调用）
        local ok, err = ngx.timer.at(60, heartbeat)
        if not ok then
            ngx.log(ngx.ERR, "failed to create timer: ", err)
        end
    end

    -- 启动第一次定时器，60 秒后触发
    local ok, err = ngx.timer.at(60, heartbeat)
    if not ok then
        ngx.log(ngx.ERR, "failed to create initial timer: ", err)
    end
}

rewrite_by_lua_block：URL 改写与重定向

这个阶段对应 Nginx 的 REWRITE 阶段，在 access 之前执行。主要职责是对请求进行变形——改写 URI、修改参数、做重定向。

适合做什么：

  • 改写请求的 URI 或 query string
  • 根据条件做 301/302 重定向
  • 设置变量供后续阶段使用

注意事项：

  • 此阶段可以访问请求头和 URI，但响应体还不存在
  • 不要在这里生成响应内容，那是 CONTENT 阶段的事

示例：根据 User-Agent 判断是否为移动端，并重定向到移动版页面

location / {
    rewrite_by_lua_block {
        local ua = ngx.req.get_headers()["User-Agent"] or ""

        -- 简单判断是否是移动端
        if ua:find("Mobile") or ua:find("Android") or ua:find("iPhone") then
            -- 如果已经是 /m/ 开头，不再重定向，避免死循环
            if not ngx.var.uri:find("^/m/") then
                return ngx.redirect("/m" .. ngx.var.uri, 302)
            end
        end
    }

    # 非移动端，走正常流程
    root /var/www/html;
    index index.html;
}

示例二：统一清理 URL 末尾的斜杠

location / {
    rewrite_by_lua_block {
        local uri = ngx.var.uri

        -- 去掉末尾多余的斜杠（保留根路径 /）
        if uri ~= "/" and uri:sub(-1) == "/" then
            local clean_uri = uri:gsub("/+$", "")
            local args = ngx.var.args
            if args and args ~= "" then
                return ngx.redirect(clean_uri .. "?" .. args, 301)
            else
                return ngx.redirect(clean_uri, 301)
            end
        end
    }

    proxy_pass http://backend;
}

access_by_lua_block：请求的门卫

这是请求能否继续往下走的关卡。主要用于鉴权——检查 token、验证签名、判断 IP 白名单等。如果检查不通过，直接在这里返回 401、403，请求就不会进入后续的 CONTENT 阶段。

适合做什么：

  • JWT token 验证
  • API 签名校验
  • IP 黑白名单
  • 请求频率限制（配合 shared dict）

示例：简单的 Token 验证

location /api/ {
    access_by_lua_block {
        local token = ngx.req.get_headers()["X-Auth-Token"]

        if not token then
            ngx.status = 401
            ngx.header["Content-Type"] = "application/json"
            ngx.say('{"error":"missing token"}')
            return ngx.exit(401)
        end

        -- 实际项目里这里会去 Redis 或数据库验证 token
        -- 这里简化成硬编码对比
        if token ~= "my-secret-token" then
            ngx.status = 403
            ngx.header["Content-Type"] = "application/json"
            ngx.say('{"error":"invalid token"}')
            return ngx.exit(403)
        end

        -- 验证通过，不做任何输出，让请求继续流转到 content 阶段
    }

    content_by_lua_block {
        ngx.say('{"data":"this is protected content"}')
    }
}

示例二：基于 shared dict 的 IP 限流

http {
    # 声明一块共享内存，所有 worker 共享
    lua_shared_dict ip_limit 10m;

    server {
        location /api/ {
            access_by_lua_block {
                local limit_store = ngx.shared.ip_limit
                local client_ip   = ngx.var.remote_addr
                local max_req     = 100    -- 每分钟最多 100 次
                local window      = 60     -- 时间窗口 60 秒

                local count, err = limit_store:incr(client_ip, 1, 0, window)
                if err then
                    ngx.log(ngx.ERR, "shared dict error: ", err)
                    -- 出错时放行，避免误伤
                    return
                end

                if count > max_req then
                    ngx.status = 429
                    ngx.header["Content-Type"] = "application/json"
                    ngx.header["Retry-After"]  = tostring(window)
                    ngx.say('{"error":"rate limit exceeded"}')
                    return ngx.exit(429)
                end
            }

            content_by_lua_block {
                ngx.say("request processed")
            }
        }
    }
}

content_by_lua_block：生成响应的主舞台

这是最核心的阶段，上一篇写的 Hello World 就是在这里跑的。一个 location 里只能有一个内容生成指令，content_by_lua_block、proxy_pass、root 这三者互斥，同时写只有最后一个生效（实际上会出错或行为不确定）。

适合做什么：

  • 生成 JSON 响应
  • 查询数据库、Redis，组装数据返回
  • 实现 REST API 逻辑
  • 反向代理（配合 ngx.location.capture）

示例：完整的查询 Redis 并返回 JSON 的接口

location /user {
    content_by_lua_block {
        local redis = require "resty.redis"
        local cjson = require "cjson"

        -- 获取 query 参数
        local args, err = ngx.req.get_uri_args()
        local user_id = args["id"]

        if not user_id then
            ngx.status = 400
            ngx.header["Content-Type"] = "application/json"
            ngx.say(cjson.encode({ error = "missing id parameter" }))
            return
        end

        -- 连接 Redis
        local red = redis:new()
        red:set_timeout(1000)

        local ok, err = red:connect("127.0.0.1", 6379)
        if not ok then
            ngx.status = 500
            ngx.header["Content-Type"] = "application/json"
            ngx.say(cjson.encode({ error = "redis connect failed: " .. err }))
            return
        end

        -- 查询用户信息
        local user_json, err = red:get("user:" .. user_id)
        if not user_json or user_json == ngx.null then
            ngx.status = 404
            ngx.header["Content-Type"] = "application/json"
            ngx.say(cjson.encode({ error = "user not found" }))
            -- 归还连接到连接池
            red:set_keepalive(10000, 100)
            return
        end

        -- 归还连接到连接池（复用，性能关键）
        red:set_keepalive(10000, 100)

        ngx.header["Content-Type"] = "application/json"
        ngx.say(user_json)
    }
}

这个示例展示了 content 阶段的典型写法：参数校验、连接资源、查询数据、连接归池、返回结果。每一步出错都要及时 return，不要让代码继续往下跑。

log_by_lua_block：请求结束后的尾声

响应已经发给客户端了，这个阶段在请求完全结束后执行。适合做一些"事后"的事情：记录自定义日志、上报监控指标、异步触发一些非关键操作。

关键特点：

  • 此阶段不能修改响应头或响应体，那些已经发出去了
  • 可以读取响应状态码（ngx.status）和请求信息
  • 适合做异步操作，不会影响响应时间（因为响应已经回去了）

示例：记录慢请求日志

location /api/ {
    content_by_lua_block {
        -- 模拟业务逻辑
        ngx.sleep(0.1)  -- 100ms
        ngx.say("ok")
    }

    log_by_lua_block {
        local latency = tonumber(ngx.var.request_time) or 0

        -- 超过 200ms 的请求记录慢日志
        if latency > 0.2 then
            ngx.log(ngx.WARN, string.format(
                "[slow_request] method=%s uri=%s status=%d latency=%.3fs client=%s",
                ngx.var.request_method,
                ngx.var.uri,
                ngx.status,
                latency,
                ngx.var.remote_addr
            ))
        end
    }
}

示例二：异步上报请求统计到 Redis

log_by_lua_block {
    local function report_stats(premature, uri, status)
        if premature then return end

        local redis = require "resty.redis"
        local red = redis:new()
        red:set_timeout(500)

        local ok, err = red:connect("127.0.0.1", 6379)
        if not ok then
            ngx.log(ngx.ERR, "stats redis connect failed: ", err)
            return
        end

        -- 按 URI 和状态码统计请求数
        local key = string.format("stats:%s:%d", uri, status)
        red:incr(key)
        red:expire(key, 86400)  -- 保留 24 小时
        red:set_keepalive(10000, 100)
    end

    -- 用 timer 异步执行，不阻塞日志阶段
    local ok, err = ngx.timer.at(0, report_stats, ngx.var.uri, ngx.status)
    if not ok then
        ngx.log(ngx.ERR, "failed to create stats timer: ", err)
    end
}

阶段选错会怎样：踩坑指南

理论讲完了，来看几个经典的"阶段选错"场景。

坑一：在 log 阶段调用 ngx.say() 

location /test {
    content_by_lua_block {
        ngx.say("hello")
    }

    log_by_lua_block {
        -- 错误！响应已经发送，这里无法再输出内容
        ngx.say("this is a log message")  -- 不会报错，但客户端收不到
    }
}

log_by_lua_block 里调用 ngx.say() 不会报语法错，但响应已经发送完毕，这句话不会出现在客户端收到的内容里，会被默默丢弃。正确做法是用 ngx.log() 写到错误日志。

坑二：在 rewrite 阶段试图输出响应 

location /test {
    rewrite_by_lua_block {
        -- 这样写貌似合理，实际上行为不符合预期
        ngx.say("hello from rewrite")
        ngx.exit(200)
        -- 后面的 content 阶段还会执行！
    }

    content_by_lua_block {
        ngx.say("hello from content")
    }
}

rewrite 阶段调用 ngx.say() 后，必须显式调用 ngx.exit() 才能阻止请求继续流转到 content 阶段。但即便如此，在 rewrite 阶段产生响应是不推荐的做法，逻辑上不清晰。需要直接返回响应的，应该放在 content 阶段。

坑三：在 access 阶段做了耗时操作却没有超时保护 

location /api/ {
    access_by_lua_block {
        -- 忘记设置超时时间
        local redis = require "resty.redis"
        local red = redis:new()
        -- 没有调用 red:set_timeout(xxx)

        local ok, err = red:connect("127.0.0.1", 6379)
        -- 如果 Redis 不可用，这里会一直等，卡住整个请求
    }
}

凡是涉及网络 I/O 的操作，一定要设置超时。没有超时的代码在网络抖动时会把 worker 连接耗尽。

正确写法：

access_by_lua_block {
    local redis = require "resty.redis"
    local red = redis:new()
    red:set_timeout(200)  -- 200ms 超时，宁可报错，不要卡死

    local ok, err = red:connect("127.0.0.1", 6379)
    if not ok then
        -- 连接失败时的降级策略：放行还是拒绝，根据业务决定
        ngx.log(ngx.ERR, "redis unavailable, skipping auth check: ", err)
        return  -- 降级放行
    end
}

坑四：content 阶段和 proxy_pass 同时出现 

location /api/ {
    content_by_lua_block {
        ngx.say("hello")
    }
    proxy_pass http://backend;  -- 这行实际上不会生效
}

content_by_lua_block 和 proxy_pass 都是 CONTENT 阶段的指令，同一个 location 里只能有一个起效。这种写法不会报语法错，但 proxy_pass 会被忽略，请求永远只走 Lua 这条路。发现 proxy_pass 不生效时，第一时间检查是否同时写了 content_by_lua_block。

坑五：把业务逻辑放进了 init_by_lua_block 

init_by_lua_block {
    -- 错误：试图在 init 阶段读取请求头
    local ua = ngx.req.get_headers()["User-Agent"]  -- 会报错！
}

init_by_lua_block 运行时完全没有请求上下文，ngx.req.*、ngx.resp.*、ngx.var.*（请求相关变量）等 API 统统不可用。在这里调用会抛出错误，导致 Nginx 启动失败。这个错误通常很容易发现，因为服务压根启动不起来，error.log 里会有明显的报错信息。

一张阶段速查表

把上面的内容整理成一个参考：

阶段                       | 对应指令                     | 能读请求 | 能写响应 | 典型用途
---------------------------|------------------------------|----------|----------|------------------
init_by_lua_block          | http 块内                    |    否    |    否    | 全局初始化、预加载
init_worker_by_lua_block   | http 块内                    |    否    |    否    | 定时任务、worker 初始化
rewrite_by_lua_block       | location/server 块内         |    是    |    否*   | URL 改写、重定向
access_by_lua_block        | location/server 块内         |    是    |    否*   | 鉴权、限流
content_by_lua_block       | location 块内                |    是    |    是    | 业务逻辑、生成响应
header_filter_by_lua_block | location/server 块内         |    是    |  仅响应头 | 修改响应头
body_filter_by_lua_block   | location/server 块内         |    是    |  仅响应体 | 修改响应体内容
log_by_lua_block           | location/server 块内         |    是    |    否    | 日志、监控上报

*标注"否"的阶段技术上可以调用 ngx.say() 然后 ngx.exit()，但不推荐。

综合实战：一个完整的带鉴权的 API

把上面各个阶段串起来，写一个完整的示例：初始化配置 → 鉴权 → 处理请求 → 记录日志。

http {
    lua_shared_dict token_cache 5m;
    lua_shared_dict api_stats   10m;

    # 全局初始化
    init_by_lua_block {
        cjson   = require "cjson"
        REDIS_HOST = os.getenv("REDIS_HOST") or "127.0.0.1"
        REDIS_PORT = tonumber(os.getenv("REDIS_PORT")) or 6379
    }

    server {
        listen 80;

        location /api/v1/ {
            # 鉴权阶段
            access_by_lua_block {
                local token = ngx.req.get_headers()["Authorization"]

                if not token then
                    ngx.status = 401
                    ngx.header["Content-Type"] = "application/json"
                    ngx.say(cjson.encode({ code = 401, msg = "authorization required" }))
                    return ngx.exit(401)
                end

                -- 先查本地缓存（shared dict），减少 Redis 查询
                local cache = ngx.shared.token_cache
                local cached = cache:get(token)

                if cached then
                    -- 命中缓存，把用户 ID 传递给后续阶段
                    ngx.ctx.user_id = cached
                    return
                end

                -- 缓存未命中，查 Redis
                local redis = require "resty.redis"
                local red = redis:new()
                red:set_timeout(200)

                local ok, err = red:connect(REDIS_HOST, REDIS_PORT)
                if not ok then
                    -- Redis 不可用时的策略：这里选择拒绝，保护后端
                    ngx.log(ngx.ERR, "redis connect error: ", err)
                    ngx.status = 503
                    ngx.header["Content-Type"] = "application/json"
                    ngx.say(cjson.encode({ code = 503, msg = "service unavailable" }))
                    return ngx.exit(503)
                end

                local user_id, err = red:get("token:" .. token)
                red:set_keepalive(10000, 100)

                if not user_id or user_id == ngx.null then
                    ngx.status = 403
                    ngx.header["Content-Type"] = "application/json"
                    ngx.say(cjson.encode({ code = 403, msg = "invalid token" }))
                    return ngx.exit(403)
                end

                -- 写入本地缓存，5 分钟过期
                cache:set(token, user_id, 300)
                ngx.ctx.user_id = user_id
            }

            # 业务逻辑阶段
            content_by_lua_block {
                local user_id = ngx.ctx.user_id
                ngx.header["Content-Type"] = "application/json"
                ngx.say(cjson.encode({
                    code = 200,
                    msg  = "ok",
                    data = {
                        user_id    = user_id,
                        uri        = ngx.var.uri,
                        request_id = ngx.var.request_id or ngx.now()
                    }
                }))
            }

            # 日志阶段
            log_by_lua_block {
                local stats = ngx.shared.api_stats
                local key   = ngx.var.uri .. ":" .. tostring(ngx.status)

                stats:incr(key, 1, 0)

                local latency = tonumber(ngx.var.request_time) or 0
                if latency > 0.5 then
                    ngx.log(ngx.WARN, string.format(
                        "[slow] user=%s uri=%s status=%d cost=%.3fs",
                        tostring(ngx.ctx.user_id),
                        ngx.var.uri,
                        ngx.status,
                        latency
                    ))
                end
            }
        }
    }
}

这个示例覆盖了三个核心阶段：access 做鉴权（带本地缓存优化）、content 生成业务响应、log 记录统计和慢请求。ngx.ctx 是 OpenResty 提供的请求上下文表，生命周期绑定在单次请求内，用来在不同阶段之间传递数据，比全局变量安全得多。

小结

这篇文章围绕 OpenResty 的请求生命周期展开，核心要点：

  • Nginx 把请求处理切成 11 个阶段，OpenResty 在其中的关键阶段提供了 Lua 切入点
  • init_by_lua_block 做全局初始化，此时没有请求上下文
  • rewrite_by_lua_block 负责 URL 改写和重定向，不应生成响应内容
  • access_by_lua_block 是鉴权的标准位置，通过 ngx.exit() 拦截非法请求
  • content_by_lua_block 是业务逻辑的主场，和 proxy_pass 互斥
  • log_by_lua_block 在响应发出后运行，适合做异步的日志和监控
  • 阶段选错不一定会报错，但行为会不符合预期，要根据职责选对阶段
  • ngx.ctx 是跨阶段传递数据的正确方式
